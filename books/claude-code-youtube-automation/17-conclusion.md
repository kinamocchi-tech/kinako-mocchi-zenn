## 本書で構築したシステムの全体像

本書では、YouTubeチャンネルの制作パイプラインを Claude Code によって自動化する方法を解説してきました。

```text
Phase 1: テーマ発見（theme-discovery）
  └── ニュースソース巡回 → スコアリング → ネタ候補 Issue 化

Phase 2: リサーチ（research）
  └── Deep Research プロンプト生成 → ユーザーが実行 → 裏付けリサーチ自動化

Phase 3: 台本生成（script-pipeline）
  └── セクション分割並列生成 → チームレビュー → ファクトチェック → PR 自動マージ

Phase 4: 素材制作
  ├── 音声生成（audio-pipeline）: VOICEVOX → 尺チェック → PR 自動マージ
  └── 画像生成（image-pipeline）: Gemini API → サムネイル生成 → PR 自動マージ

Phase 5: 動画制作（video-pipeline）
  └── Remotion レンダリング → 品質チェック → YouTube 限定公開

Phase 6: 公開管理（youtube-scheduler）
  └── バッファ管理 → スケジュール設定 → 自動公開

Phase 6.5: SNS 配信（sns-publisher）
  └── Zenn / note / Qiita / X / Bluesky への並列配信

継続改善: レトロスペクティブ（retrospective）
  └── 各パイプライン後に自動実行 → KPT 生成 → Try Issue 化 → 週次自動対応
```

多くのステップが Claude Code の自律実行によって処理されます。一方で、Deep Research の実行（Phase 2）、PR 作成前の承認（各 Phase）、動画の最終品質確認（Phase 5）はユーザーが行います。完全な無人化ではなく、人間の判断が必要な箇所に絞ってインタラクションポイントを設計しています。

## 「完全自動で80点より半自動で95点」の振り返り

本書を通じて繰り返し強調してきた設計思想が、**「完全自動で80点より半自動で95点を選ぶ」** という原則です。

完全自動化を追求すると次のような問題が発生しやすくなります。

| 問題 | 説明 |
|------|------|
| ハルシネーション見逃し | AI が自信を持って誤情報を出力し、チェックなしに公開される |
| キャラクターのブレ | 微妙なトーンのズレが蓄積し、チャンネルの個性が失われる |
| トレンドの誤読 | 表面的な情報からテーマを選定し、視聴者の本質的な関心と乖離する |

承認ゲートは自動化の妨げではなく、**自動化の品質を保証するための安全装置**です。

## AI が得意なこと・苦手なこと

### AI が得意なこと

- **反復作業の自動化**: 発音辞書の照合、バリデーションチェックなど毎回同じ手順を踏む作業
- **並列処理**: Agent Teams で複数レビュアーを同時起動し、所要時間を大幅に短縮
- **ルールの一貫した適用**: 疲れや気の緩みなく、方言ルールや字幕制約を毎回適用

### AI が苦手なこと

- **創造的判断**: 「このたとえ話は面白いか」「このリズムは聴きやすいか」は人間が磨く
- **視聴者の感情理解**: 文化的・社会的コンテキストへの深い理解が必要な判断
- **トレンドの本質的評価**: 表面的な情報量と視聴者の関心は一致しないことがある

:::message
AI は「what」（何をすべきか）を実行する能力は高いですが、「why」（なぜそれが正しいのか）を判断する能力はまだ限定的です。「what」を AI に任せ、「why」の判断を人間が担う役割分担が現時点での最適解です。
:::

人間が担うべき判断として特に重要なのは、ファクトチェックの最終承認、コンテンツの公開可否、チャンネル戦略の決定、品質基準の設定です。

## AI 生成コンテンツの透明性と開示

YouTube は AI 生成コンテンツに対する規制を段階的に強化しています。2026年4月時点で、AI によって生成または大幅に改変されたコンテンツには、クリエイター自身がラベルを付与する義務があります。特にディープフェイク（実在の人物の顔や声を AI で再現したコンテンツ）には厳しい規制が適用されており、開示なしにアップロードすると削除やチャンネルペナルティの対象になります。

一方で、YouTube は AI ツールの利用自体を禁止しているわけではありません。問題視されているのは、いわゆる「AI スロップ」——AI が生成した低品質・反復的なコンテンツを大量に投稿し、視聴者に価値を提供しないチャンネルです。こうしたチャンネルは収益化プログラム（YPP）の審査で不合格になるか、収益化が停止される方向に進んでいます。

### 本プロジェクトでの具体的な対応

本書で構築したパイプラインでは、以下の AI ツールを使用しています。

| 使用箇所 | AI ツール | 開示内容 |
|---------|----------|---------|
| 音声合成 | VOICEVOX（白上虎太郎・四国めたん） | 合成音声であることを明示 |
| 画像生成 | Gemini API（gemini-3.1-flash-image-preview） | AI 生成画像であることを明示 |
| 台本生成支援 | Claude Code | AI が台本の草案を作成し、人間がレビュー・承認 |
| メタデータ生成 | Claude Code | タイトル・説明文・タグの草案を AI が生成 |

YouTube Studio の「動画の詳細」画面では、AI 生成コンテンツに関する開示設定を有効にすると、動画に「改変または合成コンテンツ」ラベルが表示されます。なお、本プロジェクトの `content/meta/{VIDEO_ID}_meta.yaml` にある `ai_disclosure: true` は、このリポジトリ内でのレビュー・運用用フラグ（メタデータチェック時の Warning 判定など）として管理している項目であり、現状の `tools/youtube_scheduler.py` はこの値を参照せず、YouTube API に反映する処理も行っていません。そのため、YouTube Studio 側のラベル設定は別途手動で行う必要があります。

動画説明文にも以下の記載を標準テンプレートとして含めています。

```text
🤖 AI ツール使用について
本動画では以下の AI ツールを使用しています。
・音声: VOICEVOX（合成音声）
・一部画像: Gemini API による AI 生成
・台本草案: Claude Code（人間がレビュー・承認済み）
```

### 品質へのこだわりが差別化になる

完全自動で大量生産された AI コンテンツと、本書のパイプラインには決定的な違いがあります。それは **人間のレビュー・承認ゲートが組み込まれている** ことです。

台本は Claude Code が草案を生成しますが、ファクトチェック（`fact_check_flags` による検証）、キャラクターの口調チェック、PR レビューでの最終承認を経て初めて音声生成に進みます。画像も生成後にデザイン仕様との整合性を確認します。動画の公開判断は video-reviewer による自動レビューとユーザーの最終確認を経て行われます。

:::message
AI を使っていることを隠す必要はありません。むしろ、AI をどのように活用し、どこに人間の判断を入れているかを明確に開示することが、視聴者の信頼とプラットフォームの収益化要件の両方を満たす方法です。
:::

透明性の確保は単なるコンプライアンスではなく、チャンネルの信頼性を築く投資です。AI 活用の開示を積極的に行い、品質管理プロセスを視聴者に見せることで、「AI を使っているが品質にこだわっている」という姿勢を示すことができます。

## 今後の展望

**CI 品質ゲート（GitHub Actions）**: 現在のバリデーションはローカルフックですが、PRのマージ条件として TypeScript コンパイルや `fact_check_flags` 件数チェックを CI で自動化することで、品質担保をコードレベルで強制できます。

**マルチモーダル AI の活用**: 生成した画像の品質評価（文字化けチェック、キャラクターのアクセサリー確認）も将来的には AI が自動化できます。音声のイントネーション評価も実現可能な射程に入ってきています。

**コミュニティ連携**: YouTube Analytics のコメントデータと視聴維持率を組み合わせて、「どの説明が視聴者に刺さったか」を学習し、次の台本生成に活かすアーキテクチャが考えられます。

## 自分のプロジェクトに活かすために

まず、自分のプロジェクトで最も時間のかかる反復作業を1つ特定してください。YouTube チャンネルでなくても構いません。ブログ記事の校正、コードレビューのチェックリスト確認、定期レポートの生成——何でも構いません。

その作業を Claude Code に任せるためのスキル（SKILL.md）を1つ書いてみてください。「人間がやっていた作業の70%を自動化し、残り30%は人間がレビューする」という半自動化から始めることを強く推奨します。

:::message
本書で解説したシステムは、一度作れば永遠に動き続けるものではありません。モデルのアップデート、外部 API の仕様変更、チャンネルの成長に伴うニーズの変化によって、常にメンテナンスが必要です。レトロスペクティブのサイクルを通じて、システムが自分たちと一緒に成長できるよう設計することが、持続可能な自動化の鍵です。
:::

## おわりに

本書を最後まで読んでいただき、ありがとうございました。

「AI に何かを任せる」ことと「AI と一緒に仕事をする」ことは、似ているようで大きく違います。本書が目指したのは後者です。Claude Code をオーケストレーターとして、各スキルを専門家として、ユーザーをディレクターとして、それぞれの役割を明確にした上でチームとして機能するシステムを設計しました。

どれだけ自動化が進んでも、「何を作るか」「なぜ作るか」という問いに答えるのは、人間の仕事であり続けると考えています。自動化はコンテンツを作る手段であり、コンテンツを作る目的ではありません。本書が、読者の皆さんが「作りたいもの」に集中するための時間を生み出す助けになれば、これ以上の喜びはありません。
