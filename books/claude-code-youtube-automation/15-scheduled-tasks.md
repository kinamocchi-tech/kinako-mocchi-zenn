## このチャプターで学ぶこと

- Claude Code Scheduled Tasks の運用形態（単一アカウント・毎時タスク1本による24時間ルーティング）
- 毎時タスクの **時刻ルーティング** による多ワークフロー統合
- 重いタスク（image-pipeline・video-pipeline）を複数スロットへ分散し、チェックポイントで委譲する設計
- スタック検知・セッション安全ルールの実装

## 背景 — 3スロットから単一アカウント24時間稼働への刷新

claude.ai のスケジュール登録枠には当初 **最大3スロット**（2026年3月時点のMax プラン実測値）という制約があり、これが「毎時1本 + daily 2本」という3分割設計の直接の理由でした。毎時タスクには「前のセッションが実行中でも無条件に新セッションを起動する」仕様もあり、Remotion レンダリングのようなロングタスクを毎時スロットに混在させると途中で強制終了されるリスクがあったためです。

その後、動画制作・公開が安定し在庫バッファが数週間分に積み上がったことで、複数アカウントでスロットを分散させる運用上の必要がなくなりました。2026年6月、運用は単一アカウントに集約され、1つの毎時タスクが JST 時刻を確認してルーティングテーブルを参照する設計に一本化されています（2026年7月時点、`docs/rules/hourly-routing.md`）。

```text
毎時タスク（1本）: 現在時刻（JST）を確認 → ルーティングテーブルの該当スロットを実行 → 一致なしは即終了
```

ロングタスクの中断リスクは、専用枠の固定確保ではなく、フェーズ境界でのチェックポイントによる委譲で解決しています。固定枠が消えた分、時刻ルーティングの側は以前より複雑になりました。

## 時刻ルーティングの全体像

毎時タスクは現在時刻を確認し、該当時刻のワークフローのみを実行します。スケジュール外の時刻は即終了します。単一アカウントで24時間フル稼働するため、深夜帯（00:00〜04:59）にも script/audio/image/video の条件付きスロットが並びます。対象テーマが無ければ即終了するため、空振り時のコストはごくわずかです。

| 時刻（JST） | 曜日 | 実行ワークフロー |
|------------|------|----------------|
| 00:00〜04:59 | 毎日 | audio → image → video → script の深夜スロット（対象テーマがある場合のみ） |
| 06:00〜06:59 | 毎日 | script-pipeline（台本自動制作） |
| 07:00〜07:59 | 月曜のみ | 週次マーケティング管理 → retro-try-handler → project-sync |
| 07:00〜07:59 | 火〜日 | retro-try-handler → self-improvement-loop（消化モード）→ project-sync |
| 08:00〜08:59 | 毎日 | research-runner（Phase 2 リサーチ）→ theme-discovery → refinement（テーマ探索） |
| 09:00〜09:59 | 毎日 | 日次マーケティング軽量チェック + Shorts バックログチェック（不足時は shorts-pipeline を起動）+ コメント監視 + sns-organic-pipeline（朝の投稿） |
| 10:00・13:00・18:00 | 毎日 | audio-pipeline（複数スロットに分散） |
| 11:00〜11:59 | 毎日 | project-sync（スタック検知） |
| 12:00・16:00 | 毎日 | image-pipeline（複数スロットに分散） |
| 14:00・21:00 | 毎日 | video-pipeline（複数スロットに分散。21:00 は sns-organic-pipeline の夜の投稿も実行） |
| 15:00〜15:59 | 毎日 | 対応催促チェック |
| 19:00〜19:59 | 毎日 | Analytics 更新・公開スケジュール反映・公開前チェック |

すべてのスロットは、担当タスクの前に **共通プリフライト**（①〜⑮）を実行します。PR レビュー復帰・速報レーン中断復帰・コメントリアルタイム監視・ベンダーリリース検知・SNS配信滞留リカバリ・Shorts製作滞留リカバリなど、取りこぼしを最大1時間以内に検知して自律的に再開するリカバリ網です。速報級ニュースを検知すると `express-news-pipeline` が担当タスクより優先して起動し、Claude・Gemini・OpenAI・Microsoft の公式アップデートは `vendor-release-radar` が、Claude の活用ノウハウは `claude-practice-radar` がそれぞれ毎時ポーリングして専用レーンで発信します。

## 実装詳細

### 毎時タスクの設定

```text
名前: きなこもっちー YouTube スケジューラー
頻度: 毎時
モデル: Haiku
コネクタ: GitHub のみ
```

プロンプトの冒頭でルーティング判定を行い、スケジュール外は即終了します。これによりトークン消費を最小化できます。

:::message
毎時タスクには Haiku を使います。時刻確認・軽量ワークフロー実行に Sonnet や Opus は不要です。image-pipeline と video-pipeline のみ Sonnet を使います。
:::

### ロングタスクの中断対策

旧設計では image-pipeline・video-pipeline を daily 専用枠（13:00・17:00）に固定し、他のワークフローと同時に走らせないことで中断リスクを避けていました。単一アカウント24時間稼働への刷新後は、この専用枠を廃止し、代わりにフェーズ境界でのチェックポイント判定で中断リスクに対処しています。

| タスク | 想定所要時間 | 対策 |
|--------|------------|------|
| script-pipeline | 〜45分 | hourly ルーティングで完結 |
| audio-pipeline | 〜45分 | hourly ルーティングで完結（複数スロットに分散） |
| image-pipeline | 〜45分 | 複数スロット（01:00・12:00・16:00・22:00 等）に分散配置 + `session_guard.py check` でフェーズ境界委譲 |
| video-pipeline | 〜45分〜（レンダリング規模次第） | 複数スロット（02:00・14:00・21:00 等）に分散配置 + `session_guard.py check` でフェーズ境界委譲 |

各パイプラインはフェーズの区切りで `session_guard.py check` を呼び、5時間ローリング制限への接近を検知すると `exit 10` を返します。この時点で checkpoint コミットを行い、Issue や PR に再開ステップを記録してセッションを終了します。次のスロットで起動した新しいセッションは記録済みの再開ステップから処理を引き継ぐため、専用枠を固定確保しなくても長時間のレンダリングを完走できます。

## スタック検知とクリーンアップ

セッションが強制終了されると、GitHub Issue が `status:in-progress` のままスタックします。毎日 11:00 の project-sync でスタック検知を実施します。

```text
1. status:in-progress の全 Issue を取得
2. 最終更新から4時間以上経過した Issue を検出
3. スタックと判定した Issue を status:waiting-claude にリセット
```

4時間のしきい値は「正常に実行中のタスクが4時間を超えることはない」という前提に基づいています。

## セッション安全ルール

クラウド環境ではセッションタイムアウトが予期せず発生します。タイムアウト後にファイルシステムが破棄されるため、コミットされていない変更は失われます。

:::message
**重要**: タスク完了以外でユーザーへの確認を求める前に、必ず未コミット・未追跡ファイルをコミット＆push します。
:::

```bash
git status
git add .
git commit -m "[V001] wip: ユーザー確認待ち（fact_check_flags 7件）"
git push -u origin {ブランチ名}
# push 完了後にユーザーへの確認メッセージを送る
```

環境変数は `.claude/settings.local.json` ではなく、claude.ai のスケジュールタスク設定画面の「環境変数セクション」で管理します（セッション終了時にファイルシステムが破棄されるため）。

## バッファ管理

YouTube 公開の安定運用には非公開動画のバッファ管理が重要です。バッファ数のしきい値判定と対応方針は Ch.12 で詳述しています。

スケジュールタスクとの接点として、09:00 の日次チェック（`--check-buffer`）が WARNING / CRITICAL を検出した場合は Slack に @mention 通知を送ります。月曜 07:00 の週次チェックでは `--assign-schedule` による公開スケジュールの自動割り当ても合わせて実行します。

## セッションタイムアウト対策

スケジュールタスクの運用で最も厄介だったのがセッションタイムアウトです。クラウド環境には **2種類のタイムアウト** が存在し、それぞれ原因と対策が異なります。

| 種類 | 発生タイミング | 症状 |
|------|-------------|------|
| セッション切れ | スケジュールタスク・長時間アイドル | 新セッションが起動し、未コミット変更が消える |
| Request timed out | 手動セッション中・ツール実行中 | 「Request timed out」表示後にセッション再開ボタンが出る |

### PostCompact 自動コミット

Claude Code のコンテキストウィンドウが約 95% に達すると、自動的にコンテキスト圧縮（Auto Compaction）が発生します。圧縮後に新セッションが起動すると、SessionStart フックが `git checkout -- .` と `git clean -fd` を実行して前セッションの残留ファイルをクリーンアップします。つまり、未コミットの変更はすべて消えます。

この問題を防ぐために、PostCompact フック（`.claude/hooks/post-compact.sh`）を実装しました。圧縮直後に未コミット変更を検出し、自動で `git add -A` → `git commit` → `git push` を実行します。

```bash
# PostCompact フックの処理フロー（簡略版）
if [ "$CLAUDE_CODE_REMOTE" = "true" ]; then
  branch=$(git branch --show-current)
  if [ "$branch" != "main" ] && [ -n "$(git status --porcelain)" ]; then
    git add -A
    git commit -m "[wip] 圧縮前自動コミット（$(date '+%Y-%m-%d %H:%M')）"
    timeout 10s git push
  fi
fi
```

main ブランチでは直接 push が禁止されているためスキップし、ローカル環境ではクリーンアップが走らないため対象外としています。

### フック設計のタイムアウト対策

SessionStart フックの中にネットワーク通信を含む処理がある場合、タイムアウトを設定しないと無限にハングする可能性があります。実際に以下の処理がセッション開始をブロックしていました。

| 処理 | 最悪ケースのブロック時間 | 対策 |
|------|---------------------|------|
| VOICEVOX 起動ポーリング（sleep 2s × 15回） | 30秒 | 初回イテレーションで sleep をスキップ |
| VOICEVOX バイナリダウンロード | 無制限 | `curl --max-time 120` を追加 |
| gh CLI リリース情報取得 | 無制限 | `curl --max-time 15` を追加 |
| gh CLI `.deb` ダウンロード | 無制限 | `curl --max-time 60` を追加 |

ポーリングループの改善は単純です。「まず sleep してからチェック」ではなく「まずチェックしてから sleep」に変更するだけで、初回の無駄な 2 秒を節約できます。

```bash
# 改善前: 初回も sleep してしまう
for i in $(seq 0 14); do
  sleep 2
  curl -s http://localhost:50021/version && break
done

# 改善後: 初回は即チェック
for i in $(seq 0 14); do
  [ "$i" -gt 0 ] && sleep 2
  curl -s http://localhost:50021/version && break
done
```

フックに新しいネットワーク処理を追加するときは、必ず「最悪ケースで何秒ブロックするか」を見積もるようにしています。

### チェックポイント管理

audio-pipeline や image-pipeline には Phase 完了ごとにコミット＆push するチェックポイント管理が実装されていましたが、video-pipeline には未設定でした。Remotion レンダリング（30〜60分）と YouTube アップロードを含む video-pipeline がセッション切れで中断すると、どこから再開すればよいか分かりません。

対策として、video-pipeline に以下の 4 段階のチェックポイントを追加しました。

```text
Phase 0: 準備（素材確認・Remotion コンポーネント設定）
Phase 1: レンダリング完了（output/{動画ID}_draft.mp4 が存在）
Phase 2: メタデータ生成完了（content/meta/{動画ID}_meta.yaml が存在）
Phase 3: YouTube アップロード完了（youtube_video_id が設定済み）
```

各 Phase 完了時に `git commit` と `git push` を実行します。中断からの復帰時は、成果物ファイルの有無で再開地点を判定できます。長時間パイプラインを新規に作成する際は、必ずチェックポイント管理を含めるようにしています。

### 1ターン8ツール制限

手動セッションでは、1つの応答ターン内で多数のツールを連続実行すると、ブラウザと Claude API 間の HTTP タイムアウト（約 2〜5 分）に到達して「Request timed out」が発生します。実測では 14 個のコマンドを連続実行した時点で発生しました。

対策として、1ターンあたりのツール呼び出しを **最大 8 個** に制限しています。超える場合は中間報告をテキスト応答として返し、次のターンで残りを実行します。

もう一つの原因として、セッション切れ直後にリモート git コマンド（`git fetch`、`git rev-parse origin/...` など）を実行するとハングする問題があります。セッション再開直後はまずローカルコマンド（`git status`、`git log`）で状態を確認し、リモート通信が必要な場合は `timeout 10s` を付けて実行するようにしています。

## ハマりポイント

### セッション間の状態引き継ぎ

各セッションは独立した環境で起動されます。Git に push されたファイルは次のセッションで参照できますが、環境変数や一時ファイルは引き継がれません。

### 毎時タスクのコンテキスト肥大化

プロンプトの冒頭に「スケジュール表と一致しない場合は即終了」という条件を配置します。スケジュール外の時刻は最初の判定で終了するため、余分なコンテキストを読み込みません。各ワークフローの詳細説明はプロンプト内に直接書かず、スキルファイルへの委譲で記述量を抑えます。

## まとめ

- **単一アカウント24時間稼働**: 毎時1本のタスクがルーティングテーブルで多数のワークフローを時刻ごとに統合実行
- **セッション競合対策**: 専用枠ではなく `session_guard.py` のフェーズ境界チェックポイントでロングタスクを次スロットへ委譲
- **スタック対策**: 4時間しきい値でスタックした Issue を自動リセット
- **セッション安全**: ユーザー確認前に必ずコミット＆pushして作業を保護

この仕組みが整うと、朝起きたときには前日の制作が進んでおり、バッファが自動的に積み上がっていきます。制作者は「何を作るか」の創造的な判断に集中できます。
