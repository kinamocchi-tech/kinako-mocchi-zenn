## 背景・課題 — YouTube 公開の自動化がなぜ必要か

動画のレンダリングまで自動化しても、「YouTube にアップロードして公開スケジュールを設定する」工程が手動のままでは、パイプラインの最後でボトルネックが生まれます。

さらに、YouTube チャンネル運営には以下の管理業務が伴います。

- **メタデータ管理**: タイトル・説明・タグ・サムネイル・カテゴリの設定
- **スケジュール管理**: 週2本の公開スケジュールの維持
- **バッファ管理**: 制作と公開のペースを合わせ、コンテンツ枯渇を防ぐ
- **品質ゲート**: メタデータ不備のまま公開してしまう事故の防止

本プロジェクトでは YouTube Data API v3 を活用し、これらを可能な限り自動化しています。

## 設計思想

### MCP サーバー経由の API 操作

YouTube API の操作は `tools/youtube_mcp_server.py` をラップした MCP サーバー経由で実行します。OAuth 認証と Cloudflare Workers プロキシの処理が MCP サーバー内にカプセル化されており、スキルやスケジューラーからは認証を意識せず使えます。

```
Claude Code（スキル / スケジューラー）
  ↓ MCP ツール呼び出し
mcp__youtube__check_buffer / assign_schedule / ...
  ↓ 内部で
tools/youtube_scheduler.py（YouTube Data API v3）
  ↓ Cloudflare Workers プロキシ（クラウド環境の場合）
YouTube Data API v3
```

| MCP ツール | 用途 |
|-----------|------|
| `mcp__youtube__check_buffer` | 非公開動画のバッファ数確認 |
| `mcp__youtube__list_private` | 非公開動画一覧取得 |
| `mcp__youtube__assign_schedule` | 公開スケジュール自動割り当て |
| `mcp__youtube__reschedule` | リスケジュール |
| `mcp__youtube__upload_subtitles` | 字幕アップロード |
| `mcp__youtube__localize` | 多言語メタデータ設定 |
| `mcp__youtube__analytics` | Analytics 取得 |

:::message
MCP サーバーが未起動の場合は `python3 tools/youtube_scheduler.py --{コマンド}` で直接実行するフォールバックも用意しています。
:::

### 認証情報の管理

認証情報は以下の優先順位で取得します。

| 優先度 | 方法 | 用途 |
|--------|------|------|
| 1 | 環境変数 | クラウド環境（Scheduled Tasks） |
| 2 | `.env` ファイル | ローカル開発 |
| 3 | `.youtube_token.json` | ローカル開発（後方互換） |

必要な環境変数は3つです。

```bash
YOUTUBE_CLIENT_ID=your_oauth_client_id
YOUTUBE_CLIENT_SECRET=your_oauth_client_secret
YOUTUBE_REFRESH_TOKEN=your_refresh_token
```

クラウド環境では Cloudflare Workers プロキシ経由で API を呼び出すため、プロキシ関連の環境変数も必要です。

```bash
YOUTUBE_API_PROXY_URL=https://your-proxy.workers.dev
YOUTUBE_API_PROXY_AUTH_TOKEN=your_proxy_auth_token
```

### 公開戦略

データに基づいて以下の公開戦略を採用しています。

| 設定 | 値 | 根拠 |
|------|-----|------|
| 公開頻度 | 週2本 | 視聴者の習慣形成 + 制作ペースの持続可能性 |
| 公開曜日 | 火曜・金曜 | テック系視聴者の視聴ピーク |
| 公開時刻 | 20:00 JST | YouTube Analytics の視聴ピーク |
| 最低リードタイム | 3日前 | スケジュール設定からのバッファ |

## 実装詳細

### バッファ管理

「バッファ」とは、YouTube に非公開でアップロード済みかつ公開待ちの動画本数です。安定的なチャンネル運営のために、常に一定数のバッファを維持します。

| バッファ数 | ステータス | 対応 |
|-----------|-----------|------|
| 5本以上 | OK | 通常処理 |
| 3〜4本 | WARNING | ログ出力のみ |
| 1〜2本 | CRITICAL | バッファ不足 Issue を作成/更新 |
| 0本 | EMPTY | 即時 `priority:high` で Issue 作成 |

バッファの正確なカウントには YouTube Data API v3 を使います。ローカルファイルの管理状態ではなく、YouTube 上の実際の状態を権威ソースとします。

```
1. channels.list(mine=True, part="contentDetails")
   → アップロードプレイリスト ID を取得

2. playlistItems.list(playlistId=..., part="contentDetails")
   → 全動画の video ID を取得（ページネーション対応）

3. videos.list(id=..., part="snippet,status")
   → 各動画の privacyStatus を取得（50件ずつバッチ）

4. privacyStatus == "private" の動画数 = バッファ数
```

:::message
`search().list(forMine=True)` は使いません。非公開動画が検索結果に含まれない場合がある API 仕様の制限と、クォータコストが高い（100ユニット/リクエスト）ためです。`channels` + `playlistItems` の組み合わせは各1ユニットで、正確な結果が得られます。
:::

### メタデータ必須チェック

公開スケジュールを設定する前に、以下の全項目をチェックします。

| 項目 | 基準 |
|------|------|
| タイトル | 28〜50文字、キーワード先頭 |
| 説明 | 300文字以上、タイムスタンプ含む |
| カスタムサムネイル | 1280×720px PNG が存在 |
| タグ | 5〜15個 |
| カテゴリ | Science & Technology (28) |
| AI生成コンテンツ開示 | 有効化 |

1つでも欠けている場合はスケジュール設定を保留し、`status:waiting-metadata` ラベルを付与します。

メタデータは `content/meta/{動画ID}_meta.yaml` で管理します。

```yaml
video_id: "V001"
youtube_video_id: "xxxxxxxxxxxxx"
title: "動画タイトル"
description: |
  説明文（300文字以上）
  タイムスタンプ:
  00:00 オープニング
  ...
tags:
  - キーワード1
  - キーワード2
category_id: "28"
privacy: "private"
scheduled_publish_at: "2026-03-25T11:00:00Z"
thumbnail_path: "assets/thumbnails/V001_thumb.png"
ai_disclosure: true
```

### スケジューラーの自動実行タイムライン

以下は **YouTube 公開自動化に関係するワークフローの抜粋** です。1 hourly + 2 daily の計3スロットで管理しています。`theme-discovery`（08:00）、`sns-publisher`（21:00）など公開以外のタスクも同じ hourly スロットで時刻ルーティングされますが、本章では割愛します（Ch.06、Ch.13 参照）。

**毎時タスク（軽量ワークフロー統合）**:

| 時刻（JST） | 曜日 | タスク |
|------------|------|--------|
| 06:00〜06:59 | 毎日 | `script-pipeline`（台本自動制作） |
| 07:00〜07:59 | 月曜のみ | 週次マーケティング管理 + retro-try-handler + project-sync |
| 07:00〜07:59 | 火〜日 | `project-sync` |
| 08:00〜08:59 | 毎日 | `refinement`（テーマ探索・バックログ整理） |
| 10:00〜10:59 | 毎日 | `audio-pipeline`（VOICEVOX 音声生成） |
| 11:00〜11:59 | 毎日 | `project-sync`（スタック検知 + クリーンアップ） |
| 19:00〜19:59 | 毎日 | 前日チェック（翌日公開予定の動画確認） |

**daily タスク（ロングタスク専用）**:

| 時刻（JST） | タスク | 想定所要時間 |
|------------|--------|------------|
| 13:00 | `image-pipeline` | 30〜60分 |
| 17:00 | `video-pipeline` | 60〜180分 |

:::message
ロングタスク（image-pipeline、video-pipeline）を daily 専用枠に分離した理由は **セッション競合の排除** です。Claude Code Scheduled Tasks の毎時タスクは、前のセッションが実行中でも新セッションを起動します。Remotion レンダリング（最大3時間）の途中で強制終了されるリスクを排除するため、daily 枠を使います。
:::

### アカウントウォームアップ期間

YouTubeのスパム検出は `privacy_status` を区別しません。非公開動画でもアップロード行動そのものが監視されています。新規アカウントや復旧後アカウントの「信頼スコア」が低い状態で高頻度アップロードすると、アカウント停止のリスクがあります。

| 期間 | アップロード数/週 | 方法 |
|------|----------------|------|
| 復旧後1週目 | 1本 | YouTube Studio 手動 |
| 復旧後2週目 | 1本 | 手動 or API |
| 3〜4週目 | 2本 | video-pipeline（ウォームアップモード） |
| 5週目以降 | 通常運用 | video-pipeline 通常モード |

ウォームアップ状態は `content/meta/_upload_history.yaml` で管理します。

```yaml
warmup:
  enabled: true
  min_interval_hours: 24
  warmup_end_date: "2026-04-25"
```

`youtube_upload_safety.py` がアップロード安全チェック実行時に `warmup_end_date` を確認し、日付を過ぎていれば `warmup.enabled` を自動で `false` に更新します。

### 公開スロットの自動割り当て

4週先までのスロット（火金 20:00 JST）を事前に確保し、バッファにある動画を自動で割り当てます。

```
2026-03-24 (火) 20:00 JST → V003
2026-03-27 (金) 20:00 JST → V004
2026-03-31 (火) 20:00 JST → 空き
2026-04-03 (金) 20:00 JST → 空き
```

スロットが空きでバッファに動画がある場合、`mcp__youtube__assign_schedule` ツールが自動的に割り当てます。

## ハマりポイント

### 1. タイムゾーン混在バグ（CRITICAL）

YouTube API の `scheduledPublishTime` は UTC で指定します。一方、公開スケジュールの管理は JST で行います。この変換ミスで、意図しない時刻に動画が公開されてしまった事例がありました。

```python
# NG: タイムゾーン未指定（サーバー環境が UTC だとズレる）
now = datetime.now()

# OK: JST 明示
JST = timezone(timedelta(hours=9))
now = datetime.now(JST)
```

さらに深刻な問題として、JST と UTC の文字列比較で公開スケジュールの重複割り当てが発生したことがあります。日付・時刻の比較は必ず同一タイムゾーンに統一してから行うルールを徹底しています。

### 2. 非公開動画のスパム検出

「非公開ならスパム検出されないだろう」という思い込みから、短時間に複数動画をアップロードしてアカウントが凍結されたインシデントがありました。YouTube のスパム検出は `privacy_status` を区別せず、**アップロード行動そのもの** を監視しています。アカウント復旧後のウォームアップ戦略（前述）はこのインシデントから生まれた対策です。

### 3. API クォータの管理

YouTube Data API v3 の日次クォータは 10,000 ユニットです。`videos.update` は1回あたり50ユニットを消費するため、200回の更新が上限です。多言語メタデータの一括設定や、バッファチェックの頻度を考慮したクォータ管理が必要です。

`channels.list` + `playlistItems.list` の組み合わせ（各1ユニット）を選択したのも、クォータ節約のためです。

## YouTube Shorts 3分対応と長尺動画の使い分け

2024年10月以降、YouTube Shorts の最大尺が **60秒から3分に拡張** されました（2026年4月時点）。これにより、起承転結を持つ構成が Shorts でも可能になり、テック解説チャンネルにとって Shorts の活用価値が大きく変わっています。

### 長尺動画と Shorts の役割分担

本プロジェクトでは、長尺動画（15〜20分）と Shorts（〜3分）を以下のように使い分けています。

| 形式 | 尺 | 役割 | 具体例 |
|------|-----|------|--------|
| 長尺動画 | 15〜20分 | テーマの深掘り解説。チャンネルのメインコンテンツ | 「Claude Code 完全ガイド」「AI エージェント入門」 |
| Shorts（宣伝） | 30〜90秒 | 長尺動画のハイライト切り出し。本編への誘導 | 長尺動画の注目ポイントを抽出 |
| Shorts（速報） | 60〜180秒 | AI ニュース速報。3分尺で背景説明まで可能に | 「○○がリリース！何が変わる？」 |

3分に拡張されたことで、速報ショートに背景説明や簡単な比較を含められるようになりました。60秒では「事実の列挙」に留まっていた内容を、起承転結のある構成に仕上げられます。

### エンゲージビューの仕組み

Shorts の再生指標は「エンゲージビュー」です。フィードをスワイプして一瞬だけ映った再生はカウントされず、**視聴者が意識的に視聴を続けた場合のみ** 1ビューとしてカウントされます。

:::message
エンゲージビューは「意味のある視聴」を測る指標です。サムネイルやタイトルの釣りで一瞬表示されただけの再生はカウントされないため、**コンテンツの質が直接再生数に反映される** 仕組みになっています。
:::

この仕組みにより、「冒頭3秒で興味を引く」ことが Shorts では特に重要です。テック解説の場合、冒頭で「結論」や「驚きの事実」を提示し、その後で背景を説明する構成が効果的です。

### `shorts-pipeline` スキルとの連携

本プロジェクトでは Shorts の制作も自動化しています。`shorts-pipeline` スキルが以下の2パターンに対応します。

- **宣伝ショート**: 長尺動画の台本から注目ポイントを自動抽出し、30〜90秒のショート台本を生成
- **速報ショート**: `theme-discovery` スキルが検出した速報性の高いニュースから、60〜180秒のショート台本を生成

いずれも VOICEVOX 音声生成、Remotion レンダリング、YouTube アップロードまでパイプラインで自動実行されます。長尺動画と同じ品質管理フロー（メタデータチェック・AI レビュー）を適用しつつ、縦型（1080×1920）のレイアウトに自動変換します。

## 収益化戦略の多角化

YouTube チャンネルの収益源は広告収入だけではありません。本プロジェクトでは以下の5つの収益チャネルを段階的に構築しています（2026年4月時点）。

### 広告収益（メイン）

YouTube パートナープログラム（YPP）の収益化条件を満たした後、長尺動画にはミッドロール広告（8分以上の動画で有効）、Shorts には Shorts 収益分配が適用されます。

:::message
**Shorts の収益分配モデル**: Shorts フィードの広告収益は、クリエイター全体のプールに集約された後、各クリエイターの Shorts 再生回数に基づいて分配されます。分配比率は **クリエイター 45% / YouTube 55%** です（2026年4月時点）。長尺動画の広告収益とは異なるモデルのため、Shorts 単体での収益は長尺動画より低くなる傾向があります。
:::

### 広告以外の収益チャネル

| 収益チャネル | 概要 | 本プロジェクトでの活用 |
|-------------|------|---------------------|
| スポンサード | 企業からの動画制作依頼・製品紹介 | チャンネル成長後に検討（登録者数の基盤が必要） |
| ショッピング | 動画に商品リンクを紐付け | テック系ツール・書籍の紹介に活用予定 |
| アフィリエイト | 動画説明文からの成果報酬型リンク | Amazon アソシエイト等で AI 関連書籍・ハードウェアを紹介 |
| メンバーシップ | 月額課金による限定コンテンツ提供 | 台本の裏側・制作プロセスの限定公開を検討中 |

長尺動画は広告収益とアフィリエイトに強く、Shorts はチャンネル認知の拡大とショッピング連携に適しています。両形式を組み合わせることで、単一の収益源に依存しない構造を目指しています。

`python3 tools/youtube_scheduler.py --check-ypp-eligibility` コマンドが YPP 収益化条件の到達状況を定期チェックし、閾値到達時には自動で Issue 作成と Slack 通知を行います。

:::message
YPP には 2 段階の基準があります（2026 年 4 月時点）。**早期アクセス**（チャンネル登録者 500 人・総再生時間 3,000 時間）ではファンファンディングやメンバーシップが利用可能になりますが、**広告収益**（ミッドロール広告等）には **チャンネル登録者 1,000 人・総再生時間 4,000 時間** が必要です。本プロジェクトでは両方の閾値を監視しています。
:::

## まとめ

- YouTube Data API v3 は MCP サーバー経由で操作し、OAuth 認証やプロキシをカプセル化しています
- 週2本（火金 20:00 JST）の公開戦略をスケジューラーが自動管理し、4週先までのスロットを確保します
- バッファ管理は API による正確な非公開動画カウントで行い、2本を下回ると自動アラートを発します
- メタデータ必須チェックにより、不備のある動画の公開を防止しています
- 毎時タスク + daily タスクの3スロット構成で、9つのワークフローを効率的に実行しています
- アカウントウォームアップ戦略により、凍結リスクを最小化しています
- Shorts 3分対応により速報・宣伝の両パターンで起承転結のある構成が可能になり、`shorts-pipeline` で自動制作しています
- 収益化は広告だけでなく、アフィリエイト・ショッピング・メンバーシップ等の多角化を段階的に進めています
