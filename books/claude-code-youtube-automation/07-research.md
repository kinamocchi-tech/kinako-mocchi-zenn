## このチャプターで学ぶこと

- 情報ランク制度（A/B/C）の設計と運用
- リサーチフロー全 4 ステップ（プロンプト生成→Deep Research→裏付け→ギャップ分析）
- リサーチドキュメントのフォーマット（単一ファイル形式 vs ディレクトリ分割形式）
- ファクトチェックの仕組み（`fact_check_flags`）
- 正式名称の厳守ルール
- AI リサーチの限界とそれを補完する設計

---

## 背景 — テック解説チャンネルの「正確性」問題

テック解説動画の最大のリスクは **ファクトの誤り** です。特に AI/LLM 分野は情報の更新が速く、1 ヶ月前の情報が既に古くなっていることも珍しくありません。

よくある失敗パターンをいくつか挙げます。

- **数字の誤り**: 「Claude の ARR は 25 億ドル」と断言したが、出典はランク C の個人ブログだけだった
- **正式名称の不統一**: 「GPT5」と表記したが、公式は「GPT-5」（ハイフンあり）だった
- **推測の事実化**: 「OpenAI はおそらく〜するだろう」が、いつの間にか「OpenAI は〜する」になっていた

これらのミスは視聴者の信頼を損ない、コメント欄が荒れる原因になります。本プロジェクトでは、リサーチの段階で **情報の信頼度を定量化** し、台本生成時に不確実な情報を自動検出する仕組みを構築しています。

## 設計思想

### 情報ランク制度（A/B/C）

すべての事実情報に対して、出典の種別に基づきランクを付与します。

| ランク | 出典の種別 | 信頼度 | 台本での扱い |
|--------|-----------|--------|-------------|
| **A** | 公式情報（公式ブログ、論文、プレスリリース） | 最高 | そのまま使用可能 |
| **B** | ニュース記事（大手メディアの署名記事） | 高 | 使用可能。ランク A で裏付け推奨 |
| **C** | コミュニティ情報（個人ブログ、SNS、Reddit） | 参考 | 事実としては使用不可 |

ランクの判定は出典のドメインで機械的に判断できます。

```text
## ランク判定の具体例

### ランク A（公式情報）
- anthropic.com/news/* — Anthropic 公式ブログ
- arxiv.org の論文
- GitHub 公式リリースノート
- SEC 提出書類

### ランク B（ニュースメディア）
- TechCrunch, The Verge, Wired, Bloomberg
- 日経新聞、ITmedia、CNET Japan

### ランク C（コミュニティ情報）
- X (Twitter) の個人投稿（CEO 含む）
- 個人ブログ、Qiita、Zenn
- Reddit、Hacker News のコメント
```

:::message
**重要なルール**: ランク C のみの情報を台本の事実として採用することは禁止です。「〜という声もある」「コミュニティでは〜と言われている」といった引用としてのみ使えます。ランク B の情報も、可能な限りランク A で裏付けを取ることが推奨されます。
:::

### X 投稿のランク — よくある間違い

AI レビューで繰り返し指摘されるパターンとして、**CEO の X 投稿をランク A と判定してしまう** 問題があります。Dario Amodei の X 投稿であっても、ランクは C です。例外は公式アカウントからの製品リリース発表で、公式ドキュメントへのリンクが含まれる場合のみ A と判定できます。

参考文献一覧では「X 投稿はランク C だが複数 B メディアで裏付け済みのためランク B 欄に記載」として **ランク B 欄** に配置します。ランク A 欄に X 投稿を置くことは絶対にありません。

## 実装詳細

### リサーチフロー全 4 ステップ

Phase 2 のリサーチは以下の 4 ステップで進みます。

```text
Step 1: リサーチプロンプト生成 [Claude Code 自動]
  ↓
Step 2: Deep Research 実行 [ユーザー手動]
  ↓
Step 3: 裏付けリサーチ [Claude Code 自動]
  ↓
Step 4: ギャップ分析＆追加リサーチ [Claude Code → ユーザー]
```

#### Step 1: リサーチプロンプト生成

テーマに応じた Deep Research プロンプトを自動生成し、`content/research/{動画ID}_prompt.md` に出力します。プロンプトには以下を含めます。

- 調査対象の明確な定義
- 日本語・英語での検索指示（英語圏の一次情報を優先）
- 公式情報（ランク A）を最優先で収集する指示
- 参考 URL の必須添付指示
- サービス名・人物名の正式表記確認指示

#### Step 2: Deep Research 実行

ユーザーが Claude チャットの Deep Research 機能でプロンプトを実行します。結果は `content/research/{動画ID}_deep_research.md` に保存します。テーマの規模によっては複数パートに分かれることがあり、その場合は `_part1.md`, `_part2.md` のように連番で保存します。

:::message
**なぜ Deep Research はユーザーが手動実行するのか**: Deep Research は Claude の対話型機能であり、Claude Code の Scheduled Tasks からは直接実行できません。この「人間がリサーチを実行する」ステップが、品質保証のゲートとしても機能しています。
:::

#### Step 3: 裏付けリサーチ

Deep Research の結果を受け取ったら、Claude Code が自動で以下を実行します。

1. **情報ランク付け**: 全事実情報にランク A/B/C を付与
2. **URL 検証**: 参考 URL が実在するか Web 検索で確認
3. **正式名称チェック**: サービス名・人物名が公式表記と一致するか検証
4. **ランク A 裏付け**: ランク B/C の重要情報について公式情報での裏付けを試みる
5. **交差検証**: 数字・日付・人名について複数ソースでの一致を確認
6. **`fact_check_flags` 生成**: 裏付けが取れなかった情報にフラグを付与

#### Step 4: ギャップ分析

台本構成に必要な情報が揃っているか確認し、不足している観点やデータを特定します。Claude Code が Web 検索で補完可能な情報は自動追加し、Deep Research が必要な規模の不足は追加プロンプトを生成してユーザーに再実行を依頼します。

### リサーチドキュメントのフォーマット

リサーチ結果は、テーマの規模に応じて 2 つの形式のいずれかで出力します。

#### 単一ファイル形式（セクション数 5 以下）

```
content/research/V001.md
```

1 つのファイルにメタ情報・本文・fact_check_flags・参考文献一覧をすべて含めます。

#### ディレクトリ分割形式（セクション数 6 以上 or 30KB 超）

```
content/research/V007/
├── _index.md           ← メタ情報・ランクサマリー・参考文献一覧
├── section_01_ipo_timeline.md
├── section_02_valuation_funding.md
└── section_03_investors.md
```

実際の V007（Anthropic IPO 準備と評価額）リサーチでは、このディレクトリ分割形式を使用しています。`_index.md` には以下のメタ情報が含まれます。

```markdown
## 情報ランクサマリー

| ランク | 件数 | 割合 |
|--------|------|------|
| A（公式） | 18件 | 45% |
| B（ニュース） | 16件 | 40% |
| C（コミュニティ） | 6件 | 15% |
```

ランク A が 45% という数字は、このリサーチの信頼度を端的に示しています。ランク A が 30% を下回るリサーチは、追加の裏付けが必要です。

### fact_check_flags の仕組み

裏付けが不十分な情報や、特に重要な数字・日付・人名には `fact_check_flags` を付与します。これは台本 JSON のセリフレベルで管理されます。

```json
{
  "id": "045",
  "text": "Anthropic の ARR は 190 億ドルに達しているわ。",
  "fact_check_flags": [
    {
      "claim": "Anthropic ARR $19B（2026年3月時点）",
      "source_url": "https://www.bloomberg.com/news/...",
      "rank": "B",
      "note": "Bloomberg報道。Anthropic公式発表なし"
    }
  ]
}
```

フラグには以下の情報を含めます。

| フィールド | 説明 |
|-----------|------|
| `claim` | 検証が必要な主張の内容 |
| `source_url` | 出典の URL |
| `rank` | 情報ランク（A/B/C） |
| `note` | 補足（なぜ要確認なのか） |

:::message
**fact_check_flags が 5 件以上ある場合**: PR の説明文に「要確認事項が多いため重点レビューをお願いします」と明記します。これはユーザーへの注意喚起であり、PR マージ前に人間が必ず確認するゲートとして機能します。
:::

### 正式名称の厳守

サービス名・企業名・人物名は **公式の正式名称** を使用します。リサーチドキュメントの `_index.md` に「正式名称確認リスト」を設けて管理しています。

| NG | OK |
|----|----|
| アンソロピック | Anthropic |
| ダリオ | Dario Amodei（初出時フルネーム） |
| GPT5 | GPT-5（ハイフンあり） |
| カーパシー | Andrej Karpathy |

正式名称が特定できない場合は `[要確認: 正式名称]` フラグを付けて、台本生成に進む前に解決します。

## ハマりポイント

### 1. Deep Research のリンクテキストアーティファクト

Deep Research の出力には `[IntuitionLabs +2]` のようなリンクテキストが混入することがあります。これは Deep Research の集約ツールが生成するアーティファクトで、そのまま使うとリサーチドキュメントの品質が低下します。

```markdown
<!-- NG: アーティファクトがそのまま残っている -->
Anthropic の評価額は $380B に達した [IntuitionLabs +2]

<!-- OK: 適切なリンクテキストに修正 -->
Anthropic の評価額は $380B に達した [B](https://techcrunch.com/...)
```

このパターンは AI レビューでも繰り返し指摘されており（P-04）、セルフレビューの自動検出チェックに追加しています。

### 2. 定量データの出典要件

数字・指標（スター数・ユーザー数・ARR 等）には **直接出典 URL** を必ず付けます。「Awesome list」や「まとめブログ」等の二次ソースから数字を引用することは禁止です。直接出典 URL が確認できない定量データは `[要確認]` フラグを付けるか、台本から削除します。

```markdown
<!-- NG: 二次ソースからの数字引用 -->
awesome-claude-code のスター数は 5,000 超 [C]

<!-- OK: 一次ソースからの引用 -->
Claude Code の GitHub リポジトリのスター数は 15,000 超 [A](https://github.com/...)
```

### 3. Markdown テーブル構文の破損

Deep Research の出力では、Markdown テーブルのヘッダー区切り行（`|---|---|`）が欠落・結合されることがあります。リサーチ結果をファイルに保存した後、テーブル構文が壊れていないか確認が必要です。特に長いテーブルの途中で行が消えやすい傾向があります。

## まとめ

Phase 2 のリサーチパイプラインは、以下の原則に基づいて設計されています。

- **情報ランク制度による定量化**: A/B/C の 3 段階で情報の信頼度を客観的に評価
- **人間と AI の役割分担**: Deep Research はユーザーが実行し、裏付けリサーチは Claude Code が自動実行
- **fact_check_flags による追跡**: 不確実な情報をフラグで管理し、台本→音声→動画の全工程で追跡可能
- **正式名称の厳守**: 公式表記を確認リストで管理し、スペルミスや略称の混入を防止

「AI のファクトチェックには限界がある」という前提を受け入れた上で、その限界を補完する仕組みを設計することが重要です。完全な自動化ではなく、**不確実な箇所を確実にフラグ付けし、人間がレビューするゲートを設ける** ことで、テック解説チャンネルに求められる正確性を担保しています。
