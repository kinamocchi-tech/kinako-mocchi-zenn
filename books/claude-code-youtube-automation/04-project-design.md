## 設計思想 — ハーネスエンジニアリング

AI にタスクを任せるとき、最もよくある失敗は **「お願い」だけで終わること** です。

> 「台本は90文字以内にしてね」

この指示は 1 回のセッションでは守られます。しかし 10 回、100 回とセッションを重ねるうちにどこかで忘れられます。

本プロジェクトでは、この問題を **ハーネスエンジニアリング**（制約を「お願い」から「仕組み」へ段階的に強化するアプローチ。本書独自の用語です）で解決しています。

### 制約の強度ランク

| Lv | 種別 | 強制力 | 実装例 |
|----|------|--------|--------|
| **Lv1** | ドキュメント | 低（AI が読む） | `CLAUDE.md`, `.claude/rules/*.md` |
| **Lv2** | AI セマンティック | 中（スキルが検証） | `self-reviewer`, `fact-checker` スキル |
| **Lv3** | フック | 高（物理的にブロック） | `post-tool-use-validate.sh` |
| **Lv4** | CI / 構造テスト | 最高（マージ不可） | GitHub Actions での品質ゲート（2026年3月時点では未実装・将来構想） |

**昇格ルール**: 同じミスが 2 回以上繰り返されたら、上位レベルへの昇格を検討します。

**具体例（セリフ文字数制限）**:

1. **Lv1**: CLAUDE.md に「1 セリフ 100 文字以内」と記載
2. ↓ 2 回違反が発生
3. **Lv2**: `self-reviewer` スキルが文字数を自動カウント
4. ↓ それでも漏れが発生
5. **Lv3**: `post-tool-use-validate.sh` が Write 時に 100 文字超を `exit 2` でブロック

```bash
# post-tool-use-validate.sh より抜粋
over_100=$(echo "$lines_json" | jq -r '
  [.[] | select(.text != null) | select((.text | length) > 100)] |
  if length > 0 then
    (.[0:3] | map("  id=\(.id // \"?\"): \(.text | length)文字") | join("\n"))
  else empty end
' 2>/dev/null || true)

if [ -n "$over_100" ]; then
  jq -n --arg m "$over_100" \
    '{"systemMessage": "[script-validator] セリフ文字数違反: 100文字超のセリフがあります。"}'
  exit 2
fi
```

:::message
最初から Lv3 で全てをガードする必要はありません。まず Lv1 で運用し、実際にミスが発生したら段階的に強化していく方がメンテナンス性が高いです。
:::

## CLAUDE.md の構造

`CLAUDE.md` はプロジェクトルートに配置し、Claude Code がセッション開始時に自動で読み込みます。本プロジェクトの CLAUDE.md は以下のセクションで構成されています。

| セクション | 内容 |
|-----------|------|
| 応答スタイル | 言語・ペルソナの指定 |
| プロジェクト概要 | チャンネルの説明 |
| 必読ドキュメント | 作業前に読むべきファイル一覧 |
| **判断に迷った時の優先順位** | トレードオフ発生時のルール |
| GitHub プロジェクト管理 | Issue・PR・ブランチ運用 |
| 制約の強度ランク | ハーネスエンジニアリングの定義 |
| やってはいけないこと | 明示的な禁止事項 |
| 作業の進め方 | Plan Mode・脱線検知・完了条件 |

### 判断に迷った時の優先順位

AI が複数のルールを同時に満たせないとき（トレードオフ）のために、優先順位を明示します。

```
最優先: 正確性
  → ファクトチェックで不確実な情報があれば尺が短くなっても削除
  ↓
第2優先: キャラクターの一貫性
  → キャラ設定に合わない言動は削除
  ↓
第3優先: 尺（8〜20分）
  → 台本が22分になった場合は冗長セクションを削って20分以内に収める
  ↓
第4優先: 自動化の効率
  → 品質を落とすくらいなら手動フォールバック
```

:::message
優先順位が定義されていないと、AI は毎回異なる判断をして成果物の品質にブレが生じます。
:::

### 「やってはいけないこと」の明示

禁止事項は独立セクションとして列挙します。

```markdown
## やってはいけないこと
- mainブランチに直接pushしない
- ユーザーの承認なしに動画を公開しない
- 台本にハルシネーションを含めたまま進めない
- 1動画あたり20分を超える台本を作らない
- .claude/settings.local.json に環境変数を書き込まない
```

## ルールファイルの分類と配置

### シンボリックリンク方式

ルールファイルの実体は `docs/rules/` に配置し、`.claude/rules/` にはシンボリックリンクを置きます。Claude Code はセッション開始時に `.claude/rules/` を自動読み込みするため、この方式で「ドキュメントの一元管理」と「自動読み込み」を両立できます。

```
docs/rules/             ← 実体（編集対象）
├── script-rules.md
├── research-rules.md
└── pr-review-flow.md

.claude/rules/          ← シンボリックリンク（自動読み込み対象）
├── script-rules.md   → ../../docs/rules/script-rules.md
├── research-rules.md → ../../docs/rules/research-rules.md
└── pr-review-flow.md → ../../docs/rules/pr-review-flow.md
```

```bash
# 新規ルールファイル追加の手順
vim docs/rules/new-rule.md
ln -s ../../docs/rules/new-rule.md .claude/rules/   # 相対パスで作成
git add docs/rules/new-rule.md .claude/rules/new-rule.md
```

:::message alert
シンボリックリンクは必ず **相対パス** で作成してください。絶対パスにすると、クラウド環境とローカル環境でパスが変わってリンクが壊れます。Windows 環境では `git config core.symlinks true` の設定が必要です。
:::

### ルールファイルの分類

本プロジェクトでは 2026 年 3 月時点で 18 個のルールファイルを運用しています。

| カテゴリ | ファイル例 | 内容 |
|---------|----------|------|
| コンテンツ制作 | `script-rules.md` | 台本の構造・方言・字幕制約 |
| 品質管理 | `research-rules.md` | 情報ランク制度・ファクトチェック |
| 音声・映像 | `intonation-rules.md` | VOICEVOX イントネーション制御 |
| GitHub 運用 | `pr-review-flow.md` | PR レビューフロー |
| 運用 | `session-safety-rules.md` | セッション安全ルール |

CLAUDE.md との役割分担は以下の通りです。

- **CLAUDE.md**: 最重要ルール・優先順位・判断基準（常に読まれる）
- **`.claude/rules/*.md`**: Phase 別の詳細ルール（自動読み込み）
- **`docs/*.md`**: 参照ドキュメント（必要時のみ読み込み）
- **`SKILL.md`**: スキル固有の手順（スキル起動時のみ読み込み）

## セルフ改善ループ

本プロジェクトの最も特徴的な仕組みの1つです。

```
1. PR で AI レビューから指摘を受ける
  ↓
2. 指摘パターンを docs/rules/lessons.md に記録
  ↓
3. 同じ指摘が 2 回以上 → セルフレビュースキルに自動検出チェックを追加（Lv2 強化）
  ↓
4. さらに繰り返されたら Lv3 フックに昇格
```

### 学習ログの実例

```markdown
### P-02: emotion フィールドが Remotion 型と不一致 ⚠️ 繰り返し

**発生PR**: PR#79 (V001), PR#81 (V001), PR#221 (V007)
**内容**: 台本JSONの emotion に neutral, angry 等、Remotion の型に
  存在しない値が混入する。

**有効な値**: normal / neutral / excited / confused / sad /
  whisper / tsukkomi / teaching / gentle

**セルフレビュー追加チェック**:
- 全 emotion フィールドを上記リストで検証 → 不正値は Error
```

2026 年 3 月時点で 16 個の繰り返しパターン（P-01〜P-16）が記録されています。

:::message
「完璧なルールを最初から作る」のではなく、「運用しながら学習・強化する」アプローチです。まず Lv1 の簡素なルールから始め、実際のミスに基づいて精緻化していきましょう。
:::

## まとめ

| 設計原則 | 実装 |
|---------|------|
| 「お願い」から「仕組み」へ | 制約の強度ランク Lv1〜Lv4 |
| 優先順位の明示 | 正確性 > キャラ一貫性 > 尺 > 自動化効率 |
| 知識の永続化 | CLAUDE.md + ルールファイル（シンボリックリンク方式） |
| 継続的改善 | セルフ改善ループ（学習ログ → ルール化 → フック強化） |

次のチャプターでは、これらのルールに基づいて運用される GitHub Issues・Milestones・Projects V2 の自動化を解説します。
