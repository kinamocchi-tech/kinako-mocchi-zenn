## 3つの拡張機能が解決する課題

Claude Code をそのまま使うだけでも強力ですが、繰り返し発生する複雑なワークフローを自動化するには課題があります。

| 課題 | 解決する機能 |
|------|------------|
| 知識の揮発性（セッションが切れるたびにルールを伝え直す） | **Skills** |
| 品質の不安定性（AI への「お願い」だけではルール違反を防げない） | **Hooks** |
| 処理の非効率性（台本生成・音声合成・画像生成を逐次処理） | **Agent Teams** |

## Skills — 再利用可能なワークフロー定義

### 仕組み

Skills は `.claude/skills/` に配置する **SKILL.md** ファイルを中心とした仕組みです。Claude Code はセッション開始時にこのディレクトリを自動スキャンし、各スキルの `description` を認識します。ユーザーの指示がトリガー条件に合致すると、対応する SKILL.md を読み込んで実行します。

:::message
**トリガーの仕組み**: Claude Code は SKILL.md の `description` を読み、ユーザーの指示内容と LLM ベースでマッチングします。そのため `description` の記述が品質に直結します。「いつ使うか」「何をするか」を簡潔かつ具体的に書くことが重要です。
:::

### SKILL.md の構成

```yaml
---
name: script-writer
description: 「きなこもっちーのテック深掘り」の台本JSONを生成・編集する。
  台本生成、台本レビュー、台本修正を依頼された時に使用する。
---
```

| フィールド | 制約 | 役割 |
|-----------|------|------|
| `name` | 64 文字以内、小文字＋ハイフン | スキルの一意識別子 |
| `description` | 1024 文字以内 | トリガー判定に使われる概要文 |
| 本文 | 5000 トークン以内推奨 | 詳細な手順・ルール・テンプレート |

### 実例: script-writer スキルの分割生成モード

`script-writer` スキルは、コンテキスト肥大化を防ぐために以下の分割生成モードで動作します。

```
Phase A: セクション設計（メインエージェント）
  ↓
Phase B: セクション分割生成（サブエージェント × N並列・Opus モデル）
  ↓
Phase C: 統合・ファイル書き込み（メインエージェント）
  ↓
Phase D: ボリュームチェック・品質チェック
  ↓
Phase E: コミット＆push
```

:::message
Skills と Agent Teams を組み合わせることで高度なワークフローを実現できます。台本の各セクションを Opus モデルのサブエージェントが並列生成し、メインエージェントが統合する設計です。
:::

### スキル探索フロー

新しいタスクに直面したとき、以下の優先順位でスキルを探します。

1. `.claude/skills/` 内の SKILL.md を確認
2. 公式スキルリポジトリ（[anthropics/skills](https://github.com/anthropics/skills)）で検索
3. `skill-creator` スキルで新規作成

本プロジェクトでは 2026 年 4 月時点で 30 個以上のスキルを運用しています。

## Hooks — 物理的な品質ゲート

### 仕組み

Hooks はツール実行の前後に自動で走るシェルスクリプトです。`.claude/settings.json` の `hooks` セクションで登録します。

| フック | タイミング | 主な用途 |
|--------|----------|---------|
| **PreToolUse** | ツール実行前 | 危険な操作のブロック |
| **PostToolUse** | ツール実行後 | 出力の品質検証 |
| **SessionStart** | セッション開始時 | 環境セットアップ |
| **Stop** | セッション終了時 | 状態保存・通知 |

### 実例 1: post-tool-use-validate.sh — 台本 JSON バリデーション

`content/scripts/*.json` が書き込まれるたびに自動実行され、以下をチェックします（詳細は Ch.14 で解説）。

| チェック | 違反時の動作 |
|---------|------------|
| JSON 構文の妥当性 | `exit 2` でブロック |
| 必須フィールド存在確認 | `exit 2` でブロック |
| キャラクター方言ルール違反 | `exit 2` でブロック |
| セリフ 100 文字超 | `exit 2` でブロック |
| 総文字数 6,000 文字未満 | `exit 2` でブロック |
| セリフ 67 文字超 | Warning（続行） |

:::message
**exit コードの意味**: `exit 0`=成功、`exit 1`=致命的エラー（セッション全体が停止の可能性あり）、`exit 2`=ツール実行失敗（Claude Code が修正を試みる）。Warning を出すだけなら `exit 0` で終了してください。
:::

### 実例 2: session-start.sh — 環境自動セットアップ

クラウド環境でのセッション開始時に、必要なツールを自動インストールします。

- gh CLI の最新版インストールと認証
- Remotion の `npm install`
- VOICEVOX Engine のダウンロード・起動

```bash
# VOICEVOX Engine の自動起動（すでに起動済みならスキップ）
if ! curl -sf http://localhost:50021/version > /dev/null 2>&1; then
  # バイナリがなければダウンロード・解凍→バックグラウンド起動
  ...
fi
```

:::message
SessionStart フックにより、クラウド環境でもセッション開始時に自動で開発環境が構築されます。ユーザーが手動でツールをインストールする必要はありません。
:::

## Agent Teams — 並列サブエージェントの活用

### 仕組み

Agent ツールは Claude Code 組み込みのサブプロセス起動機能で、独立したエージェントを並列実行できます。Agent Teams は `Agent` ツールを使い、メインエージェントから **サブエージェント** を起動する仕組みです。

```
メインエージェント（オーケストレーション）
├── Explore エージェント（コードベース調査）
├── 実装エージェント A（ファイル編集）
├── 実装エージェント B（別ファイルの並行編集）
└── 検証エージェント（テスト実行・品質チェック）
```

### Phase 別の並列化パターン

| Phase | 並列化の例 |
|-------|-----------|
| Phase 2（リサーチ） | ファクトチェックと網羅性チェックを並列実行 |
| Phase 3（台本制作） | セクション単位で 2〜3 セクションずつ並列生成 |
| Phase 4（素材制作） | 音声合成と画像生成を並列実行 |
| Phase 5（動画制作） | 尺・同期・無音区間の自動チェックを並列実行 |

### モデル選択ルール（コスト最適化）

| モデル | 用途 | 使用場面 |
|--------|------|---------|
| **Opus** | 高度な判断・創作 | 台本生成、複雑な設計判断 |
| **Sonnet** | 標準的な実装・分析 | メインセッションのデフォルト、実装・PR 作成 |
| **Haiku** | 単純な調査・チェック | Explore エージェント、セルフレビュー個別チェック |

```
subagent_type=Explore → model="haiku"   # コードベース調査
台本生成エージェント   → model="opus"    # 創作タスク
実装・検証エージェント  → model="sonnet"  # 標準タスク
```

:::message
迷ったら Sonnet を選びます。Opus は台本生成のような高度な創作タスクにのみ使用し、調査・検証系は Haiku でコストを抑えます。
:::

### コンテキストコンパクションパターン

長時間ワークフローでコンテキストウィンドウが膨張するのを防ぐため、サブエージェントは成果物の全文ではなく **構造化サマリー** を返します。

```json
{
  "summary": "V007 セクション3: AI安全性の課題 — 生成完了",
  "outputs": ["content/scripts/V007_sections/03_safety.json"],
  "warnings": ["fact_check_flags 2件追加"],
  "next_recommended_action": "セクション4-5を生成"
}
```

また、エラーの複合蓄積を防ぐため（各ステップ 95% × 10 ステップ = 全体成功率約 60%）、**5 ステップごとに検証ポイント** を設けます。

## Claude Code の最新機能（2026 年版）

2026 年 2〜3 月にかけて、Claude Code の基盤となるモデルとクライアント機能に大きなアップデートがありました。ここでは、本プロジェクトの制作パイプラインに直接影響する 4 つの機能を紹介します。

### 1M コンテキストウィンドウ

Opus 4.6 / Sonnet 4.6 がコンテキストウィンドウ **1M トークン** に対応しました（従来は 200K トークン）。これにより、長時間ワークフロー中にコンテキスト圧縮（Auto Compaction）が発生しにくくなり、パイプラインの安定性が大幅に向上しています。

| モデル | コンテキスト | 用途 |
|--------|------------|------|
| Opus 4.6 | 1M トークン | 台本生成・複雑な設計判断 |
| Sonnet 4.6 | 1M トークン | メインセッション・実装・PR 作成 |
| Haiku 4.5 | 200K トークン | 調査・チェック系タスク |

:::message
1M トークンあっても不要な情報を詰め込むとコストとレイテンシが増加します。サブエージェントへは構造化サマリー（1,000〜2,000 トークン）を返す運用は引き続き有効です。
:::

### Adaptive Thinking

Opus 4.6 / Sonnet 4.6 に搭載された **Adaptive Thinking** は、クエリの複雑さに応じて推論の深さを自動調整する機能です。単純な質問には素早く、複雑なタスクには深く考えます。

`/effort` コマンドで明示的に制御することも可能です（2026 年 4 月時点）。

| レベル | 挙動 | 本プロジェクトでの用途 |
|--------|------|---------------------|
| `low` | 速度優先・最小コスト | 定型チェック・ルーティング判断 |
| `medium` | トークン節約 | 通常の実装・PR 対応 |
| `high` | API デフォルト | 複雑な実装・レビュー対応 |
| `max` | フル推論・最高品質（Opus 4.6 のみ） | 台本生成 |

本プロジェクトでは、台本生成（Phase 3）時に Opus 4.6 + `/effort max` を組み合わせて最高品質の出力を得ています。

### Remote Control

スマートフォンや別 PC から、実行中の Claude Code セッションに接続して操作を継続できる機能です。ターミナルの前にいなくても、外出先からパイプラインの進捗確認や承認操作が可能になります。

本プロジェクトでは、外出先からスマートフォンで Remote Control を使い、パイプラインの進捗確認や必要に応じた操作をハンズフリーで実行するワークフローが実用化されています。

### ネイティブ音声モード

20 言語に対応した音声入力機能です。キーボードを使わず、音声で Claude Code に指示を出すことができます。コードレビューのフィードバックや台本の修正指示を口頭で伝えられるため、ハンズフリーでの作業が可能です。

:::message
これらの機能は 2026 年 4 月時点の情報です。Claude Code は頻繁にアップデートされるため、最新の仕様は [公式ドキュメント](https://docs.anthropic.com/en/docs/claude-code) を確認してください。
:::

## ハマりポイント

**SKILL.md の description が長すぎる**
: 1024 文字の制限内で「いつ使うか」「何をするか」を簡潔かつ網羅的に記述してください。

**サブエージェントのファイル競合**
: 複数のサブエージェントが同じファイルを同時に編集すると競合が発生します。サブエージェントには **結果を文字列で返させ、メインエージェントがファイルに書き込む** 設計にしてください。
