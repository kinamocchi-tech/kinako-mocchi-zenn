## 背景 — 「何を作るか」が最も難しい

YouTube チャンネル運営で最も消耗するのは、意外にも **テーマ選定** です。台本や音声は一度パイプラインを構築すれば自動化できますが、「今週何の動画を出すか」は毎回人間が考えなければなりません。

テック系チャンネルの場合、以下の課題があります。

- **情報の鮮度**: AI 分野は週単位でトレンドが変わる
- **検索需要の見極め**: バズっているトピックと検索需要のあるトピックは異なる
- **差別化**: 同じトピックでも「自分たちならではの切り口」がないと埋もれる
- **ネタ切れ回避**: チャンネルを継続するには常に 3〜5 本分のストックが必要

本プロジェクトでは、このテーマ選定を **Claude Code の `theme-discovery` スキル** で自動化しています。毎日のスケジュール実行でテックニュースを巡回し、動画化ポテンシャルをスコアリングして GitHub Issue に登録するパイプラインです。

## 設計思想

### 2 段階の選定プロセス

テーマ選定は「発見」と「昇格」の 2 段階に分離しています。

```text
Stage 1: テーマ発見（theme-discovery スキル）
  テックニュース巡回 → スコアリング → [ネタ候補] Issue 作成
  ↓
Stage 2: 自動昇格（refinement スキル）
  スコア ≥ 15 のネタ候補を検出 → Video ID 採番 → Phase 2 Issue 作成
```

この分離には 2 つの理由があります。

1. **発見フェーズのコストを低く保つ**: 毎日のニュース巡回は Haiku モデルで十分実行できます。Issue を作るだけならコストは最小限です
2. **昇格の判断をシンプルにする**: スコア 15 点以上は自動昇格。ユーザーは Phase 2 Issue（Deep Research 実行依頼）の一覧を見て「リサーチする / しない」だけを判断すれば良い設計です

:::message
**設計の意図**: 当初は「スコア 15 点以上 → ユーザー確認 → 昇格」の 3 段階でしたが、ユーザーの確認待ちがボトルネックになりました。Phase 2 Issue を作ること自体はコストが低いため、「まず Issue を作り、ユーザーはリサーチするかどうかだけ判断する」方式に変更しています。
:::

### 状態永続化 — Issue コメント方式

`theme-discovery` は Claude Code の Scheduled Tasks で毎日実行されます。しかしクラウド環境ではセッション終了時にファイルシステムが破棄されるため、ファイルベースの状態管理は使えません。

そこで **GitHub Issue のコメントに状態 JSON を書き込む** 方式を採用しています。

```text
Issue #135（theme-discovery 状態管理用の固定 Issue）
  ├── コメント 1: <!-- STATE_JSON --> { ... }  ← 古い状態
  ├── コメント 2: 実行レポート 2026-03-24
  ├── コメント 3: <!-- STATE_JSON --> { ... }  ← 最新の状態
  └── コメント 4: 実行レポート 2026-03-25
```

読み込み時は `<!-- STATE_JSON -->` マーカーで始まるコメントの **最新のもの** を取得し、JSON コードフェンス内のデータをパースします。

```bash
gh api repos/{owner}/{repo}/issues/135/comments \
  --paginate \
  --jq '[.[] | select(.body | startswith("<!-- STATE_JSON -->"))]
        | if length > 0 then last.body else "" end'
```

## 実装詳細

### 情報ソースの設定

巡回先は `config/sources.yaml` で管理しています。

```yaml
rss_feeds:
  - name: "The Verge AI"
    url: "https://www.theverge.com/rss/ai-artificial-intelligence/index.xml"
    category: "AI全般"
  - name: "Anthropic News"
    url: "https://www.anthropic.com/news/rss"
    category: "Anthropic"
  # ...（他のフィード）

x_accounts:
  - handle: "@sama"
    reason: "OpenAI CEO"
  - handle: "@karpathy"
    reason: "AI教育・解説"

keywords:
  high_priority:
    - "LLM"
    - "AI Agent"
    - "RAG"
  medium_priority:
    - "機械学習"
    - "GPU"
```

RSS フィード、X アカウント、キーワードの 3 種類のソースを定義し、WebSearch ツールで巡回します。

### スコアリング基準（25 点満点）

`config/theme-discovery.json` にスコアリングの設定を定義しています。

```json
{
  "scoring": {
    "threshold": 15,
    "max_score": 25,
    "criteria": [
      { "name": "検索需要（トレンド性）", "max": 5 },
      { "name": "競合の少なさ", "max": 5 },
      { "name": "掛け合い適性", "max": 5 },
      { "name": "深掘り余地", "max": 5 },
      { "name": "独自性", "max": 5 }
    ]
  }
}
```

各項目の判断基準は以下の通りです。

| 項目 | 5 点 | 3 点 | 1 点 |
|------|------|------|------|
| **検索需要** | 直近 1 週間で検索急上昇 | 安定的にトレンド | 時事性なし |
| **競合の少なさ** | 日本語 YouTube に解説動画なし | 少数あるが差別化可能 | 多数の解説動画が存在 |
| **掛け合い適性** | 先生×生徒の掛け合いに最適 | 工夫すれば活きる | 掛け合いにしにくい |
| **深掘り余地** | 15 分以上の尺に耐える情報量 | 10 分程度は可能 | 短尺向き |
| **独自性** | 制作者の実体験が活きる | 独自の切り口がある | 誰でも作れる |

### テーマ優先順位フレームワーク（時事性 ×2）

チャンネル戦略として、テーマ選定時には **時事性に 2 倍の重み** を付けています。

```text
スコア = 時事性(0-5) × 2 + 差別化(0-5) + 制作効率(0-5) + シリーズ発展性(0-5)
```

これは `theme-discovery` の 5 軸スコアリング（25 点満点）とは別に、チャンネル運営レベルでの優先判断に使うフレームワークです。AI 分野の時事テーマは検索需要が爆発的に伸びるため、鮮度の高いうちに動画化することが重要です。

### 実行フロー詳解

`theme-discovery` スキルの実行は 6 ステップで構成されます。

**Step 0: 状態の読み込み**

Issue #135 のコメントから最新の状態 JSON を読み込みます。初回実行時や状態がない場合は、既存の `[ネタ候補]` Issue 一覧から初期状態を構築します。

**Step 1: 情報収集**

`config/sources.yaml` に定義された RSS フィード・公式ブログ・arXiv 等を WebSearch で巡回し、新規記事を収集します。

**Step 2: 重複チェック**

収集した記事を `state.known_articles` および既存の `[ネタ候補]` Issue タイトルと照合し、既知の記事やテーマの重複を排除します。タイトルのキーワードが 70% 以上一致する場合はスキップします。

**Step 2.5: Analytics フィードバック参照**

`content/analytics/analytics_summary.json` を参照し、過去動画のパフォーマンスデータをスコアリングに反映します。高 CTR 動画と類似カテゴリには +1 ポイント、低 CTR パターンには -1 ポイントのボーナス / ペナルティを適用します。

**Step 3: スコアリング**

新規記事ごとに 5 軸スコアリングを実施し、15 点以上を「動画化候補」と判定します。情報が不十分なテーマは「継続監視」リストに追加して次回以降の巡回で再評価します。

**Step 4: Issue 化**

スコア 15 点以上のテーマを `[ネタ候補]` Issue として GitHub に登録します。本文にはスコアリング結果・主要ソース・動画化した場合の想定切り口を記載します。

**Step 5-6: リファインメントと状態保存**

未リファインメントの候補が 5 件以上に達したらリファインメントを実行します。最後に状態 JSON と実行レポートを Issue #135 のコメントに投稿します。

:::message
**状態保存は絶対にスキップしない**: Step 6 を省略すると次回実行で重複 Issue が発生します。途中でエラーが発生しても、その時点までの状態を必ず保存します。
:::

### refinement スキルによる自動昇格

スコア 15 点以上のネタ候補は、 `refinement` スキルによって **ユーザー確認なしで自動昇格** します。

```text
1. スコア ≥ 15 の [ネタ候補] Issue を検出
2. 次の Video ID を採番（V001, V002, ... の連番）
3. [V{ID}] Phase 2: Deep Research 実行依頼 Issue を作成
4. Projects V2 に追加 + Video ID フィールド設定
5. 元の [ネタ候補] Issue をクローズ
```

Video ID の採番は、既存の全 Issue タイトルから `[V{数字}]` パターンを検索し、最大番号 + 1 とします。ギャップは作らず、厳密な連番を維持します。

## ハマりポイント

### 1. 状態 JSON のマーカー位置に要注意

Issue コメントの状態 JSON は `<!-- STATE_JSON -->` マーカーで識別しますが、このマーカーは **必ず行頭から始める** 必要があります。先頭に空白が入ると `startswith` でのフィルタリングが失敗し、状態が読み込めなくなります。

```markdown
<!-- STATE_JSON -->     ← OK（行頭）
 <!-- STATE_JSON -->    ← NG（先頭に空白）
```

### 2. WebSearch のレート制限

Claude Code の WebSearch ツールは claude.ai のセッションとレート制限を共有しています。`theme-discovery` で大量の記事を巡回すると、同じ時間帯にユーザーが使っている他のセッションに影響する可能性があります。巡回頻度は 1 日 1 回に抑え、1 回あたりの検索数も最小限にする設計が重要です。

### 3. ネタ候補 Issue にステータスラベルを付けない

ネタ候補 Issue（`phase:1-neta`）には `status:waiting-user` 等のステータスラベルを **付けません**。ネタ候補はバックログ扱いであり、ユーザーの To-Do リストに表示すべきではないからです。ステータスラベルを付けてしまうと、ユーザーが「このネタを見なければ」というプレッシャーを感じてしまいます。

## まとめ

Phase 1 のテーマ発見パイプラインは、以下の設計思想に基づいています。

- **発見と昇格の分離**: 毎日の軽量巡回（Haiku モデル）と、条件を満たしたテーマの自動昇格
- **スコアリングによる客観評価**: 5 軸 25 点満点で定量的に判断し、人間の「なんとなく面白そう」に頼らない
- **Issue コメントによる状態永続化**: セッションが消えるクラウド環境でも確実に状態を引き継ぐ
- **ユーザー負担の最小化**: ユーザーは Phase 2 Issue を見て「リサーチする / しない」だけ判断すればよい
