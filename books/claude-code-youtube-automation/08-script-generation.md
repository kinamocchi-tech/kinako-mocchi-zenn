- 台本 JSON の構造（character, voicevox_speaker_id, emotion, text, visual_cue 等）

## 背景 — 台本自動生成の課題

YouTube テック解説動画の台本を AI に生成させること自体は難しくありません。しかし、以下の課題を同時に解決する必要があります。

1. **キャラクターの一貫性**: 2 体のキャラクターが異なる個性で掛け合いを行う
2. **事実の正確性**: リサーチに基づく情報のみで構成し、ハルシネーションを排除する
3. **尺のコントロール**: 15〜20 分に収まるボリューム（6,000〜8,000 文字）を安定的に生成する
4. **コンテキスト制限**: 8,000 文字の JSON は巨大で、一度に生成するとコンテキストウィンドウが溢れる

本プロジェクトでは、これらを **分割生成 + 自動結合 + 多角的レビュー** のアーキテクチャで解決しています。

## 設計思想

### 台本 JSON 構造

台本は構造化 JSON で管理します。テキストファイルではなく JSON を採用した理由は、下流工程（音声合成・画像生成・動画レンダリング）が台本をプログラマティックに参照するためです。

```json
{
  "video_id": "V002",
  "title": "Anthropicの軌跡",
  "theme": "Anthropic",
  "estimated_duration_min": 17.1,
  "sections": [
    {
      "section_id": "opening",
      "section_title": "オープニング",
      "lines": [
        {
          "id": "001",
          "character": "mocchii",
          "voicevox_speaker_id": 12,
          "voicevox_style": "ふつう",
          "emotion": "neutral",
          "text": "ねえきなこ、飼い主がまた新しいAIサービスに課金してるんだけど。",
          "dialect": null,
          "action": "idle",
          "effect": null,
          "visual_cue": null,
          "fact_check_flags": []
        }
      ]
    }
  ],
  "metadata": {
    "total_characters": 6840,
    "estimated_minutes": 17.1,
    "mocchii_metaphors": ["ハムスターの回し車"],
    "kinako_history_parts": ["OpenAI創業の背景"],
    "owner_teasing": ["飼い主の課金癖"],
    "fact_check_flags": []
  }
}
```

各セリフの主要フィールドを解説します。

| フィールド | 説明 |
|-----------|------|
| `character` | `mocchii` または `kinako` |
| `voicevox_speaker_id` | VOICEVOX のスピーカー ID（キャラ × スタイルで決まる） |
| `voicevox_style` | VOICEVOX のスタイル名（「ふつう」「わーい」「ノーマル」等） |
| `emotion` | 感情タグ。音声パラメータと Remotion のアニメーションに影響 |
| `text` | セリフ本文。推奨66文字以内、67文字超で Warning（フォント縮小）、**100文字超は絶対禁止** |
| `dialect` | 方言指定。`null`（標準語）/ `osaka`（大阪弁）/ `kyoto`（京都弁） |
| `visual_cue` | 画面に表示するコンテンツの指示文（日本語で記述） |
| `fact_check_flags` | 検証が必要な事実情報のフラグ配列 |

### visual_cue の 3 状態

`visual_cue` は画面表示を制御する重要なフィールドで、3 つの状態を取ります。

```
"METR研究の概要を表示"  → 新しいコンテンツを表示
null                    → 直前のコンテンツを継続表示
""                      → コンテンツを意図的にクリア
```

同じ話題が続くセリフでは `null` を設定します。Remotion 側の `getActiveVisualCue()` 関数が自動的に直前のコンテンツを継続表示します。

### 定番枠の設計

すべての動画に以下の定番枠を含めます。定番枠は独立セクションとして配置し、長尺動画の結合時に取り外し可能です。

```
セクション配置順:
  opening    — フック＋テーマ提示（動画固有）
  title_call — チャンネルタイトルコール（毎回固定）
  本編       — 7〜9 セクション（動画固有）
  ending     — まとめ＋飼い主いじり＋コメント誘導（動画固有）
  cta        — CTA＋挨拶（毎回固定）
```

:::message
**opening を title_call より前に配置する理由**: 冒頭にブランディングを置くと視聴者の 55% 以上が離脱するリスクがあります。フック（opening）で興味を引いた後にチャンネル名を提示する構成にしています。
:::

`title_call` と `cta` は **全動画で同じセリフ・同じ音声** を使い回します。

```json
// title_call（固定）
{ "character": "kinako", "text": "きなこもっちーのテック深掘り！" }
{ "character": "mocchii", "text": "今日も一緒に深掘りしよう！" }

// cta（固定）
{ "character": "kinako", "text": "この動画が気に入ったら、チャンネル登録と高評価よろしくね" }
{ "character": "mocchii", "text": "通知オンにしてくれたら関連動画も見逃さないよ！" }
{ "character": "mocchii", "text": "ばいばーい！" }
{ "character": "kinako", "text": "またね。" }
```

## 実装詳細

### 分割生成モード

台本 JSON（6,000〜8,000 文字）を一度に生成するとコンテキストウィンドウが膨張して処理が停止します。そこで **セクション単位で分割生成し、Python スクリプトで結合** する方式を採用しています。

```
Phase A: セクション設計（メインエージェント）
  ↓
Phase B: セクション分割生成（サブエージェント × N 並列、model=opus）
  ↓
Phase C: ファイル書き出し → combine_sections.py で結合
  ↓
Phase D: ボリュームチェック＆品質チェック
  ↓
Phase E: コミット＆push
```

**Phase A** では、メインエージェントがリサーチドキュメントを読み込み、セクション一覧・各セクションの概要と目標文字数・セリフ ID 範囲を決定します。

**Phase B** では、2〜3 セクションずつのバッチでサブエージェント（Opus モデル）を並列起動します。各サブエージェントは 1 セクションを担当し、JSON 文字列をメインエージェントに返します。

:::message
**サブエージェントはファイルに書き込まない**: これは重要な設計ルールです。サブエージェントが直接ファイルを書くと、並列実行で競合が発生します。JSON 文字列をメインに返し、メインが一括で書き出します。
:::

**Phase C** では、セクション JSON を個別ファイルに書き出した後、 `combine_sections.py` で統合します。

```bash
# セクションファイルの配置
content/scripts/V002_sections/
├── 01_opening.json
├── 02_title_call.json
├── 03_birth.json
├── 04_safety.json
├── ...
└── 10_cta.json

# 結合
python3 content/scripts/combine_sections.py V002
```

`combine_sections.py` はセクションファイルを番号順に読み込み、台本 JSON 全体を生成します。メタデータ（総文字数・推定尺）も自動計算します。

**Phase D** は品質ゲートです。`combine_sections.py` は総文字数が 6,000 文字未満の場合に `exit(1)` でエラー終了します（Lv3 物理ゲート）。エラーが出たら不足セクションを特定し、サブエージェントで再生成して再統合します。

### モデル選択と推論設定

台本生成は本プロジェクトで最も品質への影響が大きい工程です。生成されたセリフがそのまま音声・字幕・動画に流れるため、ここでの品質低下は後続の全工程に波及します。そこで、Claude のモデル選択と推論設定を台本生成に最適化しています。

#### Adaptive Thinking の活用

Opus 4.6 と Sonnet 4.6 は **Adaptive Thinking** を搭載しています。これはクエリの複雑さに応じて、モデルが内部の推論深度を自動調整する機能です。単純なタスクでは思考を最小化して高速に処理し、複雑なタスクでは深く考えて品質を確保します。

台本生成では、キャラクターの掛け合い設計・技術概念の比喩表現・セクション間の構成バランスといった複合的な判断が求められます。Adaptive Thinking はこうした複雑なクエリに対して自動的に推論を深めるため、台本の質に直結します。

#### `/effort max` によるフル推論

Claude Code の `/effort` コマンドは推論の深さを 4 段階（`low` / `medium` / `high` / `max`）で制御できます。このうち **`/effort max` は Opus 4.6 および Sonnet 4.6 で使用可能** で、フル推論を実行して最高品質の出力を得られます。

本プロジェクトでは台本生成時に Opus 4.6 + `/effort max` を設定しています。この設定により、以下の品質向上を実感しています。

- **掛け合いの自然さ**: もっちーの素朴な疑問ときなこの丁寧な解説のテンポが改善
- **比喩表現の質**: もっちーの「ハムスターの回し車」的な比喩がテーマに即した的確なものになる
- **構成バランス**: セクション間の情報密度が均一化し、特定セクションだけ薄い問題が減少

:::message
台本生成では **Opus 4.6 + `/effort max`** の組み合わせを推奨します。Sonnet 4.6 でも `/effort max` は使用可能ですが、台本品質の観点から Opus 4.6 を優先します。Sonnet のデフォルトは `/effort high`（API デフォルトと同等）で、`/effort medium` に下げるとトークン節約が可能です。
:::

#### コストと品質のトレードオフ

| モデル | 入力 / 出力（per 1M tokens） | 台本生成での役割 |
|--------|---------------------------|----------------|
| **Opus 4.6** | $5 / $25 | 台本セクションの本文生成（品質最優先） |
| **Sonnet 4.6** | $3 / $15 | PR 作成・レビュー対応・パイプライン制御 |
| **Haiku 4.5** | $1 / $5 | セリフ品質チェック・ファクトチェック等のレビュー |

台本生成のみ Opus を使い、それ以外はコスト効率の良い Sonnet / Haiku を選択するのが基本方針です。Opus の料金は Sonnet の約 1.7 倍ですが、台本は後続工程すべての土台であるため、ここでのコスト増は許容しています。

#### サブエージェントのモデル指定

前述の分割生成モードでは、各セクションを生成するサブエージェントに `model="opus"` を指定しています。セクション生成はキャラクター口調の一貫性と技術解説の正確性を同時に求められるため、Haiku や Sonnet では品質が不十分でした。

一方、チームレビュー（`script-team-reviewer`）のレビュアーはタスクの性質に応じてモデルを分けています。セリフ品質チェッカーとファクトチェッカーは Haiku（パターンマッチ的な検証が中心）、構成・演出レビュアーは Sonnet（文脈理解が必要）を使用しています。この使い分けにより、レビュー工程のコストを抑えつつ必要な精度を確保しています。

### script-pipeline の自動実行フロー

`script-pipeline` スキルは、リサーチ完了テーマの検出から PR マージまでを全自動で実行するメタスキルです。

```
Step 0:  リサーチ完了テーマの検出
Step 0.5: リサーチ内容の保存・裏付けリサーチ
Step 1:  Issue ステータス更新
Step 2:  ブランチ作成
Step 3:  台本生成（script-writer スキル）
Step 4:  チームレビュー（script-team-reviewer スキル）
Step 5:  セルフレビュー
Step 5.5: fact_check_flags の Web 検索裏付け
Step 5.6: fact_check_flags 追加リサーチ
Step 5.7: セリフ改行チェック（字幕シミュレーション）
Step 6:  PR 作成 + AI レビュー依頼
Step 7:  AI レビュー監視・指摘対応
Step 8:  自動マージ
Step 9:  完了報告 + 次 Phase Issue 作成
```

1 回の実行で **1 動画のみ** 処理します。これはセッションタイムアウト防止のための制約です。想定所要時間は 40〜70 分で、fact_check_flags の裏付けに要する Web 検索の回数によって変動します。

### script-team-reviewer — 3 役割並列レビュー

台本生成後、3 つの専門レビュアーを Agent Teams で並列起動してレビューします。

| レビュアー | モデル | チェック項目 |
|-----------|--------|-------------|
| **セリフ品質チェッカー** | Haiku | 文字数・方言ルール・キャラ一貫性・emotion/style 整合性 |
| **ファクトチェッカー** | Haiku | リサーチ突合・fact_check_flags 検証・正式名称・ランク C 扱い |
| **構成・演出レビュアー** | Sonnet | visual_cue 適切性・セクション間の流れ・掛け合いの質 |

指摘は Error / Warning の 2 段階で分類します。

- **Error**: 自動修正。セリフ分割、方言ルール修正、fact_check_flags URL 補完等
- **Warning**: PR 説明文に記載し、ユーザー判断に委ねる

サーキットブレーカーとして、レビュー→修正→再レビューのサイクルは **最大 2 回** までに制限しています。

### 品質スコアの定量化

チームレビュー完了後、5 軸で定量スコア（100 点満点）を算出し、 `content/quality-history.json` に記録します。

| 軸 | 満点 | 主な指標 |
|----|------|---------|
| **ボリューム** | 20 | 6,000〜8,000 文字で満点 |
| **キャラクター品質** | 20 | たとえ話・歴史語り・飼い主いじりの有無 |
| **ファクト品質** | 20 | fact_check_flags 件数・出典 URL 付与率 |
| **演出品質** | 20 | visual_cue 数・セクション間の流れ |
| **コンプライアンス** | 20 | 方言ルール違反 0 件・100 文字超 0 件 |

90〜100 点は Excellent、60 点未満は Fail としてチームレビュー再実行を要求します。このスコア履歴により、台本品質の経時変化を追跡できます。

### ボリュームガイドライン

V001 の実績（目標 15〜20 分に対し実測 10 分 17 秒）を踏まえ、以下の文字数テーブルを策定しています。

| 区分 | 文字数目標 | 尺目安（400 文字/分） |
|------|-----------|---------------------|
| 定番枠合計 | 約 600 文字（固定） | 約 1 分 30 秒 |
| 本編合計 | 5,400〜7,400 文字 | 13 分 30 秒〜18 分 30 秒 |
| **総計** | **6,000〜8,000 文字** | **15 分〜20 分** |

本編セクション数は 7〜9、セクションあたり 700〜900 文字、セリフ数 15〜20 を推奨値としています。

## ハマりポイント

### 1. emotion フィールドが Remotion 型と不一致

台本 JSON の `emotion` に `angry`, `secret`, `nostalgic` 等、Remotion の `Emotion` 型に存在しない値が混入するパターンが繰り返し発生しています（セルフレビュー学習ログ P-02）。有効な emotion 値は以下の固定リストです。

```
normal / neutral / excited / confused / sad /
whisper / tsukkomi / teaching / gentle
```

`angry` は `tsukkomi` に、`secret` は `whisper` にマッピングする等の修正が必要です。セルフレビューの自動検出チェックで不正値を Error として検出します。

### 2. combine_sections.py のボリュームゲートを無視しない

`combine_sections.py` が `exit(1)` で終了したら、「とりあえず進む」は **不可** です。これは Lv3 の物理ゲートであり、6,000 文字未満の台本は YouTube ミッドロール広告の 8 分閾値を下回るリスクがあります。不足セクションを特定し、リサーチから有用な情報を追加して再生成します。

### 3. メタデータの空配列

`_meta.json` の `mocchii_metaphors`, `kinako_history_parts`, `owner_teasing` が空配列のまま提出されるパターン（P-01）が繰り返し発生しています。台本本文には該当箇所が存在するのにメタデータが空のままになるのは、 `generate_review.py` の品質チェックリストが誤表示される原因になります。セルフレビューで 3 フィールドすべてが空配列の場合は Error として検出します。

## まとめ

Phase 3 の台本生成パイプラインは、以下のアーキテクチャで品質と効率を両立しています。

- **分割生成モード**: セクション単位でサブエージェント（Opus）が並列生成し、Python スクリプトで結合。コンテキスト膨張を防止
- **物理的ボリュームゲート**: `combine_sections.py` の `exit(1)` で 6,000 文字未満をブロック。無視して進むことは不可能
- **3 役割並列レビュー**: セリフ品質・ファクト・構成演出を異なる観点から同時レビュー
- **自動パイプライン**: 検出→生成→レビュー→PR→AI レビュー→マージまでをユーザー介入なしで完走

台本 JSON は後続の全工程（音声・画像・動画）の土台です。ここでの品質がチャンネル全体の品質を決定します。次のチャプターでは、この台本 JSON をもとに VOICEVOX で音声を生成する Phase 4 を解説します。
