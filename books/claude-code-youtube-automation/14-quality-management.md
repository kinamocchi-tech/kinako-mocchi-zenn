## このチャプターで学ぶこと

自動化パイプラインの品質は「最後に人が確認する」だけでは担保できません。このチャプターでは以下を解説します。

- PR作成前に問題を自動検出する **セルフレビュースキル**
- 情報の正確性を担保する **ファクトチェック** のフロー
- 「お願い」から「仕組み」へ段階的に制約を強化する **ハーネスエンジニアリング**

## 設計思想 — ハーネスエンジニアリングの品質管理への適用

Ch.04 で解説した **ハーネスエンジニアリング**（制約を Lv1〜Lv4 に分類し、同じミスが繰り返されるたびに上位レベルへ昇格させる手法）は、品質管理においても中心的な役割を担います。詳細は Ch.04 を参照してください。

品質管理の観点では、各レベルが以下の役割を担います。

| Lv | 種別 | 品質管理での役割 |
|----|------|----------------|
| 1 | ドキュメント | 「fact_check_flags の付与ルール」「emotion 有効値リスト」を文書化 |
| 2 | AIセマンティック | `self-reviewer` スキルが PR 前に自動チェック |
| 3 | フック | `post-tool-use-validate.sh` が書き込み時に物理ブロック |
| 4 | CI / 構造テスト | GitHub Actions での品質ゲート（将来実装予定） |

:::message
品質問題が2回以上繰り返されたら「次は気をつける」ではなく、上位レベルへの昇格を必ず検討します。
:::

## セルフレビューの仕組み

`self-reviewer` スキルはPR作成直前に自動実行され、AIレビュアーが指摘しそうな問題を事前に検出します。

### Agent Teams による並列チェック

```text
セルフレビュースキル（オーケストレーション）
├── 台本品質レビュアー（emotion/action 有効値・文字数・方言ルール）
├── ファクトチェックレビュアー（fact_check_flags の付与状況・ランク検証）
└── コード品質レビュアー（TypeScript コンパイル・Python 構文チェック）
```

並列実行によりコストと時間を削減します。個別チェックのサブエージェントには `model="haiku"` を使います。

### 深刻度レベル

- **Error**: PR作成を中断し、必ず修正する
- **Warning**: 修正を推奨するが、スキップの判断も可
- **Info**: 改善提案、任意対応

### 代表的な再発パターン

実際の運用で繰り返し発生したパターンです。

**P-02: emotion フィールドの無効値**

有効な `emotion` 値は以下の9種類のみです。

```text
normal / neutral / excited / confused / sad / whisper / tsukkomi / teaching / gentle
```

`angry` や `secret` はよく混入する無効値です。自動修正候補: `angry` → `tsukkomi`、`secret` → `whisper`。このパターンは複数PRで繰り返されたため、`post-tool-use-validate.sh`（Lv3）に検出ロジックを追加して物理的にブロックするようにしました。

**P-13: タイムゾーン混在バグ**

`datetime.now()` をタイムゾーン指定なしで呼び出し、UTCとJSTが混在するパターンです。YouTube公開スケジュールの重複割り当てというCRITICALなバグを引き起こしました。

```python
# NG
now = datetime.now()

# OK
import pytz
JST = pytz.timezone('Asia/Tokyo')
now = datetime.now(JST)
```

## ファクトチェックの仕組み

### 情報ランク制度

| ランク | 出典 | 扱い |
|--------|------|------|
| A | 公式ブログ・論文・プレスリリース | そのまま台本に使用可能 |
| B | 大手メディアの署名記事 | 台本に使用可能 |
| C | 個人ブログ・SNS・Qiita | ランクA/Bの裏付けがなければ事実として採用不可 |

ランクCのみの情報を断定表現（「〜した」「〜だ」）で書くことは禁止です。

### fact_check_flags の付与

数字・日付・人名・企業名が登場するセリフには `fact_check_flags` フィールドを付与します。

```json
{
  "id": "042",
  "text": "2024年6月のリリース以降、累計100万ユーザーを突破したんですよ。",
  "fact_check_flags": [
    {
      "content": "累計100万ユーザー",
      "source": "https://example.com/press-release",
      "rank": "A"
    }
  ]
}
```

5箇所以上残存する場合は、PRの説明文に「要確認事項が多いため重点レビューをお願いします」と明記します。

### post-tool-use-validate.sh による自動ゲート

台本JSONを書き込む際、`PostToolUse` フックとして登録された `post-tool-use-validate.sh` が自動実行されます（Lv3フック）。

主な検査内容:

```text
1. JSON 構文の妥当性
2. 必須フィールドの存在
3. キャラクター方言ルール違反（もっちーに京都弁、きなこに大阪弁）
4. セリフ文字数チェック（100文字超はブロック、67文字超は警告）
5. 数値・年号を含むのに fact_check_flags が0件（警告）
6. 総文字数が6,000文字未満（ブロック）
```

`exit 2` でブロック、`exit 0 + systemMessage` で警告を使い分けています。フックが `systemMessage` を返すと Claude Code は次のターンで問題を修正してから再度書き込みを試みます。

## AI レビュアーの活用

セルフレビューとフックで事前に品質を担保したうえで、PR 作成後は外部の AI レビュアーによる第三者チェックを実施します。本プロジェクトでは **GitHub Copilot Code Review** と **Gemini Code Assist** の 2 つを全 PR に対して並行で依頼しています。

### Copilot Code Review — エージェント型アーキテクチャへの転換

2026 年 3 月 5 日、GitHub Copilot Code Review は従来の diff ベースのレビューから **エージェント型アーキテクチャ** に転換しました。

従来の方式では、PR の差分（diff）だけを見てコメントを生成していました。新しいエージェント型では、**リポジトリ全体を読み込み、クロスファイルの依存関係を追跡したうえで文脈に基づいたレビュー** を行います。

公式ブログによると、この転換により以下の改善が報告されています。

- レビューコメントの **71% が actionable feedback**（実際に対応が必要な指摘）
- 残りの 29% はノイズと判断して沈黙する設計（不要な指摘を減らす方向に最適化）
- 平均 **5.1 コメント/レビュー**

:::message
エージェント型になったことで、たとえば「台本 JSON のフィールド名を変更した PR」に対して、Remotion コンポーネント側の参照箇所も含めて整合性をチェックできるようになりました。diff だけでは見えないクロスファイルの依存関係を検出できる点が、従来方式との最大の違いです。
:::

出典: [Copilot code review now runs on an agentic architecture](https://github.blog/changelog/2026-03-05-copilot-code-review-now-runs-on-an-agentic-architecture/)（2026 年 3 月 5 日）

### Gemini Code Assist

Google の Gemini Code Assist は、GitHub App としてインストールすると PR 作成時に自動でレビューを実行します。PR コメントで `/gemini review` と投稿すれば手動でも依頼できます。2026 年 3 月時点では Agent mode preview も提供されています。

Copilot と Gemini の両方にレビューを依頼することで、片方が見落とした問題をもう片方が検出する **多重チェック** の効果が得られます。

### レビュー自動監視フロー

PR 作成後のレビュー監視は、GitHub MCP ツールの `subscribe_pr_activity` を使ったイベント駆動で自動化しています。

```text
PR作成
  → Copilot / Gemini にレビュー依頼
  → subscribe_pr_activity でイベント監視開始
  → 0〜10分: レビュー到着を待機、届いたら指摘を精査して対応
  → 15分: 未応答レビュアーに催促（/gemini review + --add-reviewer @copilot）
  → 25分: 催促後も未応答なら「問題なし」と判定
  → 全指摘解消 → squash マージ
```

以前は `sleep` ベースのポーリングを使っていましたが、クラウド環境のセッションタイムアウト（10〜30 分）と競合してレビュー対応が宙に浮く問題が頻発しました。イベント駆動方式に切り替えたことで、セッション切れのリスクを大幅に軽減しています。

セッションが切れた場合でも、次のセッション開始時に `check_pending_pr_reviews.py` がレビュー待ち PR を自動検出し、状態に応じてマージ・催促・指摘対応を再開します。

### サーキットブレーカー

AI レビュアーの指摘対応が収束しない場合に備え、**修正サイクルの上限を 2 回** に設定しています。2 サイクル経過しても解消できない指摘がある場合は自動マージを中断し、ユーザーに判断を仰ぎます。

## ハマりポイント

### フックのグレースフルデグラデーション

外部ツールが利用できない環境でフックがクラッシュすると、書き込み自体がブロックされます。外部ツール依存のチェックには必ず事前確認を入れ、ツールが利用できない場合は Warning のみに留めます。

```bash
if ! npx tsx --version >/dev/null 2>&1; then
  jq -n '{"systemMessage": "⚠️ Warning: npx tsx が利用できないためスキップします。"}'
  # ブロックしない
fi
```

### セルフレビューとフックの役割分担

Lv2（セルフレビュー）と Lv3（フック）で同じ問題を二重検出する場合があります。フックは「書き込み時の物理ゲート」、セルフレビューは「PR作成前の総合チェック」と役割を明確に分けることが重要です。

## まとめ

- **セルフレビュースキル**: Agent Teams による並列チェックでPR前に問題を自動検出
- **ファクトチェック**: 情報ランク制度と `fact_check_flags` で不確実な情報を明示
- **ハーネスエンジニアリング**: 同じミスが繰り返されたら Lv1→Lv2→Lv3 と制約を段階的に強化

AIを使った自動化では、品質管理自体も自動化する必要があります。人のレビューは「機械が見落とした問題の最終確認」という役割に絞ることで、持続可能な自動化の基盤が整います。
