---
title: "40個を量産する ― キャラドリフトとの戦い"
---

前章までで、AI に「正解の顔」を覚えさせるキャラクターシートと、一貫性を担保する LOCKED 構文を手に入れました。ここからが本番です。1 パック分の **40 個** を実際に作り切ります。

ところが、40 枚を続けて生成すると、必ずと言っていいほど **キャラが徐々に変わっていく** 現象に遭遇します。1 枚目はかわいいのに、35 枚目を見たら体型が違う・色が薄い・線が細い。これが本章の主役、**キャラドリフト（character drift）** です。

この章では、まず 40 個のラインナップをどう設計するか、次に量産をどう実行するか、そして最大の難関であるキャラドリフトをどう抑え込むかを、実際に 14 パック以上を販売した経験から具体的に解説します。

:::message
本章は「個別生成（1 枚ずつ作る）」を前提とします。コスト効率の良い「一括シート方式」との比較と使い分けは次章で扱います。本番品質を出すには個別生成が必須、という結論だけ先に共有しておきます。
:::

## ラインナップ設計 ― 40 個の内訳をどう決めるか

「とりあえず 40 個分の表情を考える」とやると、必ず偏ります。挨拶ばかり 20 個、感情表現が 5 個、みたいなアンバランスなパックになり、購入者が実際のトークで使える幅が狭くなります。

そこで、最初に **カテゴリ配分** を設計します。LINE スタンプは「日常会話のリアクションを肩代わりする道具」です。トークで実際に押される頻度から逆算してカテゴリを割り振ります。

### カテゴリ配分の基本形（40 個）

| カテゴリ | 個数 | 役割 | 具体例 |
|---------|------|------|--------|
| 挨拶（greeting） | 6 | 会話の起点。使用頻度が最も高い | おはよう / おやすみ / ありがとう / ごめん / おつかれ / いってきます |
| 感情（emotion） | 10 | リアクションの核。喜怒哀楽を厚めに | うれしい / 悲しい / 怒る / 照れる / 感動 / びっくり |
| 相槌（aizuchi） | 6 | 会話を回す潤滑油。短文で多用される | うんうん / なるほど / それな / マジ? / OK / 了解 |
| ツッコミ（tsukkomi） | 6 | キャラの個性が出る。差別化ポイント | なんでやねん / おい / ちょっと待って / は? |
| 日常シーン（daily） | 8 | 生活感。リピート購入につながる | ごはん / 寝る / 仕事中 / 出かける / 風呂 / 課金して |
| その他・ネタ（misc） | 4 | バズ・話題性。1 枚 2 枚の遊び | 推し活 / 締めのラーメン / 投げ銭ください |
| **合計** | **40** | | |

:::message
LINE の販売単位は **8 / 16 / 24 / 32 / 40 個** です。40 個は審査・後処理の手間が最大ですが、1 個あたりの制作コストが下がり、購入者の満足度も高くなります。初弾を 24 個で出し、反応を見て 40 個に増やすのも堅実な戦略です。
:::

この配分はあくまで基本形で、キャラの性格に応じて調整します。例えばツッコミ役のキャラ（もっちーのようなお調子者）ならツッコミを 8 個に増やし、おしとやかなキャラなら相槌と感情を厚くします。

### base_prompt + expression + pose + text の組み立て

40 個に一貫性を出す鍵は、**プロンプトを 4 つの層に分解する** ことです。1 個ずつ全文をゼロから書くと、必ず表記揺れが起きてキャラがブレます。

```text
最終プロンプト = base_prompt（全 40 個で共通・キャラの不変属性）
              + expression（個別・表情）
              + pose（個別・ポーズ）
              + text（個別・吹き出し文字）
```

- **base_prompt**: キャラの「絶対に変わってはいけない要素」。体型・配色・輪郭線の太さ・スタイル・透過指定。全 40 個で **完全に同じ文字列** を使い回します。ここがブレるとドリフトが加速します。
- **expression**: 目・口・眉の表情。「smiling, eyes closed」「angry, eyebrows raised」など。
- **pose**: 体の動き。「waving one paw」「celebratory jump」など。後述する「ポーズ設計の禁則」がここに効きます。
- **text**: 吹き出しに入れる日本語。後の章で扱う可読性ルール（5 文字以内推奨・濃茶縁取り）が関わります。

base_prompt を固定し、可変部分を expression / pose / text の 3 つだけに絞ることで、40 個を通したブレ幅を最小化できます。

### プロンプトを JSON 化する

40 個分を頭の中やメモ帳で管理するのは破綻します。**JSON で構造化** して、生成スクリプトが機械的に読み込めるようにします。これがプロのワークフローの出発点です。

```json
{
  "version": "v1",
  "character": "mocchi",
  "base_prompt": "A round, mochi-mochi golden hamster character named Mocchi, sticker style, thick clean outline (#3E2314, uniform 5px), soft pastel colors, blue bandana around neck, three tufts of fur on head, white background, transparent PNG, 1:1 aspect ratio",
  "stickers": [
    {
      "id": "M01",
      "text": "おはよう",
      "expression": "smiling, eyes closed in a cheerful way, sun rays around",
      "pose": "waving one paw",
      "category": "greeting"
    },
    {
      "id": "M02",
      "text": "ありがとう",
      "expression": "grateful, sparkling eyes",
      "pose": "polite bow, both paws together in front",
      "category": "greeting"
    },
    {
      "id": "M07",
      "text": "なんでやねん",
      "expression": "exasperated, mouth wide open",
      "pose": "both arms swinging sideways, dynamic motion",
      "category": "tsukkomi"
    },
    {
      "id": "M22",
      "text": "やったー",
      "expression": "overjoyed, eyes wide with excitement",
      "pose": "celebratory jump, both arms raised high",
      "category": "emotion"
    }
  ]
}
```

このファイルを `prompts/mocchi_lineup_v1.json` として保存し、量産スクリプトに渡します。

JSON 化のメリットは 3 つあります。

1. **一貫性の担保**: `base_prompt` が 1 箇所にしかないので、表記揺れが構造的に発生しません。
2. **レビュー容易性**: 40 個の `text` と `category` を一覧でき、配分の偏りやテキスト重複に気づけます。
3. **再現性**: 同じ JSON から何度でも再生成でき、リジェクト時の差し替えも 1 行修正で済みます。

:::message
`id`（M01〜M40 など）は最終的なファイル名（01.png〜40.png）と紐づける管理キーになります。後処理・検証・申請まで一貫してこの ID で追跡できるよう、最初に番号を確定させておきます。
:::

## 量産の実行 ― 1 枚ずつ作って即チェック

ラインナップ JSON が固まったら、いよいよ生成です。ここには鉄則が 4 つあります。

### 鉄則 1: reference_image を必ず渡す

量産で最も効くテクニックが、**リファレンス画像を毎回渡す** ことです。前章で確定したキャラシートの中から「必着要素が最も明瞭に映る 1 枚」を master として選び、`reference_image_base64`（あるいは各 API のリファレンス引数）に渡します。

```bash
python3 tools/generate_stickers.py \
  --lineup prompts/mocchi_lineup_v1.json \
  --reference assets/references/mocchi_master.png \
  --output assets/raw/V1_mocchi/ \
  --provider openai \
  --model gpt-image-2
```

リファレンスを渡すかどうかで、キャラ整合性は劇的に変わります。テキストプロンプトだけだと、AI は「golden hamster」という言葉から毎回少しずつ違う解釈の絵を返します。リファレンス画像があれば「**この子** を別ポーズで描いて」という指示になり、ブレ幅が桁違いに小さくなります。

:::message alert
リファレンスは 3〜5 枚（master 1 枚 + sister 2〜4 枚）に絞ります。9 枚など多すぎると AI にとって識別ノイズになり、かえってブレます。また master は **白背景 RGB 推奨** で、透過 PNG はエラー要因になりやすいので避けます。
:::

### 鉄則 2: 1 枚ずつ生成して即目視

40 枚を一括でバッチ生成し、最後にまとめて確認する ―― これは罠です。**バッチ生成はキャラ崩れに気づきにくい** からです。

1 枚生成したら、その場で目視します。崩れていたら次に進む前に対処します。この「生成 → 即チェック」のリズムを守ることで、後述するドリフトを早期に検知できます。

### 鉄則 3: 崩れたら seed を変えて最大 3 回再生成

AI 画像生成は確率的なので、同じプロンプトでも当たり外れがあります。1 枚が崩れていたら、**seed を変えて再生成** します。ただし無限ループは禁物で、上限は 3 回とします。

```
生成 → 目視
  ├─ OK → 次の 1 枚へ
  └─ NG → seed 変更して再生成（最大 3 回）
         └─ 3 回失敗 → フォールバックへ
```

### 鉄則 4: 失敗続きはフォールバック

3 回再生成しても整合性が出ない場合、それはプロンプトかエンジンの構造的な問題です。次のいずれかにフォールバックします。

- **エンジンを切り替える**（例: Gemini → GPT-Image-2）。balloon 系や細部の線幅再現は GPT-Image-2 が安定します。
- **pose / expression を見直す**（後述の禁則に該当していないか確認）。
- **リファレンスを差し替える**（master が崩れの原因になっていることがあります）。

## キャラドリフト対策 ― この章の核心

ここからが本章の本丸です。**40 枚を続けて生成すると、体型・色・線の太さが累積的にズレていく** ―― これがキャラドリフトです。

### なぜドリフトが起きるのか

リファレンス画像を渡していても、生成結果は「リファレンス 100% コピー」ではなく「リファレンス + プロンプト + ランダム性」の合成です。1 枚ごとにわずかな揺らぎが乗ります。問題は、**そのわずかな揺らぎが累積する** ことです。

具体的には次のような形で進行します。

| 進行 | 症状 |
|------|------|
| 1〜10 枚目 | ほぼ正解。揺らぎは目立たない |
| 11〜25 枚目 | 体型がわずかに痩せる / 配色が薄くなる傾向が見え始める |
| 26〜40 枚目 | 1 枚目と並べると別キャラに見えるほど乖離 |

固定の master 1 枚だけをリファレンスにし続けても、「平均的に少しずつズレた集団」が出来上がってしまうのです。

### 対策: 3 チェックポイントで再キャリブレーション

ドリフトを抑える実践的な方法が、**生成途中でリファレンスを「最も整合性の高い最新の 1 枚」に差し替える** 再キャリブレーションです。10 枚目・25 枚目・40 枚目の 3 点でチェックポイントを設けます。

```
[生成開始] reference = mocchi_master.png（キャラシートの確定 1 枚）
   │
   ├─ 1〜10 枚目を生成
   │
[CP1: 10 枚目] 1〜10 枚を並べて目視
   │   → 1〜10 枚から「最も整合性の高い 1 枚」を選ぶ
   │   → reference をその 1 枚に差し替える（再キャリブレーション）
   │
   ├─ 11〜25 枚目を生成
   │
[CP2: 25 枚目] 11〜25 枚を並べて目視
   │   → 最新の正解 1 枚に reference を差し替える
   │
   ├─ 26〜40 枚目を生成
   │
[CP3: 40 枚目] 全 40 枚を並べて目視
       → 崩れた個別の枚だけ再生成（最大 3 回）
```

ポイントは、**チェックポイントで選ぶのは「キャラシートの master」ではなく「直前 10〜15 枚の中で最も良かった 1 枚」** だという点です。これにより「いま生成しているこのセッションの平均的な絵柄」に正解を寄せ続けることができ、累積ズレをリセットできます。

| チェックポイント | タイミング | アクション |
|----------------|-----------|-----------|
| CP1 | 10 枚目完了時 | 1〜10 枚を目視 → 最良 1 枚を新リファレンスに |
| CP2 | 25 枚目完了時 | 11〜25 枚を目視 → 最新の正解に差し替え |
| CP3 | 40 枚完了時 | 全体を並べて目視 → 崩れた個別枚を再生成 |

:::message
チェックポイントを「目視するだけ」で済ませてはいけません。差し替えという **能動的な操作** までやって初めて再キャリブレーションです。「並べて見たけど大丈夫そう」で進めると、26 枚目以降でドリフトが顕在化します。
:::

### 「最も整合性の高い 1 枚」の選び方

CP1・CP2 で新リファレンスを選ぶときの基準は次のとおりです。

- キャラの **必着要素**（もっちーなら青バンダナ + 3 つの毛束、きなこなら銀メガネ + ネッカチーフ + まつ毛）が **すべて正確に描かれている**
- 体型・配色・線の太さがキャラシートに最も近い
- ポーズが正面寄りで、要素が見切れていない（リファレンスとして情報量が多い）

つまり「最もかわいい 1 枚」ではなく「最も **情報が揃っている** 1 枚」を選びます。

## ポーズ設計の禁則 ― ドリフトを誘発する地雷を避ける

ドリフト・ディテール消失は、運だけでなく **ポーズ設計のミス** でも起きます。第 2 弾きなこ（銀メガネ + ネッカチーフ + まつ毛という複雑なキャラ）の制作で、ポーズと品質の因果関係がはっきりしました。

### 禁則 1: 女性キャラ + 正面合掌

きなこの「ありがとう」系スタンプを正面合掌（両手を顔の前で合わせる）ポーズで作ったところ、**メガネが小さく描かれたり消えたりする** 問題が頻発しました。

根本原因は、AI が「女性キャラ + 正面合掌」という構図に対して、顔まわりのディテールを **簡略化するバイアス** を持っていることです。合掌で顔が隠れ気味になると、メガネのような小物が「省略可能な飾り」と判断されてしまいます。

### 禁則 2: うずくまり / フレームアウト

体を丸めてうずくまるポーズや、体の一部がフレームの外に出るポーズも危険です。**必着要素が見切れる** リスクが高く、ネッカチーフが画面外に出たり、足が切れたりします。

### 対策: 動きのあるポーズを選ぶ

逆に、**ディテール保持の確率が高いポーズ** が存在します。

| 判定 | ポーズ | 理由 |
|------|--------|------|
| ❌ | 女性キャラ + 正面合掌 | メガネ等のディテール省略リスク |
| ❌ | うずくまり / フレームアウト | 必着要素の見切れリスク |
| ✅ | celebratory jump（祝福ジャンプ） | 全身が開き、要素が見えやすい |
| ✅ | wings raised（翼を広げる） | 体のシルエットが明確で簡略化されにくい |
| ✅ | 動きのあるポーズ全般 | AI が体全体を描こうとし、ディテール保持率が高い |

実際、きなこの「ありがとう」は正面合掌をやめて **celebratory jump + 「glasses CLEARLY VISIBLE」の強調** に変更したことで、メガネ消失が解消しました。

:::message
LOCKED 構文や CRITICAL ACCESSORY の指示（前章参照）と、このポーズ設計は **セット** で効きます。プロンプトで「メガネを必ず描け」と命じても、ポーズが省略バイアスを誘発していたら勝てません。命令とポーズの両輪で守ります。
:::

ラインナップ JSON を作る段階で、各 sticker の `pose` に「ディテール簡略化リスクの低い構図」を選んでおくことが、量産での手戻りを劇的に減らします。

## 全数目視チェックの必須性

最後に、量産の品質を最終的に担保する原則です。**40 枚（main・tab を含めれば 42 枚）すべてを 1 枚残らず目視する。** これは絶対です。

### なぜ代表サンプルチェックでは不十分か

「全部見るのは大変だから代表 5〜7 枚だけ確認する」 ―― これで痛い目を見ました。

第 2 弾きなこの制作で、**K33「思考雲スタンプ」** が問題でした。balloon_thought（モコモコの思考雲）の灰色 fill に灰色文字を載せたため、コントラストが不足して文字が読めない状態でした。ところが代表サンプルチェックでは K33 がサンプルに含まれず、**この不合格を見逃しました**。

全数目視に切り替えたところ K33 を発見・救済でき、レイアウトを balloon_thought から balloon_round（白塗り + 焦茶文字）に変更して解決しました。代表チェックのままだったら、不合格スタンプが LINE 申請まで通過していたはずです。

### 「不合格が残る限り次工程・ユーザー確認に回さない」原則

ここから導かれる運用原則が以下です。

:::message alert
不合格画像が 1 枚でも残っている限り、**絶対に次工程（後処理・申請）にもユーザー確認にも進めない**。崩れを全部潰してから次に行く。
:::

この原則の根拠はコストです。LINE の販売手続き（申請）まで進んでから問題が発覚すると、**手戻りコストが跳ね上がります**。審査は 1〜2 週間かかることもあり、リジェクトされれば再申請でさらに数週間。量産段階で 5 分の全数目視を惜しんだ結果、リリースが 1 ヶ月遅れる、という事態になりかねません。

### 全数チェックで見るべき項目

各枚で次を確認します。

- [ ] キャラの必着要素がすべて正確に描かれている（メガネ・バンダナ・毛束など）
- [ ] 体型・配色・線の太さが他の枚とブレていない（ドリフト検知）
- [ ] ポーズで要素が見切れていない
- [ ] 吹き出しの文字が **視覚的に読める**（文字数ルール違反でも、読めれば OK）
- [ ] 表情がカテゴリ（挨拶・感情・ツッコミ等）と合っている

:::message
判定基準は「ルール違反かどうか」ではなく「**視覚的に問題ないか**」です。「文字数が 6 文字でルール上は超過だが、読めるので OK」のように、最終的には目で見て判断します。逆に、読めない・要素が消えている・透過が破綻している枚だけが「不合格＝要対応」です。
:::

## この章のまとめ

- **ラインナップは先にカテゴリ配分を決める**。挨拶 6 / 感情 10 / 相槌 6 / ツッコミ 6 / 日常 8 / ネタ 4 が 40 個の基本形。キャラの性格で微調整する。
- **プロンプトは base_prompt + expression + pose + text の 4 層に分解し JSON 化する**。base_prompt を全枚で固定することが一貫性の土台。
- **量産はリファレンス画像を毎回渡し、1 枚ずつ即目視**。崩れたら seed を変えて最大 3 回再生成、それでもダメならエンジン切り替えなどフォールバックする。
- **キャラドリフトは 10 / 25 / 40 枚の 3 チェックポイントで抑える**。各点で「直前の最も整合性の高い 1 枚」にリファレンスを差し替えて再キャリブレーションするのが核心。
- **ポーズ設計に禁則がある**。女性キャラ + 正面合掌、うずくまり / フレームアウトはディテール消失を誘発。celebratory jump など動きのあるポーズを選ぶ。
- **全数目視は必須**。代表サンプルチェックは K33 のような不合格を見逃す。不合格が 1 枚でも残る限り、次工程にもユーザー確認にも進めない。

40 枚を品質を保って作り切る方法が固まりました。次章では、ここまで前提としてきた「個別生成」と、コスト効率に優れる「一括シート生成」を **$/枚 の実測データ** で正面から比較します。なぜ本番では個別生成を選ぶべきなのか、その境界線を数字で示します。
