ここまでの章で、キャラクターシートの設計から透過処理、文字の可読性、アニメーション対応まで、「売れる形のスタンプ画像」を 42 ファイル揃えるところまで来ました。本章はいよいよ最終関門、**LINE Creators Market への申請**です。

申請はコードを書くフェーズではありませんが、**ここでつまずくと審査でリジェクトされたり、せっかくの作品が検索に埋もれたりします**。実際に 14 パック以上を審査通過・販売した経験から、「机上の説明では絶対に分からない罠」を中心に解説します。特に **申請メタデータの文字数カウント**と **private リポジトリからの ZIP ダウンロード**は、知らないと必ず一度はハマるポイントです。

:::message
本章は静止画スタンプ（40 個・120 円）を基準に説明します。アニメーション（APNG）スタンプは「スタンプタイプ」の選択と価格（240 円〜）が異なるだけで、申請フロー自体は同一です。差分は適宜注記します。
:::

## 申請の全体フロー

まず全体像を頭に入れましょう。LINE Creators Market の申請は、おおむね次の流れで進みます。

```
1. LINE Creators Market にログイン（LINE アカウント）
2. 「新規登録」→「スタンプ」を選択
3. スタンプタイプを選択（スタンプ / アニメーションスタンプ 等・あとで変更不可）
4. 表示情報を入力（English タイトル・説明 → Japanese 追加）
5. 販売情報を入力（クリエイター名・コピーライト・AI 申告・販売エリア・価格）
6. ライセンス証明（写真使用の有無・ZIP アップロード・作品確認 URL）
7. 画像確認（main / tab / 01-40 が正しく並ぶか）
8. 「リクエスト」ボタン → 審査開始
9. 審査通過 → 販売 URL 発行 → 公開
```

タブが多く感じますが、実際に入力すべき項目は多くありません。**事前に画像 ZIP とメタデータをすべて用意しておけば、入力作業は 10〜15 分で終わります**。逆に言えば、準備不足のまま申請画面を開くと「説明文が長すぎてはみ出す」「ZIP がダウンロードできない」といったトラブルで何度もスクロールを往復するハメになります。

:::message alert
LINE の利用規約 12.11 では **BOT・自動操作が明示的に禁止**されています。Playwright や Selenium で申請ボタンの押下まで自動化するのは規約違反でアカウント停止のリスクがあります。画像生成・ZIP パッケージング・メタデータ準備までは自動化してよいですが、**最後のフォーム入力と「リクエスト」押下は人手で行う**のが安全です。
:::

## 申請前チェックリスト

申請画面を開く前に、以下をすべて満たしているか確認します。1 つでも欠けると審査で弾かれます。

```text
[ ] 画像 40 枚 + main.png + tab.png = 合計 42 ファイルが揃っている
    （個数は 8 / 16 / 24 / 32 / 40 のいずれか。アニメは 8 / 16 / 24）
[ ] スタンプ画像が 370 × 320 px 以内（main は 240×240、tab は 96×74）
[ ] 全画像が PNG-24・RGBA・1MB 以下
[ ] 透過が完全（白い縁取り・残留ピクセルがない／内部は完全不透明）
[ ] サイズが偶数 px
[ ] 命名規則準拠（main.png / tab.png / 01.png 〜 40.png のゼロ埋め 2 桁）
[ ] タイトル ≤ 40・説明 ≤ 160・コピーライト ≤ 50（全角は 2 カウント！）
[ ] クリエイター名・コピーライト記入済み・誤字脱字なし
[ ] 価格・販売エリア「全エリア」設定済み
[ ] AI 利用申告フィールドを「使用しています」に設定（人手加工済み）
```

このチェックリストは機械検証スクリプト（`validate_sticker.py`）と目視レビューで担保します。特に **「内部は完全不透明・背景は完全透明」**という LINE の要求は、半透明ピクセルや穴があると即リジェクトになるため、第 7 章・第 9 章で扱った `solidify_alpha` 処理が通っていることを必ず確認してください。

:::message
チェックリストの大半は「画像の物理仕様」です。これらは前章までのパイプラインで自動的に満たされています。本章で人間が注意すべきは、**メタデータ（文字数）**と **ファイルの受け渡し（ZIP）**の 2 点に集約されます。
:::

## ZIP パッケージングと受け渡しの罠

LINE Creators Market は **ZIP 一括アップロード**に対応しています。`main.png` + `tab.png` + `01.png`〜`40.png` を 1 つの ZIP（20MB 以下）にまとめてアップロードすれば、1 枚ずつドラッグする必要はありません。公式 API は存在しませんが、この ZIP アップロードは標準機能です。

### ZIP 化スクリプト

リポジトリの `tools/pack_zip_for_line.py` で ZIP を生成します。

```bash
# 完成画像ディレクトリを LINE 提出用 ZIP にまとめる
python3 tools/pack_zip_for_line.py \
  --input assets/final/V1_mocchi/ \
  --output dist/V1_mocchi_submission.zip
```

このスクリプトはファイル名の検証（命名規則・個数）も兼ねるので、ここで `40 ファイル + main + tab` が揃っていなければエラーになります。**ZIP 化が通った時点で、命名規則とファイル数は機械的に保証された**と考えてよいです。

### private リポジトリからの DL URL の罠（最重要）

ここが多くの人がハマる落とし穴です。申請は **スマホで行うのが現実的**です（LINE アカウント連携・本人確認がスマホ前提のため）。すると「PC で ZIP を作る → スマホに渡す」というブリッジが必要になります。GitHub の private リポジトリに ZIP を置いて URL でダウンロードしようとすると、**URL の形式によって 404 になる**のです。

| URL 形式 | private リポでの挙動 |
|---|---|
| `https://raw.githubusercontent.com/{owner}/{repo}/{branch}/dist/x.zip` | ❌ **404 になりがち**（トークン認証が必要で、スマホブラウザでは通らない） |
| `https://github.com/{owner}/{repo}/raw/{sha}/dist/x.zip` | ✅ ログイン済みブラウザなら通る（`{sha}` でコミット固定） |
| `https://github.com/{owner}/{repo}/raw/{branch}/dist/x.zip` | ✅ ブランチ指定でも可（マージ前は作業ブランチを指定） |

つまり **`raw.githubusercontent.com` ではなく `github.com/.../raw/...` 形式を使う**のが鉄則です。さらにブランチ名よりも **コミット SHA で固定**したほうが、マージや force-push でリンクが切れる事故を防げます。

```text
# ❌ 404 になりやすい（スマホで開くと認証で弾かれる）
https://raw.githubusercontent.com/kai-kou/kinako-mocchi-stickers/main/dist/V1_mocchi_submission.zip

# ✅ ログイン済みブラウザで通る（github.com/.../raw/ 形式）
https://github.com/kai-kou/kinako-mocchi-stickers/raw/main/dist/V1_mocchi_submission.zip

# ✅ 作業ブランチがまだ main にマージされていない場合
https://github.com/kai-kou/kinako-mocchi-stickers/raw/claude/serene-goodall-fNRbm/dist/V7_mocchi_strip24_submission.zip
```

:::message alert
URL の罠を回避する最も確実な方法は、**ZIP をチャットや Slack で直接スマホに送る**ことです。Claude Code の `SendUserFile` のようなファイル直送機能や、Slack へのファイルアップロードを使えば、private リポの認証問題を完全に回避できます。URL ダウンロードで 1 度でも詰まったら、迷わず直送に切り替えてください。
:::

## 申請メタデータの文字数カウントの罠（最重要）

本章で**最も多くの人がつまずくのがここ**です。LINE のメタデータ入力欄には文字数制限がありますが、その**カウント方法が直感に反します**。

### 全角 1 文字 = 2 カウント

LINE は **全角 1 文字を 2 カウント、半角 1 文字を 1 カウント**で数えます。したがって各項目の実質的な日本語の文字数は次のようになります。

| 項目 | 公称上限 | 日本語の実質上限 | 英語（ASCII） |
|---|---|---|---|
| タイトル | 40 | **約 20 文字** | 40 文字 |
| 説明文 | 160 | **約 80 文字** | 160 文字 |
| コピーライト | 50 | 約 25 文字 | 50 文字 |

「タイトルは 40 文字まで」と思って日本語を 40 文字書くと、80 カウントになって**半分以上はみ出します**。たとえば「もっちー（ハムスター）の動くスタンプ」は 18 文字ですが、全角ばかりなので 36 カウント（上限 40 にギリギリ収まる）です。

### unicodedata で正確にカウントする

事前に Python でカウントを検証しておけば、申請画面で「文字数オーバー」に悩むことはなくなります。`unicodedata.east_asian_width` を使い、`W`（Wide）・`F`（Fullwidth）・`A`（Ambiguous）を 2、それ以外を 1 として合計します。

```python
import unicodedata

def line_count(s: str) -> int:
    """LINE のメタデータ文字数カウント（全角=2・半角=1）。"""
    total = 0
    for ch in s:
        # W=Wide, F=Fullwidth, A=Ambiguous は全角扱いで 2 カウント
        if unicodedata.east_asian_width(ch) in ("W", "F", "A"):
            total += 2
        else:
            total += 1
    return total

# 検証例
title_ja = "もっちー（ハムスター）の動くスタンプ"
desc_ja  = "もっちー（ハムスター♂）が動く24個セット！挨拶・了解・お疲れから、頬袋もぐもぐ・大阪弁ツッコミ・目ぐるぐるまで、毎日のトークが楽しくなる動き！"

print(line_count(title_ja), "/ 40")   # → 36 / 40  ✅
print(line_count(desc_ja),  "/ 160")  # → 制限内かを事前に確認
```

:::message
`A`（Ambiguous）を 2 カウントに含めるのが安全側の判断です。一部の記号（`♂`・`！`・全角括弧など）は環境によって幅が揺れますが、LINE の実カウントに合わせるには「迷ったら 2」で見積もると上限超過を確実に防げます。
:::

### 1 個ずつコピペできる自己完結フォーマットで用意する

文字数を検証したら、**各項目を「1 個ずつコピペできる」形で用意**しておきます。これは地味ですが極めて重要です。申請する人（あるいは飼い主）が、スマホの申請画面とメモを行き来しながら、**該当のコードブロックを 1 つタップしてコピー → フォームに貼り付け**できる状態にするのです。説明をまとめて 1 つの文章に書いてしまうと、申請者がスクロールを往復して該当部分を切り出す手間が発生し、ミスの温床になります。

理想は次のように、LINE フォームの入力順そのままに、項目ごとに独立したコードブロックを並べることです。

````text
**① スタンプタイプ**（あとで変更不可）
```
スタンプ
```
**② タイトル（English）**
```
Mocchi the Hamster - Daily Stickers
```
**③ スタンプの説明（English）**
```
40 daily expressions of Mocchi the golden hamster. Greetings, replies, family chat!
```
**④ タイトル（日本語）**
```
もっちー（ハムスター）の毎日スタンプ
```
**⑤ スタンプの説明（日本語）**
```
YouTube『きなこもっちーのテック深掘り』のもっちー40個セット。挨拶・返信・関西弁・家族用など日常で使えるラインナップ！
```
**⑥ クリエイター名**
```
kinako-mocchi
```
**⑦ コピーライト**
```
© kinako-mocchi
```
````

このフォーマットを各パックごとに作っておけば、申請は「上から順にコピペするだけ」の作業になります。**要約版（タイトル・価格だけ書いて「あとはガイド参照」）は禁止**です。申請者がチャット画面だけでフォームを上から下まで埋められる、完全自己完結の状態を目指してください。

## 商品情報（表示情報）の書き方

文字数の制約を踏まえた上で、SEO に効く商品情報を組み立てます。LINE スタンプは **ストア内検索とサジェスト**で見つけてもらえるかどうかが初動を左右するため、タイトルと説明文へのキーワード仕込みが重要です。

### タイトル

`キャラ名 + 用途キーワード`の形が基本です。

```text
もっちー｜ハムスターの日常スタンプ
```

「もっちー」（キャラ名）に加えて「ハムスター」（動物名）「日常」（用途）を入れることで、キャラ名を知らない人が「ハムスター スタンプ」で検索したときにも引っかかります。**キャラ名だけのタイトルは検索流入を捨てている**と考えてください。

### 説明文

`個性 + 想定シーン + キーワード`で構成します。実質 80 字しか使えないため、無駄な前置きは省いて要素を詰め込みます。

```text
YouTube『きなこもっちーのテック深掘り』のもっちー（ハムスター♂）40個セット。
挨拶・返信・関西弁・家族用など日常で使えるラインナップ！
```

### クリエイター名とコピーライト

クリエイター名はチャンネル名・シリーズ名で統一します（複数パックを出すと「同じ作者の他のスタンプ」として回遊が生まれるため）。コピーライトは `© 2026 kinako-mocchi` の形が無難です。

| 項目 | 入力例 |
|---|---|
| クリエイター名 | `kinako-mocchi` |
| コピーライト | `© kinako-mocchi` または `© 2026 kinako-mocchi` |

### SEO に効くキーワードの考え方

タイトル・説明文には、次の 4 カテゴリのキーワードを意識的に散りばめます。

| カテゴリ | 例 |
|---|---|
| 動物名 | ハムスター・インコ・セキセイインコ |
| 用途 | 日常・ビジネス・敬語・カジュアル・家族 |
| 感情 | かわいい・ほっこり・ゆるい・もちもち |
| シーン | 職場・家族・友達・飲み会・推し活 |

加えて、申請画面には**個別スタンプ説明**（各 1 枚ごとの説明・任意）の欄もあります。40 枚分すべてに「おはよう」「了解」などの用途を記載しておくと、検索流入の表面積が増えます。手間はかかりますが、SEO 効果は無視できません。

## 販売情報

### 価格

静止画スタンプは **120 円**が基本です。最安価格は初動のダウンロード数を最大化しやすく、新着 TOP 表示や口コミの起点を作りやすいためです。アニメーションスタンプは下限が **240 円**なので、その価格になります。

| スタンプタイプ | 推奨価格 |
|---|---|
| 静止画スタンプ | 120 円 |
| アニメーションスタンプ | 240 円 |

価格戦略の詳細（取り分・収益のリアル）は次章で扱います。ここでは「静止画は 120 円、アニメは 240 円」と覚えておけば十分です。

### 販売エリアは「全エリア」を必ず ON

販売情報で**絶対に間違えてはいけない**のが販売エリアです。デフォルトのまま「日本のみ」で出してしまうと、海外売上をまるごと取り逃がします。

:::message alert
販売エリアは **「販売可能な全てのエリア」を必ず ON** にしてください。LINE は **台湾・タイ・インドネシア**が主要な海外市場で、日本以外からの購入も少なくありません。全エリア ON は設定コスト 0 で売上機会を増やせる、純粋な得です。日本のみに絞る理由は一切ありません。
:::

| エリア | 設定 | 理由 |
|---|---|---|
| 日本 | ✅ | 主要市場 |
| 台湾 | ✅ | LINE 主要海外市場 |
| タイ | ✅ | LINE 主要海外市場 |
| インドネシア | ✅ | LINE 主要海外市場 |
| **全エリア** | ✅ **必ず ON** | 設定コスト 0 で売上機会増 |

その他の販売情報の典型的な設定値は次の通りです。

| 項目 | 設定値 |
|---|---|
| テイストカテゴリ | カワイイ・キュート |
| キャラクターカテゴリ | その他 |
| LINEスタンププレミアム | 参加する |
| 販売開始設定 | 自動で販売を開始 |
| 写真の使用 | ❌ 使用していません |

## [AI] ラベルと AI 利用申告

AI 画像生成でスタンプを作る本書の読者にとって、**ここは死活的に重要**です。

### 2025 年 6 月以降のルール

2025 年 6 月以降、**AI で生成または加工したスタンプには [AI] アイコンが自動で付与**されるようになりました。これはペナルティではなく表示ラベルですが、申請時に **「AI を使用した」かどうかを正直に申告**する必要があります。

:::message alert
申請画面の「AI の使用」欄は **「AI を使用しています」を選択**してください。これを「使用していません」で出すと、**虚偽申告としてリジェクト対象**になります。AI で生成した事実は審査側にも分かるため、隠そうとするのは逆効果です。
:::

### AI 生成のみは審査落ちリスク → 人手加工が必須

もう 1 つの落とし穴は、**「AI が出力したまま」のスタンプは審査落ちのリスクが高い**という点です。LINE は AI 生成コンテンツに対して「人の手による加工」を求めています。本書のパイプラインでは、以下の工程がすべて「人手加工」に該当します。

- 背景除去（rembg による透過処理）
- リサイズ・余白調整・PNG-24 RGBA 最適化
- テキスト・吹き出しの追加と可読性調整
- 色調整・縁取り（白縁禁止 → 濃茶 #3E2314 など）

つまり、**前章までのパイプラインを通している時点で、必要な人手加工は満たされています**。逆に「AI 出力をそのまま 40 枚並べて申請」は通りません。AI を使うこと自体は問題なく、市場の受容度も高い（LINE 公式の「LINEスタンプメーカー」にも AI 機能が内蔵されています）。**正直に申告し、必ず加工を加える**——この 2 点を守れば問題ありません。

## よくあるリジェクト理由 7 つ

実際の審査で頻発するリジェクト理由を、原因と対策のセットでまとめます。申請前にこの 7 項目を自己点検してください。

| # | リジェクト理由 | 原因 | 対策 |
|---|---|---|---|
| 1 | 既存キャラへの類似 | ちいかわ・うさまる・ポケモン等に造形が寄っている | キャラシートから造形を調整（第 3 章）。「丸い動物」は似やすいので独自要素を強化 |
| 2 | 背景透過の不完全 | 白い縁取り・残留ピクセル・半透明 | rembg + 内部不透明化（`solidify_alpha`）を確認。白縁は禁止し濃茶縁取りに |
| 3 | 画像サイズ違反 | 奇数 px・最大サイズ超過・規定外個数 | `validate_sticker.py` で偶数 px・370×320 以内・個数を機械チェック |
| 4 | AI 生成のみで人手加工なし | AI 出力をそのまま提出 | 背景除去・テキスト追加・色調整など人手加工を加える。[AI] 申告も忘れず |
| 5 | 暴力・性的表現 | 年齢制限内でも過激な描写 | 公序良俗の範囲を守る。攻撃的・性的なモチーフを避ける |
| 6 | テキストの誤字脱字 | スタンプ内文字 or 申請メタの誤字 | 申請前に全 40 枚 + メタデータを音読チェック |
| 7 | メイン画像とスタンプの世界観不一致 | main / tab とスタンプ本体のトーンがズレている | main / tab を本体と同じキャラ・配色・線で作る（第 9 章） |

:::details リジェクトされても落ち込まなくてよい理由

## 著作権・モラルの要点

リジェクト回避と並んで、そもそも**規約・法令に触れない**ことが大前提です。以下を必ず守ってください。

- **他人の作品を使わない**：既存キャラ・著名人の写真・他者のイラストを流用しない。自分のオリジナル作品のみ。
- **商標・ブランド名を含めない**：Apple / Google / 任天堂などの企業名・ロゴ・製品名を画像や文字に入れない。ペンライト・うちわなどの小物も、ロゴや特定の名前を入れず**無地**にする（パブリシティ権・商標リスクの排除）。
- **公序良俗に反しない**：差別・誹謗中傷・性的・暴力的な内容、政治・宗教的に偏った表現を避ける。

:::message
AI 生成は「学習元の作品に似てしまう」リスクを内包します。出力が既存キャラに寄っていないか、第三者の目（本書の例では `@kinako` / `@mochi` のオブザーバーレビュー）で確認する工程を入れると、類似リジェクトを未然に防げます。
:::

## 申請後にやること

「リクエスト」ボタンを押したら審査開始です。審査を通過すると **販売 URL**（`https://line.me/S/sticker/XXXXXXX` の形式）が発行されます。この URL は次章のマーケティングで使う重要な資産なので、必ず控えておきます。

販売が始まったら、次章で扱う「価格・収益のリアル」と「発売 48 時間の動き方」に進みます。1 パックの純利益は数百〜千円規模で、稼ぎは累積で伸びる構造です——その現実的な数字を、誇張なく次章で見ていきましょう。

## この章のまとめ

- 申請フローは「ログイン → スタンプタイプ選択 → 表示情報 → 販売情報 → ライセンス証明（ZIP）→ 画像確認 → リクエスト」。事前準備さえ整えば入力は 10〜15 分。
- **文字数カウントは全角 1 文字 = 2 カウント**。タイトル 40 は日本語実質約 20 字、説明 160 は約 80 字。`unicodedata.east_asian_width` で `W/F/A` を 2 として事前検証し、項目ごとに 1 個ずつコピペできる自己完結フォーマットで用意する。
- ZIP は `pack_zip_for_line.py` で生成。private リポの DL は `raw.githubusercontent.com` だと 404 になるので **`github.com/{owner}/{repo}/raw/{sha}/...`** を使うか、**ファイル直送が最も確実**。
- 販売情報では **販売エリア「全エリア」を必ず ON**（台湾・タイ・インドネシアが主要海外市場）。価格は静止画 120 円・アニメ 240 円。
- **AI 利用は正直に申告**（[AI] ラベル自動付与）。AI 生成のみは審査落ちリスクなので、背景除去・テキスト追加・色調整など**人手加工を必ず加える**（パイプラインを通せば自動的に満たされる）。
- リジェクト 7 大理由（既存キャラ類似・透過不完全・サイズ違反・人手加工なし・暴力性的・誤字脱字・世界観不一致）と著作権・モラル（他人の作品・商標・公序良俗）を申請前に自己点検する。

次章では、審査を通って販売が始まったあとの**価格・収益のリアル**と、初動を左右する**リジェクト対策・販売戦略**を、実測値とともに掘り下げます。
