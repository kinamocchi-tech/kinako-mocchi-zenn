LINE スタンプ制作で、AI 画像生成の次に多くの人がつまずくのが **背景透過** です。「`rembg` で背景を抜いたから大丈夫」と思って申請すると、審査でこう返ってきます。

> ※ 背景の一部が部分的に透過漏れ ＞ IMG17
> ※ イラストの内部が透過されています ＞ 該当多数

これは机上の話ではなく、実際に第3弾アニメスタンプ（V7・もっちー24個）で受けた **本物のリジェクト文言** です。アップロード自体は通っていたのに、審査で弾かれました。本章では、なぜこれが起きるのか、そして「内部不透明化（solidify）」という後処理がなぜ必須なのかを、実装コードと検証手順込みで解説します。

## LINE の透過要件 ― 「内部は完全不透明・背景は完全透明」

まず、LINE がスタンプ画像に求める透過の定義を正確に押さえます。

| 領域 | 求められる α 値 | NG な状態 |
|------|----------------|-----------|
| イラスト内部（キャラ本体・塗り） | **完全不透明（α = 255）** | 内部に穴（α = 0）／半透明（0 < α < 255）がある |
| 背景（キャラの外側） | **完全透明（α = 0）** | 背景に塗り残し・透過漏れがある |

つまり「アルファチャンネルが 0 か 255 の二択」で、**中間値（半透明）はイラスト内部に一切あってはならない**、という二値的な要件です。

ところが AI 生成 → `rembg` という素直なパイプラインを組むと、この要件を **自動では満たせません**。理由は次の節で説明する `rembg` のマット（matte）の性質にあります。

:::message
LINE の透過要件は「見た目で透けていなければ OK」ではありません。画素単位で α=0 / 255 になっているかを審査側が機械的に見ています。プレビューで普通に見えても、内部に α=128 の画素が散っていればリジェクトされます。
:::

### V7 で実際に起きたこと

V7 では、`rembg`（BiRefNet）で背景を抜いた APNG をそのまま申請しました。プレビューでは綺麗に見えたので問題ないと判断したのですが、審査で「イラストの内部が透過されています（該当多数）」と返ってきました。

原因を調べると、キャラのシルエット内部に **α=0 の小さな穴** と **0 < α < 255 の半透明画素** が点在していました。BiRefNet のマットがエッジ周辺や陰影の濃い部分で「ここは背景かもしれない」と判断し、α を中途半端に下げていたのです。人間の目には透けて見えないレベルでも、LINE の審査はこれを「内部透過」と判定します。

## rembg（BiRefNet）による背景除去の基本

背景除去のベースには `rembg` を使います。複数のモデルが選べますが、本書では **`birefnet-general`** を採用します。

```bash
# 依存は requirements.txt に集約（rembg[cli] + onnxruntime CPU バックエンド）
pip install -r requirements.txt

# 1枚ずつ背景除去（birefnet-general モデルを明示指定）
for f in assets/raw/V1_mocchi/*.png; do
    rembg i -m birefnet-general "$f" "assets/processed/V1_mocchi/$(basename $f)"
done
```

### なぜ birefnet-general を使うのか

`rembg` のデフォルトモデルは `u2net` ですが、スタンプ用途では `birefnet-general` のほうが結果が安定します。

| モデル | 特徴 | スタンプ用途での評価 |
|--------|------|---------------------|
| `u2net` | 軽量・高速。人物写真向けに学習 | イラスト輪郭が甘く、髪・耳・細い手足が溶けやすい |
| `isnet-general-use` | 汎用。u2net より輪郭が硬い | 中庸。細部の取りこぼしは残る |
| **`birefnet-general`** | 高精細なマット推定。エッジ追従が強い | **イラストの細い輪郭（耳・ヒゲ・指）を保持しやすく本書の標準** |

BiRefNet は境界の精度が高く、キャラクターの細いパーツを潰しにくいのが採用理由です。ただし「精度が高い」ことと「内部が完全不透明になる」ことは **別問題** です。BiRefNet は陰影や濃い塗りを背景候補と誤推定して内部の α を下げることがあり、これが後述の「内部の穴・半透明」を生みます。

:::message alert
モデルを `birefnet-general` に変えても、内部透過の問題は **解決しません**。BiRefNet は輪郭精度が高いだけで、「イラスト内部を必ず不透明にする」保証はないからです。背景除去はあくまで第一段階で、この後の `solidify_alpha` がリジェクト回避の本命です。
:::

## 内部不透明化（solidify_alpha）が必須

`rembg` の出力をそのまま使うとリジェクトされる――この穴を塞ぐのが **内部不透明化（`solidify_alpha`）** です。V7 リジェクト後、`build_apng_line.py` の `solidify_alpha()` を全フレームに適用する形で恒久対策しました。処理は次の 4 ステップで構成されます。

| ステップ | 処理 | 目的 |
|---------|------|------|
| 1 | 小さな不透明孤立島を除去 | rembg が背景側に残した「ゴミ（背景漏れ）」を消す |
| 2 | 内部の穴を `binary_fill_holes` で埋める | シルエット内部の α=0 の穴を不透明化 |
| 3 | シルエット内を α=255 に2値化 | 半透明（0 < α < 255）を完全不透明に潰す |
| 4 | 黒フリンジを近傍色で距離変換補間 | 輪郭の暗い縁取りを内側の色で塗り直す |

### ステップ 1: 小さな不透明孤立島を除去（背景漏れ対策）

`rembg` は背景の一部を消し損ねて、キャラから離れた場所に小さな不透明の塊を残すことがあります。これを連結成分（connected components）でラベリングし、面積が閾値以下の島を背景（α=0）に落とします。

```python
import numpy as np
from scipy import ndimage

def remove_islands(alpha: np.ndarray, min_area: int = 64) -> np.ndarray:
    """不透明な孤立島のうち小さいものを背景に落とす。"""
    mask = alpha > 0
    labeled, n = ndimage.label(mask)
    if n == 0:
        return alpha
    # 各連結成分の面積
    sizes = ndimage.sum(mask, labeled, range(1, n + 1))
    # 最大成分（=キャラ本体）は必ず残す
    keep = np.zeros_like(alpha)
    main_label = int(np.argmax(sizes)) + 1
    for lbl in range(1, n + 1):
        if lbl == main_label or sizes[lbl - 1] >= min_area:
            keep[labeled == lbl] = alpha[labeled == lbl]
    return keep
```

### ステップ 2 & 3: 内部の穴埋めと2値化

ここが核心です。`scipy.ndimage.binary_fill_holes` で「シルエットに囲まれた α=0 の穴」を不透明側に塗りつぶし、その内側を一律 α=255 に2値化します。これで半透明画素も穴も同時に消えます。

```python
def solidify_alpha(rgba: np.ndarray, alpha_thresh: int = 8) -> np.ndarray:
    """イラスト内部を完全不透明（α=255）にし、背景を完全透明（α=0）にする。"""
    alpha = rgba[:, :, 3]

    # (1) 背景漏れの孤立島を除去
    alpha = remove_islands(alpha, min_area=64)

    # シルエット（少しでも不透明な領域）を二値マスク化
    silhouette = alpha > alpha_thresh

    # (2) 内部の穴（シルエットに囲まれた透明領域）を埋める
    filled = ndimage.binary_fill_holes(silhouette)

    # (3) 内部は α=255、外部は α=0 に2値化（半透明を完全に潰す）
    rgba[:, :, 3] = np.where(filled, 255, 0).astype(np.uint8)
    return rgba
```

`binary_fill_holes` が効くのは、「内部の穴」が **シルエットによって完全に囲まれている** ためです。背景（キャラの外側の透明領域）は画像の縁に繋がっているので穴とは見なされず、埋められません。これにより「内部だけ不透明化・背景は透明のまま」という LINE 要件を正確に再現できます。

### ステップ 4: 黒フリンジを近傍色で補間

2値化すると、輪郭ぎりぎりに残っていた暗い半透明画素（黒フリンジ）が不透明側に取り込まれ、縁が黒く縁取られて見えることがあります。これを内側の色で塗り直します。距離変換（`distance_transform_edt`）で「最も近い確実な内部画素」を引いてきて、フリンジの RGB をそれで置き換えます。

```python
def fix_dark_fringe(rgba: np.ndarray, dark_thresh: int = 40) -> np.ndarray:
    """輪郭の黒フリンジを最近傍の内部色で補間する。"""
    rgb = rgba[:, :, :3]
    alpha = rgba[:, :, 3]
    opaque = alpha == 255

    # 暗すぎる画素（黒フリンジ候補）を「埋め直す対象」とする
    luminance = rgb.max(axis=2)
    fringe = opaque & (luminance < dark_thresh)

    # 信頼できる内部画素から最近傍の座標を引く（距離変換）
    valid = opaque & ~fringe
    # valid=False の画素について最近傍の valid 画素インデックスを得る
    idx = ndimage.distance_transform_edt(
        ~valid, return_distances=False, return_indices=True
    )
    rgba[:, :, :3] = rgb[tuple(idx)]
    rgba[:, :, 3] = alpha  # α は維持
    return rgba
```

:::message
**黒フリンジ補間はキャラ輪郭の意図した黒線まで消さないよう閾値を慎重に**。本来の太い黒アウトラインは内部色で塗りつぶしてはいけません。`dark_thresh` を低め（40 程度）にして、「輪郭線そのもの」ではなく「半透明由来の薄暗いフリンジ」だけを対象にします。配色によっては閾値調整が必要です。
:::

これら 4 ステップを通すと、出力は「α が 0 か 255 のどちらかしかない・内部は完全不透明・背景は完全透明」になり、LINE の内部透過リジェクトを回避できます。V7 ではこの `solidify_alpha()` を build 時に全フレーム自動適用する形で恒久化しました。

## 検証の罠 ― APNG 差分フレームと「偽の内部穴」

ここで、アニメ（APNG）特有の **検証の落とし穴** を共有します。これは知らないと「直したのに検証で穴が出続ける」という無限ループにハマります。

APNG を `apngasm` 系で生成すると、2フレーム目以降は **差分フレーム（前フレームとの変化分だけ・可変サイズ）** として格納されます。フルの画像データ（IDAT）を持つのは **フレーム 0 だけ** です。

この APNG を PIL（Pillow）の `seek(i)` で中間フレームや最終フレームを読み込むと、差分フレームの「変化していない領域」が透明（α=0）として読まれ、**実際には存在しない大量の内部穴** が検出されてしまいます。

```python
from PIL import Image
import numpy as np

im = Image.open("assets/final/V7_mocchi_anim/A01.png")

# ❌ 罠: 中間/最終フレームを seek で読むと偽の内部穴が出る
im.seek(3)
arr = np.array(im.convert("RGBA"))
holes = ((arr[:, :, 3] == 0) & internal_mask).sum()  # 偽陽性で大量に出る

# ✅ 正解: frame 0（フル IDAT）だけで検証する
im.seek(0)
arr0 = np.array(im.convert("RGBA"))
holes0 = ((arr0[:, :, 3] == 0) & internal_mask).sum()  # 正しい値
```

V7 でも、最初これに引っかかって「`solidify_alpha` を入れたのに穴が消えない」と誤認しました。実際には穴は消えていて、**検証コードが差分フレームを読んでいた** のが原因でした。

:::message alert
APNG の透過品質検証は **必ず frame 0（フル IDAT フレーム）で行う**。`solidify_alpha` は全コマに一律適用されるので、frame 0 が合格していれば代表として全フレーム合格と判断できます。中間フレームを `seek` で読んだ穴・半透明は「差分フレームのアーティファクト」であり、実際の不良ではありません。
:::

## 透過品質の機械チェックと目視確認

後処理を入れたら、必ず **機械チェック** と **目視確認** の両方を通します。片方だけでは見落とします。

### 機械チェック（holes / semi の検出）

`build_apng_line.py` の `verify()` 相当の処理で、frame 0 に対して「内部の穴」と「半透明画素」をカウントし、1個でもあれば FAIL にします。

```python
def verify_transparency(rgba: np.ndarray) -> dict:
    """内部の穴・半透明画素を機械チェックする。frame 0 に対して実行すること。"""
    alpha = rgba[:, :, 3]

    # シルエット内部を埋めたマスク（内部判定の基準）
    silhouette = alpha > 8
    filled = ndimage.binary_fill_holes(silhouette)

    # 内部の穴: filled だが α=0 の画素
    holes = int((filled & (alpha == 0)).sum())
    # 半透明: 内部にある 0 < α < 255 の画素
    semi = int((filled & (alpha > 0) & (alpha < 255)).sum())

    return {"holes": holes, "semi": semi, "pass": holes == 0 and semi == 0}
```

`holes` も `semi` も **0** であることを申請の必須ゲートにします。1個でも残っていれば内部透過リジェクトの種になります。

### 目視確認（α マスク可視化とグレー背景プレビュー）

機械チェックは「数」を見ますが、「縁の見え方」までは判断しきれません。次の 2 種類のプレビューを目視します。

| 確認方法 | やり方 | 見るポイント |
|---------|--------|-------------|
| **α マスク可視化** | 不透明画素（α=255）を赤、透明を透明で塗ったマスクを重ねる | 内部が一様に赤いか（穴・半透明が黒く抜けていないか） |
| **グレー背景プレビュー** | 中間グレー（#808080 等）の上に重ねて表示 | 輪郭に **白ハロー・黒フチ** が出ていないか |

```python
def alpha_visualize(rgba: np.ndarray) -> Image.Image:
    """不透明=赤で α マスクを可視化（内部の抜けが一目で分かる）。"""
    alpha = rgba[:, :, 3]
    vis = np.zeros((*alpha.shape, 4), dtype=np.uint8)
    vis[alpha == 255] = [255, 0, 0, 255]   # 完全不透明=赤
    vis[(alpha > 0) & (alpha < 255)] = [255, 255, 0, 255]  # 半透明=黄（出てはいけない）
    return Image.fromarray(vis)
```

LINE のトーク画面はダークモード利用率が高いので、グレー背景だけでなく **ダーク背景（#1a1a1a）** でも縁を確認しておくと安心です。白ハロー（白フリンジ）は明るい背景では目立たず、ダーク背景で初めて浮かび上がります。

## 「後処理で繕う」より「生成段階で正す」原則

ここまで内部不透明化の後処理を詳しく書きましたが、最後に最も重要な原則を述べます。それは **「後処理で繕うより、生成段階で正すほうが確実」** ということです。

これは第1弾もっちー（V1）で骨身に染みた教訓です。当初は「AI が文字込みで一体生成した画像から、後処理で文字とキャラを分離しよう」としました。しかし結果は失敗の4連発でした。

| 試行 | やったこと | 結果 |
|------|-----------|------|
| 1 | `brighten_dark_text` で暗い文字を明るく | 白フチがギラついて副作用 |
| 2 | 色マスクで文字だけ抜く | キャラの腕・耳・頭まで一緒に消えて壊滅 |
| 3 | 半透明帯を smoothstep でシャープ化 | 原理的に完全シャープ化は不可能 |
| 4 | supersampling で2値化 | 文字エッジは改善するが分離問題は未解決 |

真の根本原因は **「AI 一体生成済みの画像では、文字色（茶・白）とキャラ色が完全に被るため、後処理での分離は数学的に不可能」** という点でした。一度混ざった情報は、後処理では復元できません。

そこで方針を「生成段階で正す」に転換しました。文字とキャラの分離が必要なら、**最初から文字なしでキャラを生成する** か、**AI に文字込みで綺麗に一体生成させてそもそも分離しない**、のどちらかにする。後処理に分離を期待しない、という設計です（最終的には gpt-image-2 による AI 一体生成に統一しました。詳細は次章）。

:::message
**後処理の品質改善は「1〜2回試行して打ち切る」**。V1〜V4 の経験から、後処理ループは費用対効果が急速に悪化します。1〜2回試して改善が頭打ちになったら、後処理をこねくり回すのをやめて、業界ベストプラクティスをリサーチし、**生成段階のアプローチそのもの** を見直すのが正解です。透過に関しても、`solidify_alpha` は「rembg の取りこぼしを規格に合わせる仕上げ」であって、「生成の失敗を救済する魔法」ではありません。
:::

ただし、本章の `solidify_alpha`（内部不透明化）は例外的に「後処理が必須」なケースです。なぜなら「内部を完全不透明にする」処理は **情報を失わない方向の操作**（穴を埋める・半透明を不透明に上げる）であり、分離のように情報を取り出す操作ではないからです。情報を失わない後処理は安全、情報を取り出そうとする後処理は危険、と覚えておくと判断を誤りません。

## この章のまとめ

- LINE の透過要件は **「イラスト内部は完全不透明（α=255）・背景は完全透明（α=0）」**。内部に穴（α=0）や半透明（0 < α < 255）があると「イラストの内部が透過されています」でリジェクトされる（V7 で実際に発生）。
- 背景除去は **`rembg -m birefnet-general`** を使う。細い輪郭を保持しやすいが、これだけでは内部透過は解決しない。
- **内部不透明化（`solidify_alpha`）が必須**。(1) 小さな不透明孤立島を除去 (2) `binary_fill_holes` で内部の穴を埋める (3) シルエット内を α=255 に2値化 (4) 黒フリンジを近傍色で距離変換補間――の4ステップで規格に合わせる。
- APNG の透過検証は **必ず frame 0（フル IDAT）で行う**。差分フレームを `seek` で読むと偽の内部穴が出る。機械チェック（holes / semi が 0）と目視（α マスク可視化・グレー／ダーク背景プレビュー）を両方通す。
- **「後処理で繕う」より「生成段階で正す」**。文字とキャラを後から分離するのは数学的に不可能。後処理の品質改善は1〜2回で打ち切り、業界ベストプラクティスをリサーチして生成アプローチを見直す。

次章では、本章で触れた「文字をどう扱うか」を掘り下げます。**文字・吹き出しの可読性** ――白縁が禁止な理由、濃茶 `#3E2314` を使う根拠、吹き出し fill に純白を使ってはいけない理由（rembg が背景と誤判定して透明化する）まで、透過と密接に絡む文字まわりのノウハウを扱います。
