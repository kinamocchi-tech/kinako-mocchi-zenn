スタンプの売上を最終的に左右するのは、キャラクターの可愛さよりも **「文字がパッと読めるか」** です。LINE スタンプはトークルームで、白背景・濃いダーク背景・写真壁紙など、あらゆる背景の上に重ねて表示されます。どんな背景でも文字と吹き出しが破綻しないこと。これが審査を通し、かつ実際に使われるスタンプの条件です。

本章は、私たちが 14 パック以上を制作する過程で踏み抜いた「文字まわりの罠」の記録です。白縁で痛い目を見て、純白の吹き出しが透明化して消え、「締」の字が潰れて読めなくなる。そのたびに原因を突き止め、恒久対策をパイプラインに焼き込んできました。同じ罠を踏まないために、失敗と対策をそのまま共有します。

:::message
この章で扱う 3 つの核心は **「白縁を使わない（濃茶 5px）」「吹き出しを純白で塗らない（クリーム #FFF8E0）」「吹き出し内は 5 文字以内」** です。いずれも rembg（背景除去）と AI 生成の構造的な性質に根ざした、回避不能な制約への対処です。
:::

## 大原則の変遷：Pillow 後乗せから「全 AI 一体生成」へ

最初に、文字をどう画像に乗せるかという根本方針の変遷を共有します。ここを理解しないと、後続の「縁取り」「吹き出し」の議論が宙に浮きます。

私たちは当初、文字の扱いについて 2 つの方式を行き来しました。

| 方式 | 内容 | 結末 |
|------|------|------|
| AI 一体生成 | キャラ・文字・吹き出しを gpt-image-2 が 1 枚に同時生成 | 当初は文字の縁がジャギーで読めず迷走 |
| ハイブリッド | キャラのみ AI 生成 + 文字は Pillow で後乗せ | text 系は綺麗だが balloon 系で破綻 |

第1弾もっちーの初期段階で、AI が生成した文字は縁がガタつき、ダーク背景で本体が黒く埋もれました。そこで「キャラだけ AI に描かせ、文字は Pillow（Python の画像ライブラリ）でプログラム的に後乗せする」ハイブリッド方式に逃げました。テキスト系（`text_top` / `text_bottom` など）のレイアウトでは、これで縁がシャープになり一度は解決します。

ところが、吹き出し（balloon）系を Pillow 後乗せで作ると、**キャラとしっぽ・吹き出しが「別々に貼り付けた」ように分離して見える** という新たな違和感が生まれました。そこで balloon だけ AI 一体生成に戻し、「text=Pillow / balloon=AI」というハイブリッドで第1弾は申請にこぎつけました。

しかし、これは過渡期の次善策でした。第4弾の検証で決定的な事実が判明します。

:::message alert
**AI キャラと Pillow テキストには「タッチギャップ」がある。**
AI が描いたキャラクターは筆致・陰影・線の揺らぎを持つ手描き風タッチですが、Pillow が機械的に描く文字はベクター的にのっぺりしています。同じ画像内に両者が同居すると、文字だけが「後から貼った」異物に見える。第4弾 D08 でこれが明確に観測され、ユーザー判定で却下されました。
:::

第4弾以降、gpt-image-2 のキャラ再現性が安定したこともあり、**全レイアウトを gpt-image-2 の AI 一体生成に統一** し、Pillow 後乗せは完全に廃止しました。`generate_char_only.py` / `text_overlay.py` / `apply_case_a.py`（旧 Pillow 系ツール）は実行ガードを入れて凍結済みです。

```text
[現行方式] 全 layout を gpt-image-2 が一体生成
  text_top / text_bottom / text_diagonal / text_underline
  balloon_round / balloon_explosion / balloon_jagged / balloon_thought
  → すべて AI が「キャラ + 文字 + 吹き出し」を 1 枚で描く
```

この方針転換が、本章の残りの議論の前提になります。「AI が一発で読める文字を描くにはどうプロンプトを書くか」「rembg にその文字を削られないようにするか」が、ここから先のテーマです。

:::message
**教訓**：後処理で品質を繕う試みは 1〜2 回で打ち切るべきでした。私たちは「文字の縁が汚い」という指摘に対し、white-spill 除去・smoothstep・brighten・色マスクと 4 回も後処理を重ね、3 日分の試行錯誤を費やした末に「AI が描いた文字とキャラを後から分離するのは数学的に不可能」という当たり前の結論に至りました。文字色はキャラ色と必ず被るため、マスクで文字だけを抜くとキャラの腕・耳・頭まで一緒に消えます。**問題は後処理ではなく生成段階で正す** ── これが最大の学びです。
:::

## テキスト縁取りの鉄則：白縁は絶対禁止、濃茶 #3E2314 を 5px

文字を AI に描かせる際、最初に決めるのは縁取り（stroke / outline）の色です。直感的には「文字を白で縁取れば、どんな背景でも浮き上がって読みやすい」と考えがちです。これは **絶対にやってはいけません**。

### なぜ白縁が構造的に破綻するのか

理由は背景除去ツール **rembg**（私たちは `birefnet-general` モデルを使用）の挙動にあります。rembg は AI 画像から白背景を検出して透明化しますが、このとき「背景の白」と「文字縁の白」を色で区別できません。両方とも `#FFFFFF` だからです。

結果、こうなります。

```text
文字の縁の運命（白縁を使った場合）

  [文字の上端・側面] ← 透明な領域と接している
                       → 白縁が残る ✅

  [文字の下端]       ← 下にある白背景と地続きで接している
                       → rembg が「これも背景の白」と判断して削る ❌
```

文字の上と横の縁は透明領域と接しているので残りますが、文字の下端は白背景と地続きになっているため、rembg が背景もろとも削り取ります。その結果、**縁が「上と側面には残り、下端だけ欠ける」不均一な状態** になります。これは見た目に非常に汚く、しかも rembg を使う限り構造的に回避できません。

:::message alert
白縁の下端欠けは、rembg のパラメータ調整や後処理では直りません。「白い背景」と「白い縁」を色情報だけで分離するのは原理的に不可能だからです。回避策は 1 つ ── **そもそも白い縁を使わないこと** です。
:::

### 解決策：キャラの輪郭と同じ濃茶 #3E2314 を 5px

私たちが第3弾以降の全パックで採用している縁取り色は、**濃茶 `#3E2314` を 5px** です。この色には複数の必然性があります。

| 利点 | 説明 |
|------|------|
| 全周均一に残る | `#3E2314` は暗色なので、rembg は背景と誤判定せず除去しない。下端も欠けない |
| どの背景でも視認 OK | 白背景・ダーク背景・淡色背景のいずれでもコントラストが確保される |
| キャラと一体感 | `#3E2314` はキャラクターのインク輪郭線とまったく同じ色。文字がキャラの世界観に溶け込む |

プロンプトでの指定はこうなります。`prompt_builder.py` の `LAYOUT_PROMPT_FRAGMENTS` で、テキスト系全レイアウトに次の断片を埋め込んでいます。

```text
thick SOLID DARK BROWN STROKE (#3E2314, 5px)
```

そして、絶対に書いてはいけない指示も明示しておきます。

```text
# 禁止（プロンプトに含めない）
white outline
white stroke
```

縁取り 1 つで、AI 画像生成（一体描画）・背景除去（rembg）・キャラ世界観（インク色一致）の 3 つが綺麗に整合します。「文字は白で縁取る」という常識を、ツールの都合に合わせて捨てる。これが実務の判断です。

## 吹き出し fill の鉄則：純白禁止、クリーム #FFF8E0

縁取りと同じ「rembg が白を消す」問題が、吹き出し（balloon）の塗りつぶし（fill）でも牙を剥きます。しかもこちらは文字よりも深刻で、**吹き出しまるごとが消失** します。

### 純白の吹き出しはダーク背景で消える

吹き出しの内側を純白 `#FFFFFF` で塗ると、rembg はそれを「白背景の一部」と判定し、α（不透明度）を 0 にして透明化します。白背景の上では透明でも目立ちませんが、**ダーク背景のトークルームでは吹き出しの中身がぽっかり消え、文字が宙に浮く** という致命的な破綻になります。

この問題は私たちのパックで何度も再発しました。

| 発生 | 内容 |
|------|------|
| 第1弾 V1 | 申請後に発覚。balloon_round 10 枚 + explosion 1 枚の計 13 枚でダーク背景の吹き出しが完全透明 |
| 第2弾きなこ | balloon 21 枚中 18 枚で吹き出し透過。全 21 枚の再生成という大きな手戻り |
| 第4弾 | ルール適用漏れで balloon fill が純白に逆戻り（再徹底） |

3 回も再発したことが、この罠の根深さを物語っています。

### 解決策：クリーム #FFF8E0 を明示する

恒久対策は **吹き出しの fill 色をクリーム `#FFF8E0` に指定** することです。純白からほんの少しだけ黄味を足した色で、rembg は背景の純白と区別して除去しません。一方、視覚的にはほぼ白に見えるので吹き出しとして自然です。

```text
# balloon_round のプロンプト指定
balloon fill color = #FFF8E0 (cream) — NEVER pure white #FFFFFF
```

:::message
「rembg を回避する（純白でない）」と「視覚的には白に見える」を両立させる、ぎりぎりの色が `#FFF8E0` です。第2弾の途中ではベージュ寄りの淡色を試しましたが、今度は「吹き出しが薄汚れて見える」という別の問題が出ました。クリームは試行錯誤で見つけた均衡点です。
:::

### 後処理での補強：connected-component 法による穴埋め

プロンプト指定だけでは、AI がうっかり純白寄りに描いたり、rembg が一部を削ったりするケースをゼロにできません。そこで後処理で機械的に補強します。

ナイーブに考えると「透明なピクセルを全部白で埋める」となりますが、これは破綻します。キャラの外側の背景まで白く塗ってしまうからです。私たちが使うのは **connected-component（連結成分）法** で、これは数学的に誤塗りゼロを保証します。

```python
# fix_balloon_fill.py / solidify_balloon_fill の考え方
# 1. 透明 (α=0) ピクセルをラベリングして「連結した塊」に分ける
# 2. 画像の端に接している透明成分 = 外側の背景 → 触らない
# 3. 画像の端に接しない孤立した透明成分 = 吹き出しの内側 → ここだけ白で塗る
```

ポイントは **「画像端に接するかどうか」** で内外を判定することです。

- 外側の背景の透明領域は、必ず画像の端まで地続きに広がっている
- 吹き出しの内側の透明領域は、吹き出しの線（濃茶 5px）に囲まれて孤立している

この性質を使えば、「吹き出しの内側だけ」を確実に特定して白く塗り直せます。キャラの腕や背景を誤って塗ることはありません。これが「誤塗りゼロの数学的保証」の意味です。

:::message alert
**`solidify_balloon_fill()` だけに頼ってはいけません。**
この関数は「α > 0（半透明）」のピクセルにしか作用しないため、rembg が既に α=0（完全透明）にしてしまった吹き出しは回復できません。完全透明化したものは、上記の connected-component 法（`fix_balloon_fill.py`）でしか救えません。第1弾はこの違いを見落として、申請後に透過バグが残りました。
:::

### 3 レイヤー防御

これらを統合し、吹き出しの透過を「生成・後処理・検証」の 3 段階で防いでいます。

| レイヤー | 対策 |
|---------|------|
| 生成プロンプト | balloon fill を `#FFF8E0`（クリーム）と明記。`pure white` / `#FFFFFF` は禁止 |
| 後処理パイプライン | `solidify_balloon_fill()` で半透明を不透明化 → `fix_balloon_fill.py` の connected-component 法で完全透明の内側を白塗り |
| 検証チェッカー | `check_balloon_integrity.py` で吹き出し fill 比率を機械検出。FAIL 0 件が必須 |

検証チェッカーには注意点があります。初期版は「画像の上部 30% だけ」を見て吹き出しを探していました。これだと吹き出しが画面中央や下部に配置されたケースを見逃します。現行版は **画面を 3 ゾーンに分けて最大値を取る方式 + 孤立透明成分チェック** に改訂し、配置位置に依存せず検出できるようにしました。

## 吹き出し内の文字数は 5 文字以内：「締」が教えてくれたこと

吹き出しの色を解決しても、最後の壁が残ります。**吹き出しの中の文字が多すぎると、AI が文字を潰す** のです。

### 第4弾 D08「締めのラーメン」事件

第4弾 D08 で `balloon_round` に「締めのラーメン」（7 文字）を入れたところ、AI が文字を細くレンダリングし、特に **「締」（15 画）の筆画が潰れて判読不能** になりました。

ここで私たちは典型的な誤診をします。記録のために、失敗した推論ごと残します。

| 試行 | 変更内容 | 結果 | 判定 |
|------|---------|------|------|
| 初回 | プロンプト指示なし | 筆画中央値 3px、細筆画率 64% | NG（細すぎる） |
| 試行1 | BOLD / 900weight / 6px+ を指示 | 筆画中央値 10px、細筆画率 1% | **NG（太すぎて筆画が合流）← 誤診** |
| 確定解 | テキストを「〆ラーメン」（4 文字）に短縮、bold 指示を削除 | 筆画中央値 7px、細筆画率 5% | **OK** |

### 誤診：「細いから太くすればいい」は間違い

私たちは「フォントウェイトの指定がないから細く生成された。bold を指定すれば直る」と考えました。しかし bold 指示を入れると、今度は太くなりすぎて隣り合う筆画同士が合流し、「締」がただの黒い塊になりました。問題は悪化したのです。

:::message alert
**フォントウェイト（bold）の指定は逆効果でした。** 細い文字を太くしようとすると、複雑な漢字では筆画が合流して逆に読めなくなります。AI の自然なウェイトに任せるのが正解です。
:::

### 真の根本原因：文字数過多による「1 文字あたりの描画サイズ不足」

正しい診断はこうです。問題はウェイトではなく **文字数** でした。

- 7 文字「締めのラーメン」→ 限られた吹き出しスペースに 7 文字を詰めるため 1 文字が小さい → どんなウェイトでも筆画が合流する
- 4 文字「〆ラーメン」→ 1 文字あたりのスペースが広い → 自然なウェイトで筆画が分離して読める

文字数を減らせば、1 文字あたりの描画面積が増え、AI は十分な解像度で各筆画を描けます。これがすべてです。

### 確定ルールと診断順序

ここから 2 つのルールを確立しました。

1. **吹き出し内の文字数は 5 文字以内（推奨）**。超える場合は `text_top` / `text_bottom` レイアウトに変更し、吹き出しではなく画面上下の広いスペースに文字を置く
2. **bold / フォントウェイト指示は原則追加しない**。AI の自然なウェイトに任せる

そして、文字が読めない問題に出くわしたときの **診断順序** を固定しました。

```text
テキスト可読性問題の診断順序

  1. 文字数は多すぎないか？（≤5 文字に減らせるか）  ← まずここ
  2. 縁取りは適切か？（白縁になっていないか）
  3. layout を変更できないか？（balloon → text_top/bottom）
  4. フォントウェイト調整                          ← 最後の手段
```

私たちは最初に「4. フォントウェイト」から手をつけて迷走しました。**真因はほぼ「1. 文字数」にある** と知っていれば、最短で解決できます。

## 配色ルール：WCAG 4.5:1 をダーク背景で担保する

縁取り色は基本的に濃茶 5px ですが、文字本体の色とのコントラストも設計が必要です。私たちは Web アクセシビリティ基準 **WCAG のコントラスト比 4.5:1** を、最も厳しいダーク背景で満たすよう 3 つの配色プリセットを用意しています。

| 配色 | 用途 | 文字色 / 縁取り |
|------|------|-----------------|
| `warm` | 落ち着き・基本 | `#A0522D`（テラコッタ）+ 白 5px |
| `pop` | 元気・カジュアル | 白 + 濃茶 5px |
| `impact` | インパクト・「！」を含む | 焦茶 + 黄 3px + 白 5px の二重縁 |

:::message
`warm` と `pop` で白縁が登場しますが、これらは縁取りの主役（濃茶）に対する補助的な内側縁や、文字本体色との組み合わせとして使うもので、テキスト全体の最外周を白でくくる用途ではありません。最外周の縁は前述のとおり濃茶 5px が基本です。`impact` の「白 5px 二重」のように外側を白にする場合は、その外にさらにキャラ輪郭が来る balloon 内など、rembg の下端欠けが起きない文脈に限定して使います。
:::

`impact` の二重縁取り（黄 3px + 白 5px）は、感嘆符を含む派手なスタンプで文字を最大限目立たせるためのものです。コントラストを稼ぎつつ、複数色の縁で背景からの分離を強化します。

## layout 設計の禁則：balloon_thought と暗色文字の同化

吹き出しレイアウトには、文字以前に「組み合わせてはいけない」禁則があります。第2弾 K33 で踏みました。

`balloon_thought`（思考雲・モコモコした雲形の吹き出し）に暗い色の文字を入れたところ、**灰色がかった雲の背景に暗色文字が同化して読めなく** なりました。雲のモコモコ模様と文字が混ざってしまうのです。

| 設計 | 判定 |
|------|------|
| `balloon_thought` + 暗色文字（grey / inner 系）| ❌ 雲背景で文字が同化 |
| `balloon_thought` 内の文字を AI に描かせる（淡色設定）| ❌ 雲と文字が混ざる |
| `balloon_round` + default color（白塗り + 焦茶文字）| ✅ 常に読みやすい |
| `balloon_thought` を使うなら color="default" または "impact"| ✅ コントラスト確保 |

結論はシンプルです。**迷ったら `balloon_round` + default color を使う**。白い丸吹き出しに焦茶文字という組み合わせは、どんな背景でも安定して読めます。`balloon_thought` を使いたい場合は、必ずコントラストの高い配色（default 以上 / impact）に限定します。

## ルール違反 ≠ NG：最終判定は「視覚的に読めるか」

最後に、これらのルールの運用思想を共有します。本章で「5 文字以内」「balloon_thought 禁則」などを定めましたが、**これらは機械的な合否判定の基準ではありません**。

第4弾の検証で、私たちは「7 文字の吹き出し」「6 文字の吹き出し」「正面合掌ポーズ」など、ルール上は違反する画像を「念のため確認」としてユーザーに提示しました。返ってきた回答は明快でした。

> 「全く問題がないので、今後ユーザー確認しないように改善してください」

文字数 6 以上を機械的に WARN 扱いしていた検証ツールが、**実際には視覚的に読める画像まで警告で止めていた** のです。

:::message
**最終判定の唯一の基準は「視覚的に読めるか」です。**
文字数・layout・pose のルール違反は、それ自体では合否判定の材料にしません。「ルール違反だが視覚的に OK」なら **OK 扱い**（WARN ではない）。本当に止めるべきは「視覚的に読めない・必着要素が消失・透過が破綻している」という BLOCK だけです。
:::

5 文字以内・balloon_thought 禁則といったルールは、**問題を未然に防ぐためのガイドライン** です。「文字数が多いと潰れやすい」という確率を下げるための設計指針であって、「6 文字だから不合格」という判定基準ではありません。生成後に文字数オーバーの画像が出ても、目視で読めるなら通します。これは「念のため確認」を排し、自律的に判断して制作を進めるための実務的な割り切りです。

## この章のまとめ

- **全レイアウトを gpt-image-2 の AI 一体生成に統一** しました。Pillow テキスト後乗せは AI キャラとのタッチギャップが違和感を生むため完全廃止。文字の品質問題は「後処理で繕う」のではなく「生成段階で正す」のが鉄則です。
- **テキストの白縁は絶対禁止**。rembg が白背景と一緒に文字下端の白縁を削り、縁が不均一に欠けます（構造的に回避不能）。キャラ輪郭と同色の **濃茶 `#3E2314` を 5px** にすれば、全周均一に残り、どの背景でも読め、世界観にも溶け込みます。
- **吹き出し fill を純白 `#FFFFFF` で塗らない**。rembg が背景と誤判定して透明化し、ダーク背景で吹き出しが消えます。**クリーム `#FFF8E0`** を指定し、後処理は「画像端に接しない孤立透明成分＝吹き出し内側」を白塗りする connected-component 法で補強（生成・後処理・検証の 3 レイヤー防御）。
- **吹き出し内の文字は 5 文字以内**。文字数過多で 1 文字あたりの描画サイズが不足すると、複雑漢字の筆画が潰れます。bold 指定は逆効果（太くしすぎて筆画が合流）。診断順序は「文字数 → 縁取り → layout 変更 → フォントウェイト（最後）」。
- **最終判定は「視覚的に読めるか」のみ**。5 文字ルールや layout 禁則は問題予防のガイドラインであり、機械的な合否基準ではありません。ルール違反でも視覚的に読めれば OK 扱いとし、自律的に制作を進めます。

ここまでで、キャラ・文字・吹き出しを 1 枚の画像として AI に綺麗に描かせる方法が揃いました。次章「**仕上げと検証 ── LINE 仕様への落とし込み**」では、生成した画像を rembg で透過し、Pillow で 370×320px へ内接フィットさせ、内部不透明化を施して LINE が要求する正確な仕様に落とし込む、最後の仕上げ工程を解説します。
