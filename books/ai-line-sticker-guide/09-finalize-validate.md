前章までで、背景透過と文字の可読性をクリアした「絵としては完成した」スタンプ素材が手元に揃いました。しかし、ここからが本番です。AI 画像生成で良い絵が出せる人は多いのに、申請まで辿り着けない人が多い最大の理由は、**「売れる形＝ LINE 仕様」に正確に落とし込めない** からです。

この章では、生成済みの素材を LINE Creators Market が受け付ける正確なフォーマットに変換し、**機械検証 → 全数目視 → 合格判定** までの仕上げ工程を解説します。本書の核である「実際に 14 パック以上を制作・申請し、審査通過の実績を積んだパイプライン」の、もっとも地味で、もっとも事故が多い部分です。

:::message
この章のゴールは「全 42 ファイル（スタンプ 40 + main + tab）が `validate_sticker.py` を Error 0 件で通過し、全数目視レビューに合格した状態」を作ることです。ここを曖昧に済ませると、審査で透過不完全・サイズ違反として弾かれ、1〜2 週間の審査待ちが無駄になります。
:::

## 仕上げ工程の全体像

仕上げは大きく 5 ステップです。前章までで背景透過（rembg）は終わっている前提で、ここでは **リサイズ・余白付与・最適化・販売用画像作成・検証** を扱います。

| ステップ | 入力 | 出力 | 主なツール |
|---------|------|------|-----------|
| 1. リサイズ・余白 | 透過済み PNG | 340×290 等の透過 PNG | Pillow（LANCZOS） |
| 2. PNG-24 RGBA 保存・最適化 | リサイズ済み PNG | 1MB 以下の PNG | Pillow（`optimize=True`） |
| 3. メイン画像作成 | 看板向きの 1 枚 | main.png（240×240） | Pillow |
| 4. タブ画像作成 | 識別しやすい 1 枚 | tab.png（96×74） | Pillow |
| 5. 検証 | 全 42 ファイル | 合否レポート | `validate_sticker.py` ＋目視 |

## ステップ 1：リサイズと余白 ― 340×290 への内接フィット

LINE の静止画スタンプは **最大 370×320px** ですが、いきなり最大サイズに引き伸ばすのは禁物です。透明な余白（マージン）を確保しないと、トーク画面で隣の吹き出しやアイコンと密着して窮屈に見え、審査でも「余白なし」を理由に印象が悪くなります。

本パイプラインの標準は以下です。

1. 元画像を **320×270 に内接フィット**（アスペクト比を保ったまま、はみ出さない最大サイズに縮小）
2. その周囲に **上下左右それぞれ 10px の透過マージン** を足す
3. 結果として **最終 340×290**（= 320+20 × 270+20）になる
4. これは LINE の最大 370×320 に余裕で収まる

```python
from PIL import Image

def fit_into_line_canvas(src: Image.Image,
                         inner=(320, 270),
                         margin=10) -> Image.Image:
    """元画像を inner に内接フィットし、margin の透過余白を足す。"""
    img = src.convert("RGBA")

    # 1) アスペクト比を保って inner に収まる最大サイズへ縮小
    iw, ih = inner
    scale = min(iw / img.width, ih / img.height)
    new_w = max(2, round(img.width * scale))
    new_h = max(2, round(img.height * scale))
    # 線をシャープに保つ高品質補間
    img = img.resize((new_w, new_h), Image.LANCZOS)

    # 2) 透過キャンバス（inner + 余白）を作り中央に貼る
    canvas_w = iw + margin * 2
    canvas_h = ih + margin * 2
    canvas = Image.new("RGBA", (canvas_w, canvas_h), (0, 0, 0, 0))
    off_x = (canvas_w - new_w) // 2
    off_y = (canvas_h - new_h) // 2
    canvas.paste(img, (off_x, off_y), img)
    return canvas  # 340 × 290 RGBA
```

:::message
**`Image.LANCZOS` を必ず使う理由**：縮小時の補間方法を `NEAREST` や `BILINEAR` にすると、キャラの輪郭線や文字のエッジがボケたりジャギーになったりします。`LANCZOS`（Lanczos リサンプリング）は高周波成分を保持するため、線がシャープに残ります。前章で苦労して残した濃茶 #3E2314 の輪郭線を、ここで台無しにしないための一手です。
:::

### なぜ最大サイズいっぱいにしないのか

「370×320 まで使えるなら大きいほど目立つのでは」と考えがちですが、実際のトーク画面でのスタンプ表示は縮小されます。重要なのは絶対サイズより **被写体の占有率と余白のバランス** で、余白なしで端まで詰めると窮屈で安っぽく見えます。本パイプラインが 320×270＋余白 10px を標準にしているのは、14 パックの試行で「最も自然に見えた」実測ベースの落としどころです。

:::message alert
**偶数 px 厳守**：LINE は幅・高さともに偶数 px を推奨し、奇数 px はリジェクト理由になり得ます。`canvas_w = inner_w + margin*2` の計算では、inner と margin を両方偶数にしておけば結果も偶数になります（320+20=340、270+20=290）。元画像を端数スケールで縮小したときに奇数になっていないか、検証ステップで機械チェックします。
:::

## ステップ 2：PNG-24 RGBA 保存とファイルサイズ最適化

LINE 仕様はファイル形式が **PNG-24・RGBA（アルファチャンネル付き）** で、**1 ファイル 1MB 以下** です。Pillow で RGBA のまま保存すれば PNG-24 RGBA になりますが、最適化を忘れると 1MB を超えることがあります。

```python
def save_line_png(img: Image.Image, path: str):
    """PNG-24 RGBA で 1MB 以下になるよう最適化保存する。"""
    img = img.convert("RGBA")  # PNG-24 RGBA を保証
    img.save(path, format="PNG", optimize=True)

    # 1MB 超なら色数を抑えて再保存（透過は維持）
    import os
    if os.path.getsize(path) > 1_000_000:
        # RGBA のまま可逆減色（透過アルファを保持する quantize）
        q = img.quantize(colors=256, method=Image.FASTOCTREE, dither=0)
        q = q.convert("RGBA")
        q.save(path, format="PNG", optimize=True)
```

`optimize=True` を付けるだけで、340×290 程度のスタンプは通常 100〜300KB に収まり、1MB を超えることはまずありません。超過するのは、グラデーションが多く色数が膨大なケースです。その場合のみ減色（quantize）を挟みますが、スタンプは「太い輪郭 + 数色のパステル」というフラットな配色が基本なので、減色しても劣化はほぼ目立ちません。

| 項目 | 設定 | 効果 |
|------|------|------|
| カラーモード | RGBA | アルファ透過を維持（PNG-24） |
| `optimize=True` | 必須 | zlib 圧縮レベル最大化でサイズ削減 |
| `quantize`（条件付き） | 1MB 超のみ | 色数を抑えてさらに圧縮 |

## ステップ 3・4：メイン画像とタブ画像の作り方

スタンプ本体だけでは申請できません。**main.png（240×240）** と **tab.png（96×74）** が看板とアイコンの役割を担い、購入率を左右します。

### メイン画像（240×240）― 看板としての設計

メイン画像はスタンプショップで最初に目に入る「店の看板」です。設計方針は次の通りです。

- **そのパックを最もよく表す表情・ポーズを 1 枚選ぶ**。もっちーパックなら「ノーマル＋笑顔」のもっちーが鉄板です。
- 必要に応じてロゴ風のテキスト（例：「もっちー」）を添え、何のスタンプかを一目で伝えます。
- 240×240 という小さい正方形なので、**被写体を大きめに配置** し、引きすぎないこと。

```python
def make_main_image(src_path: str, out_path: str):
    src = Image.open(src_path).convert("RGBA")
    # 240x240 の正方キャンバスに内接フィット（余白少なめ＝看板は大きく）
    canvas = Image.new("RGBA", (240, 240), (0, 0, 0, 0))
    scale = min(232 / src.width, 232 / src.height)  # 4px ずつ余白
    w, h = round(src.width * scale), round(src.height * scale)
    src = src.resize((w, h), Image.LANCZOS)
    canvas.paste(src, ((240 - w) // 2, (240 - h) // 2), src)
    canvas.save(out_path, format="PNG", optimize=True)
```

### タブ画像（96×74）― 識別性が命

タブ画像は横長（96×74）で非常に小さく、トーク画面のスタンプ選択タブに並びます。複数パックを持つユーザーが「どのタブがこのパックか」を一瞬で見分けるための **識別アイコン** です。

設計方針は **「看板/識別しやすい単一表情」** に尽きます。

- **最も識別しやすい単一表情**（笑顔・正面）を 1 つだけ使う。複数キャラや複雑なポーズは小さすぎて潰れます。
- 文字は入れない（96×74 では読めません）。
- 横長なので、縦長の被写体は上下が切れないよう内接フィットで収める。

| 画像 | サイズ | 設計の要点 |
|------|--------|-----------|
| main.png | 240×240 | 看板。最も魅力的な表情＋（任意で）ロゴ風テキスト |
| tab.png | 96×74 | 識別アイコン。単一表情・正面・笑顔。文字なし |

:::message alert
メイン画像とスタンプ本体の **世界観が一致していること**。看板だけ別タッチで作ると「メイン画像とスタンプ画像の世界観不一致」というリジェクト理由に該当します。main/tab はスタンプ 40 枚と同じキャラ・同じ画風で作ってください。
:::

## ステップ 5：検証スクリプトによる機械チェック

人間の目はサイズ違反や奇数 px を見逃します。だから **機械検証を必ず先に通します**。本パイプラインの `validate_sticker.py` は全ファイルに対し以下を判定します。

```bash
# 全 42 ファイル（01-40 + main + tab）をまとめて検証
python3 tools/validate_sticker.py assets/final/V1_mocchi/
```

### チェック項目

| # | チェック項目 | 合格条件 |
|---|------------|---------|
| 1 | **PNG-24 形式** | PNG であること |
| 2 | **RGBA 透過** | アルファチャンネルが存在すること |
| 3 | **サイズ上限** | 幅 ≤ 370px かつ 高さ ≤ 320px |
| 4 | **偶数 px** | 幅・高さがともに偶数 |
| 5 | **ファイルサイズ** | ≤ 1MB |
| 6 | **背景透過の実在** | **完全に透明な画素（α=0）が 1 個以上ある** |
| 7 | **縦横比** | 極端でない（極端な細長は警告） |

このうち見落としやすいのが **項目 6** です。「RGBA チャンネルがある」だけでは透過の証明になりません。アルファチャンネルは持っているのに全画素が不透明（α=255）＝実質背景が残っている、というケースが rembg のモデル選択ミスや内部不透明化のかけすぎで起こり得ます。そこで **「完全透明な画素が最低 1 個存在するか」** を確認し、背景が本当に抜けていることを保証します。

```python
def has_transparent_pixel(img: Image.Image) -> bool:
    """完全透明（α=0）の画素が 1 個でもあれば True。"""
    alpha = img.convert("RGBA").getchannel("A")
    return alpha.getextrema()[0] == 0  # 最小 α が 0 なら透過あり
```

### balloon 整合チェック ― FAIL 0 件が必須

前章で扱った吹き出し（balloon）スタンプには、固有の落とし穴があります。rembg が純白に近い吹き出しの内側を背景と誤判定し、**吹き出しの中身まで透明にしてしまう** ことがあるのです。これを見逃すと「吹き出しの中が抜けて文字だけ浮いている」不良品が申請に混ざります。

そこで balloon 系スタンプには専用の整合チェックを別途かけます。

```bash
# 吹き出しの fill が透明化していないか連結成分ベースで検査
python3 tools/check_balloon_integrity.py \
    --lineup prompts/mocchi_lineup_v1.json \
    --input-dir assets/final/V1_mocchi/
```

:::message alert
**balloon 整合チェックは FAIL 0 件が絶対条件** です。FAIL が 1 件でも残っている場合、その吹き出しは内部が抜けているので **申請に進んではいけません**。`fix_balloon_fill`（連結成分ベースで吹き出し内部を穴埋め）を再実行するか、該当スタンプを再生成してください。前章で扱ったクリーム色 #FFF8E0 を吹き出し fill に使う設計は、この透明化を防ぐための予防策ですが、最終確認としてこの機械チェックを必ず通します。
:::

## ステップ 6：全数目視レビュー ― 機械が見ない部分を見る

機械検証はサイズ・形式・透過の「物理的な不良」を捕まえますが、**キャラの整合性・表情の自然さ・絵の崩れ** は判定できません。ここは人間の目で **全 42 枚を 1 枚ずつ** 確認します。

| 観点 | 確認ポイント |
|------|------------|
| キャラ整合性 | 40 枚を通して同じキャラに見えるか（体型・色・線の太さがブレていないか） |
| 表情の自然さ | 不自然・不気味な表情になっていないか |
| 文字 | 誤字脱字はないか（申請後の修正は審査やり直し） |
| 透過の見た目 | ダーク背景に重ねて白い縁取り・残留が見えないか |
| 余白 | 端が切れていないか・偏っていないか |

本パイプラインでは、この目視を機械的に流さないために 3 つの視点を併用します。

- **セルフレビュー**：全 40 枚を縦並びにして、キャラの一貫性を視覚的に通し見る。
- **@kinako（キャラ監修・Lv2）**：きなこ単独パックなら、技術・キャラ設定の正確性をオブザーバーとして確認。
- **@mochi（素人目線・Lv2）**：「素人目に分かるか」「不自然な表情はないか」を初心者代理としてチェック。

:::message
**鉄則：不合格が残る限り、申請に進まない。** 機械検証 Error 0 件・balloon 整合 FAIL 0 件・目視レビュー Error 0 件、この 3 つがすべて揃って初めて次の申請フェーズに進みます。「9 割いいから出してしまおう」は、1〜2 週間の審査待ちの末にリジェクトを食らう最短ルートです。LINE の審査 NG 多発項目には「透過不完全」「サイズ違反」「メイン画像との世界観不一致」が並んでおり、これらはこの章の検証で 100% 事前に潰せます。
:::

## トラブルシューティング

仕上げ・検証の段階で出る代表的な不合格と、その原因・対処をまとめます。多くは「前の工程の設定ミス」が原因で、検証段階で初めて顕在化します。

| 症状 | 原因候補 | 対処 |
|------|---------|------|
| **キャラがブレる** | 量産時に reference（リファレンス画像）を未指定 | 必ず reference を渡して再生成。ドリフトなら最良 1 枚を reference に差し替えて再キャリブレーション |
| **背景が透過されない** | rembg のモデル選択ミス | `birefnet-general` モデルを使う。検証項目 6（完全透明画素の有無）で機械検出 |
| **ファイルサイズ超過** | PNG 最適化不足 | Pillow の `optimize=True` を指定。なお超過時は RGBA quantize で減色 |
| **線がボケる** | リサイズ時の補間方法が低品質 | `Image.LANCZOS` を使う（`NEAREST`/`BILINEAR` は不可） |
| **奇数 px でリジェクト** | 端数スケールで奇数になった | inner・margin を偶数にし、検証項目 4 で機械チェック |
| **吹き出しの中が抜ける** | rembg が純白 fill を背景誤判定 | fill を #FFF8E0 に。`fix_balloon_fill` で穴埋め・整合チェック FAIL 0 件 |
| **LINE 申請でリジェクト（類似）** | 既存キャラに類似 | キャラシートから造形を調整（前章までのキャラ設計に戻る） |

:::message
これらの症状の大半は「検証で初めて気づく」のではなく「検証で確実に捕まえる」ことに価値があります。機械検証を全数にかけ続けることで、量産途中で混入した不良が申請前に必ず止まります。これが 14 パックを通して透過不完全・サイズ違反のリジェクトをゼロに抑えられた理由です。
:::

## この章のまとめ

- 元画像は **320×270 に内接フィット → 上下左右 10px の透過マージン → 最終 340×290**（最大 370×320 に収める）。縮小は **`Image.LANCZOS`** で線をシャープに保つ。
- 保存は **PNG-24 RGBA**、**`optimize=True`** で 1MB 以下に。超過時のみ RGBA quantize で減色する。
- 申請に必要なのは **スタンプ 40 ＋ main.png（240×240）＋ tab.png（96×74）＝ 42 ファイル**。main は看板、tab は識別しやすい単一表情で作る。
- **`validate_sticker.py`** で全数を機械検証（PNG-24・RGBA・サイズ≤370×320・偶数 px・≤1MB・完全透明画素の存在・縦横比）し、balloon 整合チェックは **FAIL 0 件必須**。
- **全 42 枚を 1 枚ずつ目視**し、キャラ整合性・誤字・透過の見た目を確認する。**不合格が残る限り申請に進まない。**

ここまでで、静止画スタンプを LINE 仕様に完全に落とし込み、検証まで通せるようになりました。次章では一歩進んで、**動く＝アニメーションスタンプ（APNG）の絶対要件** に踏み込みます。静止画の知識がそのまま使えると思って APNG に挑むと、「総再生時間がちょうど整数秒でないと弾かれる」「Pillow 製 APNG は受け付けられない」といった、静止画には存在しない壁に正面衝突します。次章でその罠を一つずつ外していきましょう。
