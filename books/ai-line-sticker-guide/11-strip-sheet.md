title: "帯状シート量産でシームレス化 ― アニメの根本解決"
---

前章では、APNG アニメーションスタンプの「整数秒・片辺 270px・内部不透明化」という絶対要件を確認しました。仕様を満たすだけならテクニックの積み上げで何とかなります。問題は、その先にある「**アニメとして成立しているか**」という審査の壁です。

本章では、個別 6 コマ生成で何度も却下されたアニメを、**strip-sheet（帯状シート一括生成）方式**で根本解決した一次ノウハウを、失敗ログ込みで共有します。これは机上の方式論ではなく、実際に 8 個パックを全数クリーンで審査通過させた「動くパイプライン」の核です。

## なぜ個別コマ生成は構造的に却下されるのか

最初に採用したのは、素直な発想でした。「1 コマずつ AI で生成して、6 枚を並べれば APNG になる」というものです。これが審査で次のように却下されました。

| 却下理由 | 個別生成での発生メカニズム |
| --- | --- |
| 非シームレス（ループの継ぎ目が飛ぶ） | コマごとに独立生成するため、最終コマ → 1 コマ目の連続性が担保されない |
| コマ間でキャラの大きさが変わる | 1 コマずつ生成すると、キャラのスケール・位置が毎回わずかにドリフトする |
| 透過漏れ多数 | コマごとに rembg をかけると、吹き出しとキャラの透過処理が安定せず半透明・穴が残る |

これは運やプロンプトの問題ではなく、**個別生成という構造そのものの限界**です。N 個の独立した確率的生成を後から「同一キャラの連続動作」に見せかけようとする以上、サイズ・色・形・位置のドリフトは累積し続けます。コマを増やすほどズレも増える。float 方式（背景上をキャラが浮遊する案）も試しましたが、こちらは「いまいち」で、根本解決にはなりませんでした。

:::message
個別生成の限界は第 5 章「40 個量産とキャラドリフト対策」で扱ったドリフト問題と同根です。静止画は「最良 1 枚を reference に差し替え再キャリブレーション」で吸収できましたが、アニメは **1 スタンプ内で複数コマの一貫性**を同時に満たす必要があり、難易度が一段上がります。
:::

## 確定方式: strip-sheet（縦 1 列 N 行を 1 枚一括生成 → 分割）

たどり着いた結論は、発想の逆転でした。**N 個のコマを別々に生成するのをやめ、縦 1 列 N 行のストリップ（帯状シート）を 1 枚の画像として一括生成し、後から機械分割する**という方式です。実装は `tools/gen_strip_anim.py` が担います。

最大の効能は次の 2 点です。

- **コスト**: 1 スタンプ = 1 API 呼び出し。個別 6 コマ生成の **約 1/6** のコストで済みます。
- **一貫性**: 1 枚の画像の中に全コマが描かれるため、キャラの大きさ・色・形が **生成時点で一定**になります。サイズドリフトと非シームレスを、後処理ではなく生成段階で解消するのです。

### なぜ「帯状（strip）」でなければならないか

ここが過去の失敗を覆す最重要ポイントです。実は「複数コマを 1 枚に詰める」というアイデア自体は、以前モデルシート（複数ポーズを 1 枚に並べた参照シート）として試して**却下されています**。その真因を突き止めたことが、strip-sheet 確定の鍵でした。

```text
過去却下の真因: 2×3 の密グリッド = 1 コマ 512×512 → 日本語テキストが潰れる

縦 1 列 × 3 行 (1024×1536) なら 1 コマ 1024×512
→ 横解像度が 2 倍 → 日本語テキストが生存する
```

つまり、却下されていたのは「1 枚にまとめたこと」ではなく「**密グリッドで 1 コマあたりの解像度が足りなかったこと**」でした。2 列×3 行のように横にも詰めると 1 コマの横幅が 512px しか取れず、吹き出しの日本語が読めなくなります。縦 1 列に並べれば、1 コマの横幅をキャンバス全幅（1024px）使えるため、横解像度が倍になり、テキストが潰れません。

| レイアウト | キャンバス | 1 コマ解像度 | 日本語テキスト |
| --- | --- | --- | --- |
| 2×3 密グリッド | 1024×1536 | 512×512 | 潰れる（却下） |
| 縦 1 列×3 行（strip） | 1024×1536 | 1024×512 | 生存（採用） |

縦に並べる、ただそれだけの違いで、横解像度が 2 倍になり、品質と一貫性とコストを同時に満たせるようになりました。

## 必須テクニック 5 点

strip-sheet を「動く」ものにするには、5 つのテクニックが揃って初めて機能します。1 つでも欠けると透過漏れか文字ブレで却下されます。

### ① balloon 分離生成 ― キャラの上に「白い隙間」を空けさせる

吹き出し（balloon）をキャラに接触させてはいけません。プロンプトで「balloon をキャラの上に**明確な白い隙間を空けて**描く（吹き出しの尻尾もキャラに触れさせない）」と指示します。

狙いは後処理にあります。balloon とキャラが画像上で離れていれば、rembg で背景を抜いた後、両者が **別の連結成分（blob）**として分離されます。すると成分間が透明に保たれ、透過漏れが起きません。逆に接触させると 1 つの連結成分に融合し、後段の処理が破綻します。

:::message alert
このとき特に危険なのが「手・拳を頭上に挙げるポーズ」です。挙げた手が balloon と接触して 1 成分化し、後述の balloon 凍結が機能しなくなります。**手・拳は胸の高さ（CHEST height, kept LOW）**に保つようポーズ指示してください。実際の量産でも、頭上ポーズのコマで「balloon 凍結: スキップ」ログが出て再生成になりました。
:::

### ② solidify_alpha は連結成分ごとに穴埋め

内部不透明化（前章の絶対要件）は、画像全体に一括で fill / closing をかけてはいけません。`build_apng_line.py` では **連結成分ごとに穴埋め**します。

理由はシンプルです。全体一括で塗り潰すと、近接した balloon とキャラの**隙間を橋渡しして塗ってしまう**のです。これが透過漏れリジェクトの直接原因になります。成分単位で処理すれば、balloon の内部・キャラの内部だけが不透明化され、成分間の透明は維持されます。

| solidify の単位 | 結果 |
| --- | --- |
| 全体一括 fill / closing | balloon とキャラの隙間を橋渡し → 透過漏れ（却下） |
| 連結成分ごと | 各成分の内部だけ不透明・成分間は透明（合格） |

### ③ balloon 凍結 ― 文字を完全静止させる

アニメで文字が 1px でも揺れると「非シームレス」に見えます。これを潰すのが balloon 凍結です。

各コマから balloon（画像最上部の成分）を分離し、**面積最大の最良 balloon を 1 枚選んで全コマに固定合成**します。これにより、文字はループの継ぎ目で一切動きません。重要なのは、これが Pillow で後から文字を描き足すのではなく、**AI が生成した balloon を凍結して再利用している**点です。AI 生成の質感を保ったまま、動きだけを止められます。

キャラ側は balloon 以外の最大成分のみを残します。こうすることで、セル境界にわずかにはみ出した小さな残骸を自動除去できます。

### ④ 分割しやすいレイアウト指示

機械分割を前提に、プロンプトで次を明示します。

```text
- これは機械分割される画像である
- N 個のコマを均等 N 等分する
- セル境界は白にする
- 各要素はセル内で完結し、セルをまたがない
```

この指示があるかないかで、隣セルからの色片混入（例: 青いバンダナの切れ端が隣コマに飛び込む）が**激減**します。要素がセル境界をまたぐと、分割線でちょうど切断され、隣のコマに異物が残ってしまうのです。

### ⑤ 体固定の微モーション ― ping-pong で完全シームレス

動かす範囲を最小化します。**体・足・位置・スケールは全コマで固定**し、変化させるのは小さなアクセント（手・目・頭）だけにします。大きなポーズ変化は位置ドリフトを招き、コマ間でキャラが「跳ねて」見えます。

さらに ping-pong（`0..N-1..0`）でコマを往復させると、最終コマから 1 コマ目への継ぎ目が完全に滑らかになります。3 ポーズなら `0,1,2,1` の 5 コマで 1 ループになり、どこから見ても連続します。

```text
3 ポーズ → ping-pong 5 コマ:  pose0 → pose1 → pose2 → pose1 →（pose0 へループ）
体は固定、手/目/頭だけが微動 → 跳ねず、継ぎ目も飛ばない
```

## 仕様（LINE アニメ・最小構成）

strip-sheet 方式は、コストを最小化して「まず 1 パック出す」のに最適です。LINE アニメスタンプは静止画（8/16/24/32/40 個）と異なり **8/16/24 個**から申請できるため、最小 8 個・120 円スタートで出せます。

| 項目 | 値 |
| --- | --- |
| 個数 | 最小 8 個から申請可（8/16/24） |
| 価格 | 120 円スタート可 |
| コマ数 | APNG 5〜20 コマ |
| 再生時間 | 1〜4 秒（**ちょうど整数秒**。端数は Error） |
| ファイルサイズ | 各 1MB 以下（実運用は 300KB 目安で圧縮） |
| 画像サイズ | 各 ≤320×270 かつ片辺 ≥270px |
| 1 ループ | ちょうど 1.0 秒 |

`build_apng_line.build()` が、ping-pong 化・1 ループ = ちょうど 1.0 秒の整数秒調整・連結成分ごとの solidify・300KB 圧縮・apngasm での APNG 生成・verify までを一括処理します。Pillow 製 APNG は LINE に弾かれるため、生成は apngasm 系で行う点は前章のとおりです。

## 重大な落とし穴 2 つ

ここからは、量産で実際に踏んだ「物語」です。再現性が高く、知らないと必ずハマります。

### 落とし穴 1: 参照画像の透明部 RGB が黒だと「黒パネル」が量産される

V8（きなこ）の量産で、ほぼ全コマに黒い塗りつぶしパネル・額縁が混入する事故が起きました。原因の特定に時間がかかった、最も厄介なバグです。

根本原因はこうです。**gpt-image-2 の `images.edit` は、参照画像の RGB チャンネルを見ます。** 透過 PNG であっても、透明部の RGB が黒（0,0,0）だと、その黒を「背景／コマ枠」として生成画像にコピーしてしまうのです。きなこの sister 参照 3 枚が**黒背景透過**だったため、黒が量産されました。もっちーは参照が白背景だったため軽症で済んでおり、この差が原因究明を遅らせました。

:::message alert
**対策（必須）**: `gen_strip_anim._flatten_white()` で全参照を白地に合成してから渡します。これで solid 黒パネルは根絶できます。新規キャラに着手するときは、**参照画像の透明部 RGB を必ず確認**してください。「透過 PNG だから安全」ではなく、透明部の RGB 値そのものが問題になります。
:::

```python
# 参照画像を白地に平坦化してから edit に渡す（黒パネル根絶）
def _flatten_white(img):
    # 透過 PNG を白背景に合成 → 透明部 RGB の黒を白に置換
    bg = Image.new("RGB", img.size, (255, 255, 255))
    bg.paste(img, mask=img.split()[3])  # アルファをマスクに合成
    return bg
```

### 落とし穴 2: コマ枠（細線・額縁）の残骸

gpt-image-2 は、縦ストリップを「コマ割り漫画」と解釈し、勝手に枠線・額縁を描く癖があります。対策は二段構えです。

**プロンプト側**: 「cell / band / slice / gutter」といったコマ割りを連想させる語を**全廃**します。代わりに「**NOT a comic（漫画ではない）／枠なし／白地に分離した N 枚の絵**」と明示します（`build_strip_prompt`）。コマ語を 1 つでも残すと、AI が枠を描く確率が上がります。

**後処理側**: `remove_panel_borders()` が、純黒の塗りパネル（大面積・高充填率）と長い直線ランを除去します。ただし注意点があります。

:::message
`remove_panel_borders()` は **balloon に接続した細線を取れません**。balloon 成分を保護する処理が優先されるため、balloon に繋がった枠線は残ります。この個体は再生成で対応します（1 個 = 1 生成なので安価）。白背景平坦化 + プロンプト改善で大半はクリーンになり、残る細線個体だけを再ロールする運用です（もっちーでも同じ運用でした）。
:::

## main / tab の正攻法 ― 吹き出しを生成しない専用ストリップから作る

ストアに表示される main（240×240 アニメ）と tab（96×74 静止）は、**全数検証の対象外にしていたために事故が起きました**。これも実体験です。

V8 きなこでは、**tab 画像が吹き出しの文字「お」**になっていました。本体ではなく文字グリフが選ばれてしまったのです。さらに両パックの main アニメの動きコマに、黒いコマ枠（中空の額縁）が混入していました。LINE のサムネは 1 コマ目しか表示しないため、1 コマ目さえ綺麗なら審査前の目視で見逃してしまう厄介なバグでした。

根本原因は 3 つでした。

1. main / tab 生成が**ツール外の場当たりインラインコード**で、再利用もテストもされていなかった。
2. tab を「吹き出し入りスタンプの raw セルの balloon 以外の最大成分」で抽出していた。きなこは balloon の水色塗りが rembg で消え、残った文字グリフ「お」が本体より大きく**誤選択**された。さらにコマによっては balloon と本体が 1 連結成分に融合し、本体だけの抽出が**原理的に不能**だった。
3. main / tab を**全数検証の対象外**にしていた（01〜NN しか目視していなかった）。

恒久対策として、`tools/gen_strip_anim.py --main-tab SRC_ID` に正式モードを実装しました。要点は「**吹き出しを生成しない専用ストリップの本体から作る**」ことです。

| 対策 | 内容 |
| --- | --- |
| 正式モード化 | main / tab を `build_main_tab` としてツールの正式モードに格上げ |
| 専用素材 | 素材は **balloon=False の専用ストリップ**の本体。文字混入が原理的に起きない |
| main の作り方 | 1 枚のクリーン本体を緩やかに上下フロート。単一静止画フロートならマルチコマ由来のアーティファクトが出ない |
| tab の作り方 | main と**同一の本体静止画**から作る。必ず一致し、文字も枠も入らない |
| 検証 | main / tab を**必ず全数検証に含める**（グレー背景目視 + 最長黒ラン点検） |

:::message
「アニメ本体は float 禁止」と前章で述べましたが、それはスタンプ本体（01〜NN）の話です。ストアアイコンの main は、単一静止画を上下にフロートさせるのは可です。むしろ動きコマを持たないことで、稀に出るコマ枠アーティファクトを原理的に回避できます。
:::

## 量産チェックリスト（全数必須）

実際の 8 個量産では、**8 個中 5 個が一発 OK・3 個が再生成（計 12 生成）で全数クリーン**を達成しました。以下を全コマ・全スタンプで機械検証 + 目視するのが、全数クリーンへの最短ルートです。

```text
1. 生成ログで「balloon 凍結: OK」を確認
   → 「スキップ」が出たら、手を頭上に挙げていないかポーズを修正して再生成
2. αマスク可視化（不透明 = 赤）で
   balloon とキャラが【別 blob・間が透明】であることを全コマ確認
3. グレー背景プレビューで、残骸・ハロー・黒フチがなく
   テキストが鮮明であることを全コマ目視
4. build_apng_line.verify() で寸法・コマ数・300KB を機械検証
5. NG はピンポイント再生成（1 個 = 1 生成なので安い）
   ※ main / tab も必ずこの検証に含める
```

| 量産で出た症状 | 原因 | 対策 |
| --- | --- | --- |
| 「balloon 凍結: スキップ」ログ | 拳・手がキャラ頭上で balloon と接触 → 1 成分化 | 手・拳は胸の高さに固定してポーズ指示 |
| コマ端の黒い矩形はみ出し | セル境界・セパレータの誤描画が分割後に残存 | 同一プロンプトで再生成（確率的事象・1 回で解消が多い） |
| 隣セルからの色片混入 | 要素がセル境界をまたいだ | 「分割しやすいレイアウト」指示で激減 |

「balloon 凍結: OK」を全数で確認できるかどうかが、品質ゲートの実質的な合否ラインです。スキップが出る個体は、ポーズ（特に手の高さ）を直して再生成すれば、安く確実にクリーンに持ち込めます。

## この章のまとめ

- 個別 6 コマ生成のアニメは、**サイズドリフト・非シームレス・透過漏れ**という構造的限界で却下される。後処理では治らない。
- **strip-sheet（縦 1 列 N 行を 1 枚一括生成 → 分割）**が確定方式。1 スタンプ = 1 API 呼び出しで個別の約 1/6 コスト、かつ全コマのキャラ一貫性を生成時点で確保する。縦 1 列にすることで 1 コマ 1024×512 の横解像度が得られ、日本語テキストが潰れない。
- 必須テクニックは 5 点（**balloon 分離生成・連結成分ごとの solidify・balloon 凍結・分割しやすいレイアウト指示・体固定の ping-pong 微モーション**）。1 つでも欠けると透過漏れか文字ブレで却下される。
- 最大の落とし穴は **参照画像の透明部 RGB が黒だと黒パネルが量産される**こと。`_flatten_white()` で全参照を白地に平坦化してから渡す。コマ枠残骸はコマ語全廃 + 後処理 + 再生成で潰す。
- main / tab は**吹き出しを生成しない専用ストリップの本体**から作り、必ず全数検証に含める。場当たりインラインコードと検証漏れが文字混入・黒枠事故を生んだ。

これで「LINE 仕様を満たし、審査を通る形」のスタンプ素材は、静止画もアニメも揃いました。次章では、いよいよ完成した素材を **LINE Creators Market に申請する完全手順**を、メタ情報の文字数カウントの罠や [AI] ラベルの扱いまで含めて解説します。
