# 品質ゲートと自動改善 ── Claudeが自分でミスを直す仕組み

## 品質ゲートとは

**品質ゲート** は「ある基準を満たさなければ次のフェーズに進めない」仕組みです。

自動化システムでは、人間がチェックしないぶん、
自動で品質を担保する仕組みが重要になります。

## 4段階のハーネスシステム

本書では、ルールの強制力を4段階で管理しています。

| レベル | 種類 | 強制力 | 例 |
|-------|------|--------|-----|
| Lv.1 | CLAUDE.md（文書） | 低 | 「8分未満の台本は出さない」と記述 |
| Lv.2 | セルフレビュースキル | 中 | `self-reviewer` が尺をチェック |
| Lv.3 | Hooks（自動スクリプト） | **高** | 台本JSONを保存すると自動バリデーション実行 |
| Lv.4 | CI/GitHub Actions | **最高** | バリデーション失敗時はPRをブロック |

:::message
「ハーネスエンジニアリング」とは、AIの行動を「お願い」から「仕組み」で制御する設計手法です。
本書では、同じミスが2回以上繰り返されたとき、上位レベルへの昇格を検討します。
:::

## 台本の品質チェック項目

`post-tool-use-validate.sh`（Lv.3 Hook）が台本JSON保存後に自動チェックします。

| チェック項目 | 合格基準 |
|------------|---------|
| JSON構文 | エラーなし |
| 必須フィールド | `video_id`, `sections`, `fact_check_flags` が存在する |
| 方言ルール | もっちーに京都弁なし・きなこに大阪弁なし |
| 数値のファクトフラグ | 数字・年号を含むセリフに `fact_check_flags` が設定されている |

バリデーション失敗時は、台本ファイルの保存がブロックされます。
Claudeは自動で修正してから再保存を試みます。

## セルフレビューの仕組み

`self-reviewer` スキルは、PR作成前に複数の観点から品質をチェックします。

```
自動チェック項目（抜粋）:
□ 尺は8〜20分か（VOICEVOX速度換算）
□ fact_check_flagsが5件以上の場合、説明文に記載したか
□ キャラクターの口調が設定と一致しているか
□ 台本のJSON構文に問題はないか
□ 参考文献は最低3件以上あるか
```

## lessons-core.md / lessons/*.md ── 繰り返しミス防止パターン集

過去に発生したミスのパターンと対策は、2つの層に分けて記録しています。`docs/rules/lessons-core.md` は全セッションで常に読み込まれる Hot 層で、肥大化を防ぐため350行の上限を設けています。個別スキル固有のミスは `docs/rules/lessons/{カテゴリ}.md` という Warm 層に記録し、そのタスクを実行するときだけ読み込みます。Hot 層のエントリは、コードやフックへの実装が完了して再発しなくなった時点で物理削除し、常駐分を軽量に保ちます。

例（lessons-core.md の1エントリ）:

```markdown
### L-028: 発音辞書のスペース区切りカタカナがVOICEVOXで不自然なポーズになる

パターン: 「Claude Code」の読みを「クロード コード」（スペースあり）で辞書登録すると、
VOICEVOXがスペースを単語境界として認識し、長いポーズを挿入する。

対策: tools/generate_audio.py に「カタカナ語間スペース自動除去」を追加。
```

Claudeはセッション開始時に lessons-core.md を読み込み、同じミスを繰り返さないようにします。Warm 層の lessons/*.md は、該当するスキルのSKILL.mdが読み込み先として指定しているため、そのタスクに着手したタイミングで読み込まれます。新しいミスが発生するたびに、Claudeが自動でこれらのファイルに追記します。

## retrospectiveスキル ── 振り返りの自動化

品質ゲート未達や繰り返しエラーが発生した際、
`retrospective` スキルが自動で KPT 分析を実行します。

```
振り返りの流れ:
1. 問題を検出（品質ゲート未達など）
2. retrospective スキルが起動
3. Keep・Problem・Try を自動分析
4. Try 項目を GitHub Issue として登録
5. retro-try-handler が Try を次のセッションで実行
6. lessons.md に学びを記録
```

これにより、人間が振り返り作業をしなくてもシステムが自律的に改善し続けます。
