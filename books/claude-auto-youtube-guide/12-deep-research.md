# Phase 2 ディープリサーチ ── 最後の手作業が自動になるまで

動画1本につき、人間がどうしても手を動かす工程が一つだけ残る。長いあいだ、そう考えていました。台本も音声も画像も動画も自動で流れていくのに、リサーチだけは Claude のチャット画面を開いて、自分でボタンを押すしかない。むしろその一手間こそが品質の最後の砦なのだと、半ば言い訳のように納得してもいました。ところが 2026年7月時点のパイプラインでは、この最後の手作業も Claude 自身が回しています。

とはいえ、いきなり全自動から入るのは勧めません。最初の1本くらいは、自分の手でリサーチを回してみるほうがいい。何が肩代わりされたのかを一度自分の目で見ておくと、あとで自動化の中身がすっと入ってきます。ここではまず手動の手順をたどり、そのうえで、その工程を丸ごと引き受ける仕組みに移ります。

## まず手で回してみる ── 手動ディープリサーチ

**Claudeのリサーチ機能（Research）** は、Claudeがウェブを検索・横断して詳細な調査を行う機能です。通常のウェブ検索と違い、複数のサイトを横断して情報を統合し、信頼性の評価や矛盾の検出まで行ってくれます。動画の土台になる事実を集めるのに向いています。

使い方はシンプルです。

1. [claude.ai](https://claude.ai) をブラウザで開く（Pro/Max プランにログイン）
2. 新しい会話を開始する
3. テキスト入力欄の **「リサーチ」ボタン（"Research" アイコン）** をクリック
4. 調べたいテーマのプロンプトを入力して送信

このリサーチ機能は、ウェブチャット・Claude Desktop・Claude Mobile からのみ利用できます。API からは呼べません。この点は 2026年7月時点でも変わっていません（出典: [Anthropicサポート公式ドキュメント](https://support.claude.com/en/articles/11088861-using-research-on-claude)）。手動で回すなら、素直にブラウザで開くのが正解です。

### リサーチプロンプトの書き方

GitHub Issue の Phase 2 には、**ディープリサーチ用のプロンプトが自動生成されています**。中身はテーマごとに変わりますが、骨格は共通です。

```markdown
## Deep Research プロンプト

以下のプロンプトをClaudeのリサーチ機能に貼り付けてください:

---
「◯◯（調査テーマ）について詳しく調査してください。
以下の観点で情報を収集・整理してください:

1. 機能の詳細（何ができるようになったか）
2. 従来との違い（比較）
3. 実際の使用例・ユースケース
4. 開発者コミュニティの反応
5. 利用上の注意点・制限事項
6. 公式発表日・バージョン番号・価格変更の有無

信頼できる一次ソース（公式ブログ・論文・公式ドキュメント）を優先してください。
推測で情報を補完せず、不明な点は『要確認』と明記してください。」
---
```

観点を最初から列挙しておくのは、あとで台本の骨組みにそのまま流用できるからです。特に6番目の「公式発表日・バージョン番号・価格」は、ファクトチェックで最後まで残りやすい項目なので、この段階で釘を刺しておきます。

### リサーチ結果の保存

リサーチが終わったら、回答を `content/research/V{動画ID}_research.md` として保存します。Claude Code に頼めば、コピー＆ペーストの手間も省けます。

```
「このリサーチ結果を content/research/V005_research.md として保存してください」
```

ここまでが、最初の1本を自分の手で回すときの流れです。所要時間はおよそ10〜20分。慣れてくると、作業というより、出来上がってくる調査を眺めているだけの時間になります。

## その手作業が、自動になった

ここで当然の疑問が出ます。リサーチ機能がチャット画面でしか使えないのなら、いったいどうやって自動化したのか。

答えは、抜け道を探したのではなく、正面から入れるようになった、というものです。以前のこの章には、リサーチ機能は API から呼べず、ブラウザを自動操作して無理に動かすことは規約で禁じられている、と書いていました。これは当時の事実で、いまも API 直叩きやブラウザ RPA が禁止という点は変わりません。変わったのは、 **Claude Code 自身がディープリサーチを正規の機能として持つようになった** ことです。規約の外側で無理をする必要は、もうありません。

この工程を丸ごと引き受けるのが **research-runner** というスキルです。Phase 2 の Issue が「Claude 待ち」の状態になると、次のスケジュール実行でこのスキルが起動し、リサーチを生成して結果を保存し、レビューを通して本流に取り込むところまでを、人間の合図なしで進めます。

中身のエンジンは二段構えです。

- **主エンジン**: Claude Code ネイティブの `/deep-research`。Opus モデルが調査全体を指揮し、`tools/run_deep_research_workflow.py` を経由して実行されます。1本あたり実測でおよそ36分・約 $18.7。対象は長尺動画にしぼり、月10本までとしています。
- **フォールバック**: 主エンジンが動かないときに限って、DIY（Sonnet + ウェブ検索の並列）へ切り替えます。

一つめの主エンジンは、手動でチャットに向かっていたときと同じ「複数ソースを横断して統合する」調査を、Claude Code の内側で再現するものです。二つめのフォールバックは保険で、ふだんは出番がありません。

なお、以前フォールバックの候補に挙げていた Gemini のディープリサーチは、2026年7月時点では利用しない方針に決めました（2026-07-16 決定）。1本あたりの単価が高く、コストに見合わないと判断したためです。

コストには天井を設けてあります。月の合計が **$50 に達したら止まり、$45 で警告** が出ます。ただし、既定のサブスク経路で回っているぶんには、この $18.7 が請求として上乗せされることはありません。従量課金の経路を使うときにだけ、この天井が効いてきます。

## 自動でも、品質は落とさない

自動化でいちばん心配になるのは、誰もチェックしないまま怪しい事実が台本に流れ込むことでしょう。実際、多くの場合はそこが自動化の弱点になります。ここで問題にしたいのも、まさにその一点です。

リサーチ結果の数字・日付・人名・企業名には、 **fact_check_flags** という印が付きます。この印が5件以上たまるか、一次ソースで裏の取れていない主張（ランクC）が1件でも残ると、結果はそのまま先へ進めません。 **ランクC自律検証ループ** が回りはじめます。

- まず機械的に情報源を再評価します。
- 足りなければ追加の裏付けリサーチをかけ、一次ソースにたどり着いたものは採用します。
- どうしても裏付けられないものは、削るか、意見の引用として扱いを落とします。

人間に判断が上がってくるのは、この検証ループでも救えなかった残りだけです。しかも、テーマの成立に必須の「核心主張」が裏付け不能なときに限られます。細かな数字が一つ確認できないくらいでは、あなたの手を煩わせません。週に一度 Slack を開いたとき、そこに相談が並んでいなければ、リサーチは静かに全部通ったということです。

## 章の冒頭に戻ると

動画1本につき人間が手を動かす工程が一つだけ残る、と考えていました。いまその工程は、あなたが週に一度レポートを開くまで、あなたの手を一度も借りずに終わっています。最初の1本を手で回した記憶が、そのまま自動化の中身の地図になります。

残ったのは、線引きの問題です。どこまでを自動に任せ、どこで人間が最後の責任を持つのか。fact_check_flags の天井も、$50 のコスト上限も、その線引きを機械の側に埋め込んだものでした。同じ問いは、このあとの台本生成や公開の工程でも、形を変えて何度も戻ってきます。
