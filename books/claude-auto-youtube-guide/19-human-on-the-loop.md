# Human-on-the-loop設計 ── 週15分の監視だけで回し続ける

## Human-on-the-loopとは（再確認）

第1章で紹介した概念を、ここで実装の観点から深掘りします。

```
Human-in-the-loop（参加型）: あなたが各ステップで承認・指示する
Human-on-the-loop（監視型）: システムが自律的に動き、異常時だけあなたが介入する
```

本書のシステムは後者を目指しています。

## あなたの週次作業リスト

理想的な状態では、あなたの定期作業は以下だけです。

| 作業 | 場所 | 所要時間 |
|------|------|---------|
| Slackの週次レポートを確認 | Slackアプリ | 5分 |
| `status:waiting-user` のIssueを対応 | GitHub Issues | 5〜10分 |
| ディープリサーチを実行（動画1本あたり） | Claudeチャット | 10〜20分 |

**合計: 週15〜30分程度**

## `status:waiting-user` とは

GitHub Issuesの `status:waiting-user` ラベルが付いたIssueは、
**Claudeがあなたの判断を待っている** 状態です。

主なケース:

| ケース | 内容 |
|--------|------|
| ファクトチェック要注意 | 不確かな情報が台本に含まれている |
| 品質ゲート未達 | 台本の尺が短すぎる等 |
| テーマ選定の確認 | デリケートなテーマについて |
| APIエラー | YouTube APIの認証エラーなど |

このIssueが0件なら、すべて自律的に処理が進んでいます。

## Claudeからの通知を受け取る仕組み

本書のシステムには **Slack通知** が組み込まれています。

| 通知の種類 | タイミング |
|-----------|-----------|
| 週次レポート | 毎週月曜 07:00 |
| 日次進捗報告 | 毎日 19:00（要対応があれば） |
| 緊急通知 | 品質ゲート未達・APIエラー等 |

Slackを使わない場合は、GitHub Issuesのメール通知で代替できます。

## 「止まるべき時だけ止まる」原則

本書のシステムでは、以下の「境界外」リストに該当するときだけClaudeが止まります。

| 境界外アクション | 理由 |
|----------------|------|
| mainブランチへの直接push | 保護ブランチのため |
| 動画の即時公開（手動） | ブランド・収益に影響するため |
| ファクトチェック致命的NG時の続行 | 誤情報リスクのため |

**これ以外のすべてはClaudeが自律的に判断・実行します。**

「〇〇してよいですか？」という確認はしません。実行して結果を報告します。

## セルフ改善ループ

本書のシステムには **自己改善の仕組み** が組み込まれています。

```
ミスが発生
  ↓
retrospective スキルが KPT（Keep/Problem/Try）を自動生成
  ↓
Try 項目が GitHub Issue として自動登録
  ↓
retro-try-handler が Try を自動実行
  ↓
lessons.md（繰り返しミス防止パターン集）に自動追記
  ↓
次のセッションでは同じミスを繰り返さない
```

この仕組みにより、システムは時間とともに自律的に品質が向上します。

## PR自動マージとは

各フェーズの成果物はGitHubのPRとして管理され、
AIレビュー通過後に**自動でマージ**されます。

```
台本生成完了
  ↓
自動でPRを作成
  ↓
Gemini Code Assist + GitHub Copilot が自動でレビュー
  ↓
指摘があれば自動で修正
  ↓
問題なければ自動でマージ
  ↓
次のフェーズへ自動で進む
```

あなたがPRを手動でマージする必要は原則ありません。
